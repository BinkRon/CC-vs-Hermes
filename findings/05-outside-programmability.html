Loop 与用户 · 上：你如何改写它

铸好的壳，谁被允许改写哪一层？（这是双向关系的上半；下半「它如何适应你」见下篇）

谁拥有代码 固定插槽 vs 可换 provider 市场 vs 策展目录 厂商扛信任 vs 你自己扛 企业只读总闸
两家都把 harness 做成了高度可编程的平台——但开放的是两种不同的东西，而且「信任由谁来扛」正相反。分水岭还是那个隐藏变量的延伸：谁拥有代码。Claude Code 碰不到核心，只能在封死的引擎上凿一批固定插槽，并被迫自建一整套信任与治理；Hermes 把源码给你，几乎一切做成可替换 provider，而它不必自建那种「企业市场式」治理墙——「你拥有它」已替它回答了大半个信任问题（剩下用 opt-in、显式授权、安全扫描等轻边界兜底）。但这只是人机关系的上半——你居高临下地改写它；下半反过来：它趁你不在，把你记进文件、越用越懂你（见下篇）。

01问题：让它做设计者没预料过的事

用产品语言讲：可编程性不是「功能多少」，而是「你想让它长成它出厂时没有的样子，路走不走得通」。

你想让 agent 做一些它的设计者从没预料过的事：强制「不过测试不许合并」、把它接到你公司内网的 API、灌进你团队的编码规范、换一个别家的模型来跑。怎么做到？

答案取决于一件事：你能不能碰源代码。这一个事实，就把两家的设计劈成了两半——一边你只能用官方替你凿好的开口，另一边你可以把手伸进任何一层。

02两套扩展体系的形状，源自「你能不能碰核心」

左蓝 Claude Code，右紫 Hermes。一个把核心封死、只留凿好的固定插槽、厂商替你扛信任； 另一个把源码交给你、几乎一切可换、信任随所有权交还你自己。

Claude Code：封死的引擎 + 凿好的插槽 + 厂商扛信任

碰不到核心，官方只能主动把开口留出来，并替陌生人之间的分发扛下信任。

两套扩展体系的形状差异，全部源自一个问题的不同答案——「你到底能不能碰它的核心代码」。

对照：Hermes 遇到同一个判断 —— 「直接改那段决策」

同样是「这一轮算不算干完」这类因团队而异的判断，Hermes 不必等官方凿出一个插槽——因为你握着全部源码，那段决策逻辑本身就在你手里，能直接改；要分享给别人、要随主干升级的定制，则收编成「零核心改动」的插件。它不需要「把每一个会变的判断都预先凿成固定开口」，因为改核心这条路本就为你敞开。这与 Claude Code 正好两极：一个把判断权凿成插槽外包给你，一个把整段代码交给你自己动手。

04对比：七组对照看清分歧

能不能改核心、扩展长什么形状、换不换得了模型、谁来扛信任……同一个「可编程」的目标，答案却指向两种所有权现实。

不能（闭源，只能用官方开口）。
能（开源，可 fork / 改源码 / 改配置）。
扩展的形态
在封死引擎上凿的固定插槽——二十多个钩子节点由官方枚举、不可新增，也不能替换调度内核。
几乎一切是可替换 provider（模型 / 平台 / 搜索 / 浏览器 / 语音 / 调度…），中间件可改写请求，也能直接改源码。
换模型
锁的是线协议 / 推理内核（必须说 Anthropic API）——但可用 ANTHROPIC_BASE_URL 把请求指向任意网关（如 LiteLLM 代理），照样跑 GPT / 开源等第三方模型。即"换任意能说该协议的后端"可行，"换推理内核"不行。
一等公民——把提供商做成注册表，配好凭证后一条命令切任意模型，连自建端点都行。

插件市场——可订阅任意市场、安装他人插件，像应用商店。
无市场，策展式目录——进目录 ＝ 一个审核过的 PR，明确「没有目录之外的信任信号」。

直接改配置 / 改源码（开源天然赋予的近场路径）。
为何不同（谁拥有代码）：轴一我们问「谁拥有模型」，第 04 篇问「谁拥有部署」，这一篇问的是「谁拥有代码」。Anthropic 拥有代码、且闭源托管、要卖给从个人到大企业的所有人 → 用户碰不到核心，官方只能把「会因人而异的行为」凿成固定插槽；又因为扩展要在陌生人之间分发、厂商要背安全与合规责任，它必须自建一整套信任与治理（市场审核 ＋ 企业只读总闸）。Nous 把代码给你 → 终极扩展是改源码，于是它一面把几乎每类外部依赖做成可替换 provider（它分发的是产品不是服务，模型 / 平台 / 密钥都得你自己定），一面把「fork 改核心」这条不可维护的路收编成「零核心改动、可随主干升级」的插件；而它不必自建那种「企业市场式」治理——注意它并非"扩展不流转"（pip/PyPI 插件就是不折不扣的陌生人分发通道），而是没有多租户组织边界、也不做市场审核/信任评分，改用更轻、与所有权匹配的边界兜底：插件默认 opt-in、覆盖内置须显式授权、技能安全扫描、平台允许名单。换句话说，"能不能信这个扩展"很大程度上被"你拥有源码和机器、你自己决定装什么"这件事回答掉了。反直觉的收尾：开源不等于扩展点更少（Hermes 的可插拔面反而更铺得开）；真正的分野是治理城墙的有无与归属——闭源托管产品要为「陌生人间分发扩展 + 企业多租户合规」额外建一座市场＋总闸式的治理墙，开源自托管产品省掉的正是这一种墙，因为「你拥有它」已经替它回答了大半个信任问题。

以上是这段关系的上半：你如何居高临下地改写它。但关系是双向的——它也在你看不见的地方反过来记住你、适应你。那是下篇 「它如何适应你」 的故事。

05悬而未决 · 边界说明

  • [CC · 已更正] 早期稿说 CC「不能接任意第三方提供商」是错的：CC 可用 ANTHROPIC_BASE_URL(+ AUTH_TOKEN / CUSTOM_HEADERS) 把请求指向任意网关，业界普遍借 LiteLLM 之类代理让它跑 GPT / 开源等第三方模型，代码里还专门有 isFirstPartyAnthropicBaseUrl() 处理"指向别处"的情况。CC 真正锁死的只是线协议 / 推理内核（必须说 Anthropic API），不是"第三方模型"。所以两家的差别是"经协议网关接 vs 原生 provider 注册表"，不是"能不能接"。
  • [HM] 「改源码是主路径」是开源赋予的可能，不代表多数用户真会去改核心；它的正式插件面主要服务「要分享给别人、要随主干升级」的定制，一次性私人定制才坦然交给改配置 / 改源码。
  • [定位而非成熟度] Hermes 缺的是 CC 那种「企业市场式」治理（多租户 RBAC、插件市场信任体系），是定位选择而非成熟度问题。它并非「没有治理」——仍有插件 opt-in、覆盖内置须显式授权、平台允许名单 / 角色 / 配对门、技能安全扫描等轻边界；只是把大部分信任交还给「你能读全部源码、你自己的机器」。
  • [不评优劣] 本篇讲的是「可编程性的形态与信任归属」，不评判两种模式孰优——取决于你是个人、团队还是企业，以及你能否 / 愿否碰源码。
  • [钩子枚举其实对称] 「固定插槽 vs 几乎一切可换」在钩子这一层被我说得过于不对称：Hermes 的插件同样只能挂到固定的 VALID_HOOKS 集合、注册非法钩子名会被拒。真正的差别不在"插件能不能挂任意钩子"（两家都不能），而在"能不能改源码新增钩子"——CC 不能、Hermes 能。核心「谁拥有代码」结论不变。
  • [两家] 两家扩展体系都在快速演进，本篇是某一时间切片（CC main / Hermes @daf4f1a7a）。
