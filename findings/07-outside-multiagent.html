Loop 与其他 Agent

分身之间，靠喊话还是靠看板？

CC 三种分身：subagent / fork / 团队 全新隔离子代理趋同 · CC 另有 fork（Hermes 无） 团队层：消息同事团队 vs 共享状态看板流水线 两家都能盯单个分身 托管产品的 live-team 更完整
先分清一件常被混为一谈的事：光是 Claude Code 自己，就有三种机制不同的分身——① 全新子代理（subagent）：leader 写一段自包含简报、派一个零共享上下文的隔离分身，它跑完回一条结论；② fork：不给角色时触发，分身继承 leader 的完整上下文与系统提示、共享提示缓存（所以给它的是"做什么"的指令、而非重述背景）——这是"接着我的记忆去干"，和①正相反；③ 团队（teammates）：一队具名队友围着共享看板、用邮箱互相发消息、人当队长双向协作。三者作用与机制都不同。
把两家摆开：在「全新隔离子代理」这一支上两家趋同——CC 的 subagent ≈ Hermes 的 delegation（都是 fresh 会话、隔离、只回结论；leader 可用消息续跑它，但分身之间不通信）；但 CC 还多一条 Hermes 没有的 fork（继承上下文、复用缓存），Hermes 的委派被明确设计为"只有 fresh 会话、无父历史"。真正戏剧化的分野在「多角色团队」的协调媒介：Claude Code 让代理互相发消息（邮箱 + 共享看板 + 具名队友，人以队长身份在同一张消息网里），这套 live-team 产品面做得更完整、更产品化；Hermes 则让独立实例靠共享数据库状态协调（看板/蜂群：依赖图、黑板评论、原子认领，彼此不发消息），且要显式进入才用。一句话：「消息驱动的同事团队」 vs 「共享状态驱动的看板流水线」。

01问题：拆开之后，这些分身怎么协调

用产品语言讲：多智能体不只是「怎么把活拆开并行」，更是「拆开之后，它们靠什么协调、汇聚、不互相踩脚」。

你把一个大活交给 agent——跨 12 个文件重构，或者一次调研 5 种方案。一个代理的上下文和时间扛不下，于是它拆成好几个分身。两个问题立刻冒出来：(1) 这些分身怎么不互相踩脚？(2) 它们靠什么互相协调——是像一支团队那样互相发消息，还是像流水线工人那样各自读写一块共享白板？

这第二个问题，就是两家多智能体设计真正分岔的地方——它决定了整套东西长成「一支你带的同事团队」还是「一条你摆好的看板流水线」。

02核心机制：委派层大致趋同（CC 多一条 fork），团队层才分岔

左蓝 Claude Code，右紫 Hermes。「全新隔离子代理」这一支两家趋同（fresh 会话、隔离、只回结论）；CC 另有一条 Hermes 没有的 fork（继承上下文、复用缓存）。 真正戏剧化的分野在「多角色团队」的协调媒介：一边靠代理互相发消息，另一边靠共享数据库状态。

对照：Hermes 面对同一个大项目 —— 「一条看板流水线」

Hermes 的默认做法是——模型自己把活 fan-out 成几个分身并行跑、只把结论收回来；这些委派分身之间刻意不互相对话（不是一支会喊话的团队）。真要多角色协作，就显式进入看板 / 蜂群这条重路径：一张任务图，每个角色是一个完全独立的 Hermes 实例，它们不发消息给彼此，而是各自读写一块共享数据库白板（依赖图、黑板评论、原子认领）来协调——像流水线工人从看板上领活、交活。这里也不是黑盒：你能通过 TUI 的 /agents 面板旁观、暂停、打断单个分身。差别不在「能不能监督」，而在协调的隐喻——CC 是互相喊话的同事，Hermes 是共享白板上的工人；而且 CC 把「同事团队」这层做成了更完整、更产品化的一等体验，Hermes 的看板流水线则要你主动走进去才用。

05悬而未决 · 边界说明

  • [CC 有三种分身，别混] Claude Code 的多智能体不止「团队」一种，且委派本身分两条：subagent（自包含简报、零共享上下文的全新分身）与 fork（继承 leader 全上下文、复用缓存——和 subagent 正相反）；再加上双向协作的团队。本篇把「消息驱动的同事团队」这个特征限定在团队层；「零共享上下文」只对 subagent 成立、不适用于 fork。（早期稿曾把 fork 误并进 subagent，已更正。）
  • [委派非纯单向] 即便普通 subagent，leader 也能用消息把它续跑（分身带完整上下文恢复、再回一条结果），coordinator 模式更把"续跑已完成/失败的 worker"当核心工作流。准确的说法是「分身之间不互相发消息」，而非"分身完全单向一次性"。
  • [两家都能监督] 别读成「CC 能盯、Hermes 是黑盒」——两家都能旁观、暂停、打断单个分身（CC 靠分屏面板，Hermes 靠 TUI 的 /agents 面板）。分野在协调隐喻，不在能不能监督。
  • [不是「默认关闭」] Hermes 的看板/蜂群不是被开关关掉的功能，其派发默认是开的；准确说法是「要显式走看板/蜂群这条路径才进入」，不是普通聊天的默认形态。
  • [CC 计划审批自动通过] 桥接给人的是敏感工具/权限确认与叫停；「计划模式审批」在 leader 侧是自动通过的，并非每次必经人工。别把它读成「什么都要人批」。
  • [peer 未闭环] CC 的「跨会话/跨机器 peer 通信」在当前代码树里提示词与地址已在、但若干实现文件缺失，未能闭环证明可用——本篇按存疑处理，不作为既成能力。
  • [隔离没那么强] Hermes 委派分身默认共享同一容器工作目录（除非注册隔离 override），不是每个都独立沙箱；真正的强隔离在看板蜂群（每角色独立实例）。
  • [归因是解读] 把差异归到「第 04 篇的在场重心」是一个合理解读，非代码可证的铁律——代码只能证明两套协调机制的形态差异。
  • [呼应轴一] 「委派分身结论自报、父代理须自行核验」呼应轴一——Hermes 对任何模型（含自家）的「我做完了」都不全信。
  • [计划清单不在此篇] 单个 agent 给自己列的「计划 / 待办清单」不是多智能体——它是一个 loop 内部的过程纪律，归轴一·控制侧（第 01 篇 §04 第④维）：那里对比了 CC「任务清单＋壳主动催」与 Hermes「待办工具刻意不挂钩子」。
  • [移动标靶 · 不评优劣] CC 团队多在特性门后灰度放量，是移动标靶（本篇是 CC main / Hermes @daf4f1a7a 切片）；两种协调隐喻不评孰优，取决于你要「一支会喊话的同事团队」还是「一条靠共享状态协调的流水线」。
