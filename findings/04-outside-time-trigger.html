Loop 与时间：你不在的时候，它把「你不在」当成什么

交代完就走了——从这一刻起，它干活的时间线和你的注意力时间线彻底分了家。

TL;DR

你把要做的事交代完，人就走了。从这一刻起，agent 干活的时间线和你的注意力时间线解了耦——它继续走，你已经不在看。

同样面对「你不在」，两家的读法相反：CC 把它当成一个被持续测量、要设法弥合的信号——设法把你的注意力找回来；Hermes 把它当成一个由语境注定、要照章办的事实——超时=没同意、错过=不补、无事=沉默。

00问题：你不在，才是这台机器的常态

凌晨三点，你睡了。一个新 PR 刚合进来，或者你上周设的定时冒烟任务正好到点——agent 该动的时候，你不在。这段时间里它得能存在、被叫醒、把活干完；万一半路卡住需要你拍板，它还得找到你；干完了，把结果送到你早上会看见的地方。你不在，是常态，不是异常。于是真正的问题不是「它会不会干活」，而是：当你不在，它到底把「你不在」这件事读成了什么——一个要设法弥合的缺口，还是一个照章办事的既定事实？

01你交出去的，是「在场监督权」

交代完就走，你让渡的其实是一样东西：在场监督权——不再守在旁边随时喊停、随时确认。这一让渡，内生出四件它必须替你兜住的事。这四件事，就是本章的问题骨架，也是下面两家各自的答卷。

干得完 —— 你不在，它的时间线不能断在半路（谁让它醒着、崩了怎么办、错过了补不补）。

找得到我 —— 真需要你拍板时，它得够得到你（等多久、超时算什么、无人时敢不敢自己定）。

送得到我 —— 干完了，结果要送到你回来会看见的地方（送哪、要不要主动把你叫回来）。

别乱来 —— 没人盯着的时候，谁替你踩刹车（危险动作在无人时走向哪）。

02两家靠什么知道「你不在」——一个靠测量，一个靠注定

在应答四诉求之前，先看最底层的一道分野：它到底怎么知道你不在。CC 靠传感器测出来（信号长在自家终端/桌面上）；Hermes 靠语境注定——你从哪扇门进来，就注定了你在不在。

Claude Code · 一排装在你终端里的传感器，实时读你在不在：

  • 一回合干完、你 60 秒没搭理 → 判定人不在，推一条「Claude 在等你」的通知。默认
  • 终端失焦满 5 分钟（靠 DECSET 1004 焦点上报）→ 生成一张「你走开时这边在干嘛」的离开摘要卡。特性门控·默认关
  • 终端焦点状态（focused / unfocused）→ 直接当「自治度旋钮」：你在看就更协作，你走了就更大胆。门控
  • 你此刻在手机上 → 把终端弹窗换成手机 IM 上的短码批复。门控

Hermes · 它不测量，它被告知——你从哪扇门进来，在不在就已经写死：

  • CLI 态（你就在本地终端前）＝在场，无限等你敲键。
  • 网关态（你从 Telegram/Discord 等聊天里来）＝半在场，聊天可达、但你可能几分钟才看一眼。
  • cron 态（定时任务到点自己开火）＝结构性不在场，根本没有人。

两家最能说明问题的，是这一对并排的真制品——一家替你惦记「你会回来」（主动生成塞给你），一家干脆没有「离开时主动摘要」这一手：

CC · 替你惦记「你会回来」特性门控·默认关

失焦 5 分钟，主动生成一张摘要卡塞给你

离开摘要 · 提示词原文The user stepped away and is coming back. Write exactly 1-3 short sentences. Start by stating the high-level task ... Next: the concrete next step.

Hermes · 没有「离开时主动摘要」这一手

只有一个纯本地、零成本的 recap 助手（设计取向：instant/free）

session recap · 文档头原文Inspired by Claude Code's /recap command ... Pure local computation. No LLM call ... A recap should be instant and free.

同一件事——「人走开、又回来」——CC 有「离开时主动生成摘要」这一手（特性门控），Hermes 这边只有一个纯本地、零成本的 recap 助手、未见「离开时主动推送」；一个主动替你惦记，一个不主动惦记。

这道分野背后是一句话的部署现实（不展开，只作约束变量）：CC 出厂是个装在你终端里的客户端，所以能测量你的焦点；Hermes 交付的是你自己跑的一台服务器，它只拥有「网关」这一扇门，所以只知道「你从哪扇门进来」。

03它替你把四件事办成什么样

带着上一节那道「测量 vs 注定」的分野，逐一看四诉求的答卷。四件事里，「找得到我」最见功力——放在中间重点画。

3-a干得完：错了之后，一个先问人、一个按规矩办

时间线断在半路怎么办？两家都为「崩了 / 错过了」设了语义，但第一反应相反。

Claude Code · 错过了？先问人 cron/loop 默认开·可急停

定时任务绑的是「项目」不是某个会话（持锁的会话崩了，别的会话 5 秒内接管），只在会话空闲时开火，周期任务满 7 天自动过期。错过语义还分叉：一次性任务错过 → 先从磁盘删掉、再让模型用提问工具问你「要不要现在补跑」、绝不自动补；周期任务 → 静默开火一次、向前重排。

一次性任务错过后的通知 · 原文The following one-shot scheduled task was missed while Claude was not running ... Do NOT execute this prompt yet. First use the AskUserQuestion tool to ask whether to run it now. Only execute if the user confirms.

Hermes · 崩了？宁漏跑不重跑 默认

常驻网关每 60 秒敲一次节拍。崩溃语义是宁漏跑不重跑（执行前先把下次时间往后推，把「至少一次」压成「至多一次」）；宕机太久攒下的积压不会一次补发，塌缩成只补一次，宽限窗＝周期的一半、夹在 120 秒到 2 小时之间。

一句话对照：错过之后，CC 的第一反应是问人，Hermes 的第一反应是按预定语义办——因为 cron 态里根本没人可问。

3-b找得到我：同一个「要你拍板」，五种命运

这是四诉求里最见功力的一件，也是本章唯一的重点图。干到一半需要你拍板，两家不是同一个动作的两种参数——连分叉的轴都不一样：CC 按「你此刻注意力在哪块屏」分叉，Hermes 按「你从哪扇门进来」分叉。越往下（越不在场），CC 越倾向「自己拍」、Hermes 越倾向「要么等满要么不许问」。

那个「等 1 小时」不是拍脑袋定的：旧默认是 10 分钟，结果晚点回来的用户点到已经过期的条目、把 agent 卡死，才调长到 1 小时——既够久让离开的人回来还能点，又够短让真被遗弃的提问最终能松开线程。

0.5 秒该看到的差别：CC 问的是「你的注意力在哪块屏」，Hermes 问的是「你从哪扇门进来」——连分叉的轴都不一样，不是超时长短之别。

3-c送得到我：把结果送回你，和把你的注意力找回来

干完了，结果去哪、要不要把你叫回来？这里两家的重心也不同：CC 一层层往外够、想把你的注意力找回来；Hermes 把结果安静地放回你出发的地方。

Claude Code 是一道渠道阶梯（按离你远近排，越往外越依赖门控与服务端，纯本地那圈才是默认）：

  • 本地响铃 / 桌面通知「Claude is waiting for your input」默认 GA
  • 离开摘要卡（回来一眼接上）门控·默认关
  • 桌面深链 claude://resume + teleport 跨机续会话 门控
  • channels 手机 IM 收发 多层门

（纯本地那圈没有把通知推到手机 app 的路径——跨端叫回全靠上面外圈的门控能力。）

Hermes 则是回出生地：在哪个聊天里建的任务，结果就回哪个聊天——渠道是建任务时定下的代码规则，不是模型每次自选。提示词甚至明说「别自己发、系统负责送达」；真没事发生就保持安静。

投递归属 · 提示词原文DELIVERY: Your final response will be automatically delivered to the user -- do NOT use send_message or try to deliver the output yourself. Just produce your report/output as your final response and the system handles the rest.
无事发生就保持安静 · 提示词原文（另有调度器代码识别 [SILENT] 抑制投递，双层）SILENT: If there is genuinely nothing new to report, respond with exactly "[SILENT]" (nothing else) to suppress delivery.

对照：CC 忙着把你的注意力找回来（一层层往外够），Hermes 忙着把结果安静地放到你回来会看到的地方。

3-d别乱来：没人看着时，谁替你踩刹车

这是四诉求里两家最不对称的一件——Hermes 有一整套「无人时危险动作怎么办」的词汇，CC 这块更薄（因为它默认你会回来，可以拖到你回来再说）。这个不对称本身就是论点证据。

Hermes（默认）把话说得极硬：网关审批阻塞等 5 分钟、超时=拒；cron 态默认拒（要放行得显式打开信任开关）。回执措辞是这道闸的语言核心：

危险命令被拒 / 超时的回执 · 原文BLOCKED: Command {reason}. The user has NOT consented to this action. Do NOT retry this command, do NOT rephrase it, and do NOT attempt the same outcome via a different command ... [超时附加] Silence is not consent.

一处坦白取舍：既不是交互 CLI、又不是网关、又不是 cron 的语境（根本没有审批面），危险命令反而会自动放行——「没有审批面即信任」。但硬红线命令（rm -rf /、格式化磁盘等）在任何开关之前就被拦死，一律绕不过。（审批机制本体属环境章 06，这里只看「你不在时它走向哪」。）

CC（门控）这侧的对应物更薄：channels 的六层准入门 + 手机回批必须「y/yes + 5 位字母码」防聊天误触；remote 容器里权限请求既不自动拒也不自动放，而是合成一条消息桥回挂着的客户端等人批。

收一句到主动性天花板：Hermes 无任何自发唤醒 / 自发外联（研究员 grep 搜遍未发现，故用「未发现」不用「绝无」）——连自动化建议都 consent-first、永不自动建任务。CC 的自治态给了 agent「找活干」的裁量，但首拍克制（新会话第一下只打招呼）、空转必睡。

别停在「谁的表更全」。退一步看，上面每一格都在回答一个更根本的问题——它到底把「你不在」当成什么？

CC 是注意力路由器：重心是把你的注意力找回来（响铃默认，摘要 → 深链 → 手机多为门控），在场性对它是一个待感知、待路由的信号。它也有 cron 补跑语义，但力气主要花在「够到你」上。

Hermes 是无人值守规约：重心是替不在的你把每件事的语义定死（超时=没同意、错过=不补、无事=沉默），它不找你的注意力。它也有工作中的进度心跳，但那是给回来的人看，不是去够你。

最后再收一层——在它眼里，你的「不在」到底是什么？

CC：你的不在是一段终会弥合的缝隙——它的世界里，你 stepped away and is coming back，所以它一路布点，等你回来接上。
Hermes：你的不在是一纸生效中的委托——你走前签好的那套规则，就是你不在时的你，它照章办事。（这只是「与时间」这一维照见的判断；与「用户」「自己」等维合起来才是完整答案。）

05悬而未决 · 防夸大红线

  • [口径三分] CC 的「别问、自己拍板」有两个来源要分清：仓库内的自治态（proactive/KAIROS 特性门控，prompts.ts:892-913）是本章证据；另有一段更强的自治注入语来自 Agent SDK / 服务端、不在 CLI 仓库内、本篇未独立核实，故不作事实断言，仅此说明。
  • [CC 在场机制多为门控] 离开摘要、channels、teleport、自治态都在特性开关之后；默认交互态是「先问再做危险操作」。别把门控能力写成 CC 的默认姿态。
  • [Hermes 无自发唤醒] 「无自发外联」是 grep 未发现的缺席结论，非穷尽证明；所有唤醒都是外部触发（时钟 / 入站消息 / webhook / API）。
  • [审批本体归环境章] 本章只取审批的「用户不在时走向」这一切面，完整的审批分类学在环境章（06）。
  • [持久化够用级非事务级] Hermes 单机 60 秒节拍 + 文件锁，无分布式精确一次；崩溃时宁漏不重。
  • [不评优劣 · 时间切片] 本篇讲两家「怎么读你的不在」的形态，不评孰优。切片：CC main / Hermes @daf4f1a7a。
