Loop 与时间

两次触发之间，谁让它活着？

谁替它保活 常驻 vs 会话 定时 · 唤醒 主动投递 自主休眠省钱（实验·默认关） 会话内：同步阻塞 vs 后台并发
同样要让 agent「跨越单次对话而存在」，两家被相反的部署现实推着走。分水岭不是「谁功能多」——两家都有定时、后台、远程触发、推送——而是一个隐藏变量：谁替 agent 保活。Claude Code 是厂商托管产品，答案是「厂商云替它保活」，于是一切围着人的注意力缝隙编排；Hermes 是自托管开源 agent，答案是「没人替它保活」，于是它被逼着自己变成那个常驻活体。

01问题：两次触发之间，谁让它活着

用产品语言讲：模型是个「被叫一次算一次」的函数，两次调用之间它根本不存在。谁来填补空窗？

一个具体场景：凌晨 3 点，你的代码仓库来了个新 PR 需要 review；或你想让 agent 每天早上自动跑一遍冒烟测试。可你本人在睡觉、在上班，不在终端前。这时 agent 要能存在、被叫醒、干活、再把结果送到你眼前。

但模型本身是个无状态、被叫一次算一次的函数——两次调用之间它根本不存在。谁来填补「两次触发之间」的空窗？谁在你不在时让它活着、按时醒来、干完还能找到你？这就是「时间与触发」的题眼。两家给出的答案，暴露了它们对「agent 到底活在谁的地盘上」的根本不同假设。

对照：Claude Code 的这一天 —— 「围着人的注意力缝隙转」

Claude Code 的时间机制几乎全部假设「人拥有一个会话、终会回到它」。它在人离开的注意力缝隙里干内务（记忆整理、后台清理），本地定时只在会话还活着且空闲时才触发；真正需要「无人长驻」的活儿则外包给厂商云，本地进程有意不做成常驻守护。人回来时，它用通知把人叫回、用离开摘要帮人接上刚才发生的事。它不需要「自己成为一个会醒来会睡去的活体」——因为厂商的基础设施就是它的保活者。这与 Hermes 正好两极：一个把「跨时间存在」靠在厂商身上，一个把它全扛在自己身上。

单会话内 · 一条长命令怎么办
（同步阻塞 vs 后台并发：这是「跨时间存在」在一次会话内部的缩影）
两家都不让长命令冻住 loop——都能把活甩到后台、干完再回来通知，而不是干等；也都覆盖「一条命令」和「一整个子代理」两种粒度。CC 的特点是会自动帮你解绑：一条本该等结果的命令跑满 15 秒还没完，harness 主动把它转入后台让 loop 继续（模型没开口也解绑；「原地干等 N 秒」这类除外）；若某命令像是卡在等你敲输入（约 45 秒没新输出、末行像个提示符），才专门提醒——纯粹跑得慢的（长构建等）不打扰。也明令禁止用「空转等待」反复轮询。
同一副骨架，但走显式声明：模型自己标「这条要后台」；想在前台等太久会被直接拒绝、劝你改后台（而非跑到一半自动转）。后台进程可挂完成通知，或盯住特定输出（限频、连命中太多次自动降级为只报完成）。同样是「一条命令 / 进程」能后台、「一整个委派子代理 / 批次」也能后台，完成时在主 agent 空闲档口作为一个新回合回到对话，刻意不硬插进正在进行的那一步。
为何不同（谁拥有部署 / 托管）：轴一里我们说，两家的根本分歧是「谁拥有模型」——Nous 拥有模型、能把可靠性训进权重。到了这一维度，所有权的对象换了：分水岭变成「谁拥有部署 / 托管」。Anthropic 拥有托管平台 → agent 可以把「跨时间存在」这件重活儿靠在厂商基础设施上（云会话保活、平台在，本地便只需假设人会回到终端）；Nous 交付的是你自己下载、自己部署的开源软件 → 没有厂商这一层，agent 只能自己扛起「跨时间存在」的全部基础设施（自带常驻网关、自我唤醒、自我投递，甚至还有实验性的自我休眠省钱）。所以同样叫「时间与触发」，一个是「给一个有人照看的会话，配上时钟和缝隙里的自动化」，另一个是「让一个没人照看的软件，自己成为一个会醒来会睡去的常驻活体」。反直觉的收尾：做「最小开源壳」的一方，反而在「让 agent 独立地跨时间活着」上做得最重——因为独立部署这件事本身，不允许它依赖任何外部的保活者。

05悬而未决 · 意料之外

  • [CC] 「本地不做真 · 常驻」是就本地交互进程而言；把无人长跑外包给厂商云是一种设计选择，不是能力缺失。随产品演进，本地调度也可能加强。
  • [非绝对二分] 本篇对比的是两家的默认姿态与「把跨时间存在的担子放在哪」，不是绝对二分：Claude Code 也有无头 / SDK 的常驻调用方式，Hermes 也能把调度换成外部 provider。分歧在重心与默认，不在「有没有」。
  • [对比立场] 关于「厂商侧自动任务有每日次数上限 / 只能用单一模型 / 只发代码平台评论」——这是 Hermes 仓库里一篇自我定位文档的说法，是 Nous 的对比立场；本篇未独立核实那款厂商功能的具体配额，故不作事实断言。
  • [HM] Hermes 的持久化是「够用级」而非「事务级」：单机 60 秒节拍器，靠文件锁与检查点兜底，没有分布式高可用 / 精确一次（exactly-once）的保证——宕机期间积压的重复任务不会逐次补齐（至多下次补跑一次），且崩溃时宁可漏跑也不重跑。
  • [两家] 这套机制都在快速演进，本篇是某一时间切片（CC main / Hermes @daf4f1a7a）。
