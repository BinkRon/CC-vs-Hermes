# CH-A2R-04 · 视觉脚本（渲染前的「设计」这道工 · design-principles Part 2.1 五问）

## 问 1 · 一句话论点
自我进化不是训练问题，是**控制回路问题**——两家都在长大（积累环实线闭合），却都没长出量「变好」的那只眼（评估/信用分配虚线断开）；差别在 **CC 自觉把眼外包给厂商，Hermes 拥有模型却连眼都还没造**。

## 问 2 · 最大卡壳点 →（决定重档给谁）
**「积累闭环」被误读成「改进闭环」。** PM 读者最容易把「它自己写记忆、造技能、下次还记得」直接等同于「它在变好/自我进化」。这正是全章要打破的直觉：**记住 ≠ 变好，变大 ≠ 变强。** 不精心画，读者带走的就是被 README 植入的那句「它越用越强」。
→ **重档（第 1 名）= 一张「断裂的飞轮」主图**：0.5 秒让读者看到——左半环实线闭合（自改→写盘→下次载入→再自改），右半那条「这次帮到了吗？」的边是**虚线缺口**，fitness 节点是**虚线框**。

## 问 3 · 叙事顺序
问题（拆穿最窄读法）→ 机制（拆成控制回路·铺共同实线半环）→ 差异（缺的那只眼）→ 判词（眼去哪了+verdict+反思）→ 诚实边界。**默认顺序，本章不偏离**（偏离需论证，此处无必要）。

## 问 5 · 重档草案（断裂的飞轮）

**推荐做法（渲染员可在 A/B 择一，A 优先）**：

- **方案 A（首选·单张统一飞轮）**：一张 `flowchart TD` 画通用控制回路——
  - 实线环：`使用/会话` →（实线）`自改：写记忆·造技能` →（实线）`写盘固化` →（实线）`下次会话载入` →（实线回到）`自改`。这条闭环标注「积累环·实线可证」。
  - 缺口：从 `写盘固化` 引一条**虚线**边、标 `这次帮到了吗？` → 指向 `fitness：什么算变好`（**虚线框**、标「运行时无」）。这条是**信用分配缺环**，视觉主打点。
  - fitness 节点再分两条**虚线**支路，把两家分野压进同一张图：`CC：外包给厂商 eval+A/B+在场人（会话外）` / `Hermes：哪都没接（悬空）`。
  - 旁挂 GC 支路：`写盘固化` →（实线）`curator/dream 剪枝`（**amber 判断色**）、标「按闲置/整洁·不按效用」——防读者把剪枝误读成 fitness。
  - **读者 0.5 秒看到**：环的左侧实线闭合、右侧一个虚线缺口；缺口下 CC 有根线通向"会话外厂商"、Hermes 纯悬空。
- **方案 B（备选·`.flow2` 双面板蓝/紫）**：若单图太挤，退回两面板，同一套图形词汇，差异节点打亮——CC 面板"自改"标 amber「人/厂商触发」+ 一条实线通出面板到"厂商 eval+A/B"；Hermes 面板"自改"实线「后台自主写」+ fitness 纯虚线悬空。两面板 fitness 均虚线（同缺）。

**训练管线不进 Hero**（保焦点）：降为节 1 的轻档制品（`dataset_file`/无训练代码），不画进飞轮。

**配套真实制品**（节 3 贴 Hero 后）：Hermes 侧 `use=0 is not evidence a skill is valuable` + `a pass that does nothing is a missed learning opportunity`；CC 侧 `Eval-validated (…0/2 → 3/3)`。

**词条解码**：Hero 里若出现「信用分配」「fitness」这类词，图下附注一句白话或 `data-tip`（judgment：这两个是黑话，需解码；「剪枝」「载入」是常识，不加）。

## 机械/门 自查预挂
- 语义配色：判断/剪枝=amber、终点(写盘/载入)=green 或中性、缺口=虚线+faint/red。图内字号 ≥15px。
- 所有 mermaid 调用裹 `MERMAID_OK`（离线守卫）。
- 标题必论断：节 3「量『变好』的那把尺子，两家都没有」、节 4「那只眼去哪了：一个外包，一个没造」——非栏目名。
- 粗体每段 ≤2（两家着色不计）；正文裸文、`.artifact`/`.matrix`/`.verdict` 才进框。
