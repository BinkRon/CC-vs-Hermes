## 0 · thesis / TL;DR（论断式横幅）

**一句话**：轴二五章看着是五件事，分野却由**同一侧的关键约束**决定——**你为谁造、怎么交付**（用户世界的约束）。这与综合①那侧（你站在什么模型上）并列，是 harness 夹在两个世界之间、各自那张脸。

- CC 处处按「**开发者对着仓库改代码、人多半在场、身后有厂商**」优化。
- Hermes 处处按「**IM 私人助理、常态无人、自托管你自己扛**」优化。
- **"在场"不是根，是这个场景在时间/环境上的症状**；根是场景本身。

（写法：先不许说"五次显形一个变量"——那是被证伪的旧脊。要说"两个世界、两个约束，本篇解外侧那个"。）

## 2 · 两种场景素描（先把两个用户世界画清楚）

| | Claude Code | Hermes |
|---|---|---|
| 用户在哪 | PC 前、对着 CLI、改一个真实仓库 | 通过 IM 布置任务、跑完 IM 交付验收 |
| 活的性质 | 系统性、重、要求代码可靠 | 助理式、你使唤、结果导向 |
| 人在不在 | **多半在场**（长跑离开是要兜的边缘） | **常态无人**（在场是例外） |
| 交付形态 | session-based（resume/teleport 是 session 级操作） | **消解 session、当你一个个体**（USER.md 画像、跨会话连续） |
| 身后有谁 | 厂商接盘（可轻、把风险交还"你选对输入"） | 没有厂商、**自托管你自己扛** |
| 只需给它 | 一个终端 + 权限门 | 一个能跑的环境 + 足够权限 |

**⚠ 三处事实红线（骨架别长歪）**：
1. **session 方向别写反**：CC 才是 session-based；**Hermes** 才是"当你一个个体、消解 session"。
2. **CC 的 remote-control 大半是灰度门控**（channels/teleport/摘要卡/自治态），标"意图/可开启能力"，非默认姿态（记忆 cc-source-tree-feature-flag-gated）。
3. **Hermes 不是纯远程**：还有本地 CLI 那扇"在场"门（04 三扇门：CLI/网关/cron）。

## 3 · 场景怎么把五章往两边掰（主体·逐章）

每行标：**话题侧 · 驱动约束 · 场景掰成什么 · 掰不动处**

| 章 | 话题侧 | 驱动约束 | 场景把它优化成 | 掰不动/要点名的地方 |
|---|---|---|---|---|
| **04 时间** | 外侧 | **场景（直驱）** | CC=注意力路由器（测焦点、找回你）；Hermes=无人值守规约（定死语义）。成本铁证：CC 替厂商机群削峰 / Hermes 替你云主机省钱 | — 干净 |
| **05 用户** | 外侧 | **场景（直驱·最强拟合）** | CC=多租户生产力工具（org 可治理明文、从不建个人画像）；Hermes=单人自托管陪伴（主动建档、单机单人） | 边界软(记忆)/硬(权限)分层要说准 |
| **06 环境** | 外侧 | **部署/谁自担（场景近亲）** | 主事实是**趋同**；分野在"风险归谁"→逼出诚实度（Hermes 亲口"这不算隔离"） | 地板判据分野（物理不可逆 vs 治理配置）根在部署所有权，**不是场景**，要点名 |
| **07 其他 agent** | 外侧 | **一半场景 · 一半自托管多实例** | CC=消息网+共享看板、父亲自综合；Hermes=纯共享黑板、综合外化成角色卡 | CC 团队/编排=灰度实验，别当默认产品（记忆同名）；profiles 根在自托管多实例 |
| **08 自己** | **外侧** | **⚡交棒模型侧约束：训练所有权** | 两家都缺评估；缺法不同：CC 不拥有模型→外包厂商 eval/A-B；Hermes 拥有模型→有杠杆没造评估 | **显式说：本章话题在 Loop 之外，根因在 Loop 之内——接综合①的"所有权/第二杠杆"榫** |

## 4 · 两处交叉：约束横切话题轴（呼应综合①·本篇最强证据）

**论断**：真正的引擎是两个约束，它们横切"章节话题"在跑——一章讲什么、和差异由什么逼出，不总是同一侧。

- **02 动态注入 44 vs 8**：话题在**内侧（轴一）**，根因是**外侧约束（开发者产品场景）**——CC 的 IDE/终端/待办信号是"对着工作台改代码"长出来的。（顺手认领了综合①标为"产品 vs 底座、另一回事"的那个孤儿。）
- **08 自我评估**：话题在**外侧（轴二）**，根因是**内侧约束（训练所有权）**。

一内一外、正好镜像 → 说明约束是引擎、话题只是落脚点。**（这段是综合②回扣综合①、让两篇押同一个韵的关键。）**

## 5 · verdict：两种处境，抄谁的作业看你像谁

**论断**：单位不是功能对照表，是**两套各自咬合的处境**——外侧（为谁造/谁自担）与内侧（站在什么模型上）一起把每个默认拧到位。

- 你若做**开发者代码工具、有可信在场的人、身后有基座厂商** → 像 CC：假设有人、把注意力找回来、风险交还"你选对输入"。
- 你若做**自托管、无人值守、你自己扛后果的个人系统** → 像 Hermes：假设没人、把语义定死、把核验焊进机制。
- 呼应综合①收尾："别问谁做得对，问你的处境像谁"——综合①问的是模型脸，综合②问的是用户脸。

## 6 · 悬而未决 / 防夸大红线

- **[两条轴别硬拧成一条]** 五章不是"一个变量五次显形"（旧脊已弃）。外侧约束（场景/部署）主解 04/05/06/07；08 由内侧约束（训练所有权）驱动。硬说成一条脊就是以偏概全。
- **[在场是症状非根]** "假设有没有人"是场景在时间/环境的显形，也是唯一在轴一露过脸的线（01 桥）；当连接接缝用，别当龙骨。
- **[CC 门控别当默认]** remote-control/channels/teleport/团队编排多为灰度门控——标"意图/可开启"，非出厂姿态。
- **[08 权重级效果是意图]** 两家都证不到"你的使用让它权重级变强"（CC 断在没这能力，Hermes 断在产数据管线不接活会话）；沿综合①/08 原有 caveat。
