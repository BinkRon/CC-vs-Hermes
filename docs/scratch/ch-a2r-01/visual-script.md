# CH-A2R-01 时间章 · 视觉脚本（design-principles Part 2.1 五问）

**1. 一句话论点**：用户离开后，CC 把「你不在」当成一个**需要持续测量、并设法弥合的信号**（响铃→摘要→深链→手机，把你的注意力找回来）；Hermes 把「你不在」当成一个**由语境注定、需要照章办的事实**（你从哪扇门进来就注定在不在，超时=没同意、错过=不补、无事=沉默）。

**2. 最大卡壳点**：「干到一半需要用户拍板」时，两家不是一个动作的两种参数，而是**连组织答案的方式都不同**——CC 按「你此刻注意力在哪块屏」分叉（终端弹窗／自治自决／手机短码），Hermes 按「你从哪扇门进来」分叉（网关等 1 小时／cron 直接摘掉提问工具）。不画出这个「分叉轴本身就不同」，读者会以为只是超时长短之别。→ 这是唯一重档（Hero）。

**3. 叙事顺序**：让渡（问题）→ 定义你不在（第一道分野）→ 四诉求应答（机制主体，Hero 在其中）→ 矩阵+verdict+反思（收束）→ 悬而未决。默认序，不偏离。

**5. 重档草案（Hero）**：
- 形式：**手工 HTML 双面板**（非 stepper——这是「分叉/并置」不是「时序步进」，静态并排更准；也省 JS 引号风险）。左面板蓝框 CC、右面板紫框 Hermes，沿全书色语法。
- 每面板内按「在场程度」纵向排子命运线：
  - CC（左）：① 默认态·你在终端 → 弹选择题、等你回来（amber 等待）；② 自治态·terminalFocus=unfocused（标门控）→ 自己拍板往前走（green 放行）；③ channels·你在手机（标门控）→ 弹窗被禁用、改「yes+短码」（中性改道）。
  - Hermes（右）：① 网关态·你在聊天那头 → 推按钮、阻塞等 1 小时、超时收回执自己继续（amber→继续）；② cron 态·你结构性不在场 → clarify 工具从工具箱摘掉，根本走不到「问」（red 摘除）。
- 0.5 秒该看到：越往下（越不在场），CC 越倾向「自己拍」、Hermes 越倾向「要么等满要么不许问」——两条不同的应对曲线，且分叉轴不同（屏 vs 门）。
- 嵌 3 条真制品（amber 高亮关键短语）：`[user did not respond within 60m]`、`Silence is not consent.`、channels 注释「The multiple-choice dialog would hang with nobody at the keyboard.」
- 图下一句 punch：CC 问「你的注意力在哪」，Hermes 问「你从哪扇门进来」。
