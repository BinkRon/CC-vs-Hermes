# 输入侧：每轮那个请求怎么拼出来

*壳每一轮要现拼一个发给模型的请求包——这个包分三层，你以为每轮全新，其实八成逐字复用。*

> **【TL;DR / 核心洞察】**
> 拼同一个请求包，两家的手艺 八成是重合的 ——都为吃缓存把稳定内容冻在最前，都把每轮变化的东西塞进消息窗口、变了才补，都跑几乎一样的「先删后摘」压缩阶梯。
> 真正的分叉都挤在同几条缝上，而每条缝都在泄露 它到底是什么 ： Claude Code 首先是个 产品 （动态层塞满 IDE、终端、任务的信号）； Hermes 首先是个 面对一群自家开源模型的拥有者 （连冻在最前的『静态』前缀都得按模型分叉，且只有它能把压缩训回权重）。

本章只拆「每轮那个请求包怎么拼、分几层」；循环为什么能一直转下去见 01 章；记忆 怎么写、怎么蒸馏、跨会话怎么演化 的完整生命周期见 08／05 章。

## 你以为每轮是一份新输入，其实八成是逐字搬过来的

这一章不谈抽象的「上下文管理」，只追一件反直觉的事：模型每轮"看到"的那份输入，绝大部分根本不是这一轮写的。

【问题框】
  直觉里，模型每轮"看到"的，应该是一份为这一轮现写的输入。实际恰恰相反——壳每一轮都在拼 同一个请求包 ，而这个包的绝大部分，是上一轮 逐字不变 地搬过来的。于是那个"凭什么"就冒了出来：一个请求包由哪几部分组成， 哪些逐字不变、哪些每轮才换，壳又凭什么这么分 ？更奇的是，两家壳的分法几乎一模一样—— 是什么共同的约束，逼得它们不约而同都这么拼 ？

头一个"凭什么"的答案朴素得意外：钱和速度。给模型发一次请求，最贵的成本是让它把长长的前缀重新"读"一遍；只要前缀一个字节都没变，服务端就能"认出这叠没动过"、直接命中缓存、又快又省（这就是 prompt cache）。于是两家做同一件事：把几乎不变的东西（身份守则、工具目录）死死钉在最前面当"冻结前缀"，把每轮要变的东西全赶到后面去——为的就是别惊动缓存。
所以"每轮那个请求怎么拼"，可以干净地拆成三层：最前面 冻起来吃缓存的静态层 ，后面 每轮现塞进去的动态层 ，以及当整个包快撑爆窗口时才启动的 压缩应急阀 。这一章就沿这三层各走一遍——你会看到两家在每一层都先高度趋同，再在同一处分叉。

## 一次请求长什么样：两家共享的三层装配骨架

先把一次请求摊平成一张图。最该看清的不是每一格写了什么，而是 两家的装配骨架几乎是同一张 ——差异不在骨架，而在骨架的每一层内部（后面三节逐层拆）。自上而下就是发给模型的顺序：冻结的静态层在最前吃缓存，动态层每轮补在后面，压缩阀只在快撑爆时才动。
▲ 请求包顶部 · 最先发给模型 · 最该保持字节不变
▼ 请求包底部 · 最后发出 · 每轮增长
读法： 静态层 冻在最前吃缓存、 动态层 每轮补在后面、 压缩阀 满了才动——这套骨架两家同构。真正的分叉全在每一层内部：静态层里谁的前缀更"静"、动态层里各自塞些什么、压缩层里谁能把它训回权重。下面三节逐层拆。
鼠标悬停任一词条，可见一句话解释。 「三层」是为讲清楚而搭的分析框架，Hermes 代码里恰好也把系统提示词显式分了三层（stable/context/volatile），但两者不是一回事——见 §02 与文末边界。

## 冻结吃缓存是共识，但 Hermes 连『静态』前缀都得看模型下菜

两家都把稳定内容冻在最前面吃缓存，逻辑一模一样。 Claude Code 用一个字面分界符 __SYSTEM_PROMPT_DYNAMIC_BOUNDARY__ 把系统提示词切开，只有它 之前 的部分（身份→任务政策→执行守则→工具用法→语气）跨会话冻结进全局缓存； Hermes 干脆把系统提示词显式设计成三层——stable / context / volatile，拼成一整串、每会话只建一次、之后逐字重放，代码注释直说这是「唯一能让上游缓存保温的办法」。到这里，两家是同一套缓存经济学。
分叉在于—— Claude Code 只面对一个自家托管的模型，这段"静态"前缀就是真·统一的。 Hermes 要伺候一群异质开源模型，它的"静态"前缀 根本不是一整块 ：命中 gemini/gemma 就注入一段 Google 模型执行守则，命中 gpt/codex/grok 换一段，遇到阿里的模型还要专门写一句「你其实是某某模型」去纠正 API 返回的错名字，弱模型名单上的还要再加一段工具强制纪律。 两个不同模型，拿到的"静态层"实质是不同的。
更彻底的一层：连"缓存"这件事，都得挑模型/provider。显式的 cache_control 断点主要给 Anthropic 系、外加少数几个经特定网关的兼容 provider（如 Qwen / MiniMax / 阿里）开着；更多本地/开源模型吃不到，只能退而求其次——靠把消息空白和工具调用的 JSON 规范成"逐字节一致"，去蹭底层推理引擎（llama.cpp / vLLM / Ollama）的 KV 复用。 同一套省钱机制，不同模型享受的档位不同。

【VERDICT】
  接回轴一： 这正是 01 章那根「按模型加压」的独有轴，在装配层的第二次现身——CC 只面对一个模型，前缀能真·统一；Hermes 面对一群模型，连冻在最前面的料、连吃不吃得到缓存，都得看是哪个模型。 所有权与异质性，从纪律层一路渗到了装配层。

## 每轮现塞什么：都塞进消息窗口、变了才补——但塞的东西暴露了产品 vs 底座

冻结前缀之外，才是每轮真正变的东西。两家的 机制 惊人地一致：都不把这些新内容写回那个冻结的前缀（否则缓存全废），而是塞进 消息窗口 —— Claude Code 包成一个个 <system-reminder> 的"合成 user 回合"（模型看得见、你在对话里看不见）， Hermes 直接追加到当轮 user 消息的尾部。而且两家都 只在内容变了才补 ：CC 用一本 readFileState 台账记住每个文件上次的样子、只发变化的，还给这些信号限流（待办/任务每 10 轮、计划/自主模式每 5 轮才补一次）；Hermes 在回合开场一次性预取、整轮复用、不重复查。
分叉在于—— 塞进去的都是些什么 。 Claude Code 每轮可注入的信号有四十多种，绝大多数是 产品面信号 ：被外部或 linter 改动过的文件、IDE 报来的 lint/类型诊断、你在编辑器里选中的行、打开的文件、待办/计划/自主模式状态、钩子输出、伴生宠物、队友信箱…… Hermes 每轮的主注入只有八种左右，都是 底座信号 ：记忆召回、插件 hook、 /steer 打断、几个网关注入。
绝大多数注入都长这样：包在 <system-reminder> 里、标记 isMeta（模型可见、你在对话里看不见），贴在触发它的那一回合旁；少数例外（你排队输入的命令、@提及的文件）走别的形态。
追加到当轮 user 消息尾部；只改这一次调用的副本，绝不写回持久化历史（不污染会话库）。

## 满了怎么办：几乎一样的阶梯，只有一家能把它训回权重

当请求包快撑爆窗口，压缩阀启动。两家的阶梯几乎是同一条，从轻到重：先做不花钱的确定性瘦身（去重/裁掉超大的旧工具结果、换成一行摘要占位），护住开头的任务锚点和最近的现场，只对 中间那一大段 下手；实在不够，才叫一个 LLM 把中段摘成一份结构化检查点。连"谁来写这份摘要"都一样—— 都是外包给一个 LLM （CC 用同一个通用主模型，摘要提示词打头就一句「You are a helpful AI assistant tasked with summarizing conversations」、再接一份 9 段模板；Hermes 默认也用主模型、可选配一个便宜的辅助模型）。 到这一步为止，两家都是纯改壳。
  - ① 工具结果预算 ：单条结果超上限（约 50k 字符）就落盘、正文只留 2KB 预览
  - ② snip-compact ：掐中段、保首尾（ant-only 内部特性）
  - ③ micro-compact ：清 8 类工具的旧结果、留最近 5 条，换占位 [Old tool result content cleared]
  - 一步做完 ：① 相同结果去重（旧的换回指）② 老输出压一行（「命令 → 退出码 → 行数」）③ 截断超大参数，换占位 [Old tool output cleared…]
真正的分叉不在运行时这条阶梯上，而在阶梯 之外 。 Hermes 仓库里另有一条 独立的离线管线 （ batch_runner + trajectory_compressor ），专门把跑完的任务轨迹压缩、做成训练数据，教模型"看到一段压缩摘要后照常继续干活"。铁证不在运行时那条阀里，而在这条离线管线本身：它把跑完的轨迹压成训练样本，还往训练数据里注入一句 notice，直白告诉模型「你之前的一些工具响应可能被摘要过」——教它把「被压过的历史」当成正常。
这条离线管线把跑完的任务轨迹压成训练样本， 训练样本本身就在教模型：看到摘要，别停、接着干 。它产出的摘要用 [CONTEXT SUMMARY]: 标记——正是运行时压缩器 至今仍认得 的 legacy 格式（ context_compressor.py:71 ；运行时如今实际吐的是更长的「 [CONTEXT COMPACTION — REFERENCE ONLY] 」交接前缀）。两者不是"同一个字符串"，但 同源 ：运行时保留着对训练标记的识别。这是 CC 结构上不可能有的东西——它不拥有模型，压缩这件事只能一路改壳到底。

「更细」在哪一眼可见： 橙色 那几节是 Hermes 拆得比 CC 更碎的地方——它把「约束/偏好」「已答问题 vs 未答请求」「进行中 vs 待完成」「相关文件」都单列，还专门把一批标成「Historical（历史·别再去做）」，因为它更怕弱模型压缩后把旧任务当成新指令。
就在你眼前的证据： 本文档这次会话开头那份「Primary Request and Intent / Key Technical Concepts…」的 summary，正是这个 prompt 的产物。
对照 CC：CC 靠画一条边界丢掉原文；Hermes 额外用这段长前缀反复叮嘱「这是历史、别重做、只听最新消息」——同样是怕弱模型压缩后误把旧任务当新指令。

## 记忆每轮怎么进请求：一个当便签，一个当底稿

记忆也是每轮装配的一部分，但两家塞法不同，正好照出两种记忆观。 Claude Code 每轮对你这句话做 语义召回 、最多挑 5 条相关记忆，包在会标注"存于几天前"的 <system-reminder> 里（而且这条每轮召回 本地默认是关的 ，要服务端开旗才生效），并反复叮嘱"用前先读当前文件验证"——它把记忆当成 会过期的便签 。 Hermes 则把整本 MEMORY.md 和 USER.md 原样 冻进系统提示词的易变层、在压缩交接里标为 authoritative（不被摘要覆盖），外部记忆插件召回的片段也裹着一句「Treat as authoritative reference data」——它把记忆当成 不该被冲淡的底稿 。一个防陈旧，一个保权威。
这里只讲"记忆怎么进 这一轮请求 "这一个切面；记忆 怎么写、怎么蒸馏、怎么跨会话演化 的完整生命周期，是轴二的事——见 08「Loop 与自己」／05「Loop 与用户」。本章不重复展开，以免与后文双写。

## 装配是共享手艺，分叉泄露身份

三层都先趋同、再在同一处分叉 ：静态层都冻结吃缓存，动态层都塞进消息窗口且变了才补，压缩层都跑"先删后摘、外包 LLM"的同一条阶梯。请求装配这件事，八成是两家共享的工程手艺——别把它想成两套哲学。
但三条分叉缝，是两个不同的根因，不是一个 ：静态层的分叉（前缀按模型分叉、缓存挑模型家族）和压缩层的分叉（只有它能把压缩训回权重），根子都是 模型的所有权与异质性 ——呼应轴一脊椎；而动态层的分叉（四十多 vs 八种注入），根子是 产品 vs 底座 ，是另一回事。我 没有 把三层硬收进一个漂亮的单一根因——那样会把动态层那条缝掰歪。
所以两家在输入侧的"身份"是 ： Claude Code 首先是个 产品 （它的动态层为 IDE/终端/任务而生）， Hermes 首先是个 面对模型动物园的拥有者 （它的静态层为异质模型而分叉、它的压缩层为自家权重留了训练接口）。同一个请求包，装法八成一样； 不一样的地方，恰好各自暴露了自己是什么。

## 悬而未决：八条别被主线吞掉的边界

【悬而未决 / 防夸大】
  - [边界] "CC 约 44 种注入"是 满配上限 ，外部默认 build 远没这么多——snip、cached-microcompact、每轮语义记忆召回、验证代理等为内部灰度或默认关；Hermes 的 8 种也含 MoA/prefill/网关注入等只在网关模式出现的项。 别把"能注入的种类"当成"每轮都注入"。
  - [HM] Hermes 那层叫"volatile 易变"的其实 名不副实 ：记忆快照/USER.md/时间戳虽叫易变，却冻在缓存的系统提示词内部，只在压缩/重建时才刷新，并非每轮变；真正每轮变的在消息窗口。所以"静态/动态"并不等于"系统提示词/volatile 层"。
  - [CC] "静态前缀"也不是整个系统提示词：分界符之后的记忆索引/环境/语言/输出风格虽在同一个 system 数组里，却按会话变、不进全局缓存。CLAUDE.md 更是根本不在系统提示词正文里，而是搭在一条 合成的首条 user 消息 上；git 状态又落在系统提示词尾部——三样东西分居三处。
  - [CC] prompt cache 其实有 三套模式 （自家全局缓存 / 含 MCP / 第三方），断点位置和块数都不同；而且"缓存断点检测器"只是事后监控 break 原因、不负责放断点。"缓存断点打在哪"没有单一答案。
  - [边界·压缩] CC 的压缩不止"整段摘要"一种：默认的 auto-compact 确实丢中段原文，但还有 partial 压缩（保留首尾原文消息）、session-memory、context-collapse 等路径（多为手动或特性门后）。"CC 只会整段摘要"只在默认路径成立。
  - [HM] 别把运行时压缩器和训练管线当成同一套：它们是 两个独立模块、代码不共用 ，也 不吐同一个字符串 ——运行时如今实际吐「 [CONTEXT COMPACTION — REFERENCE ONLY] 」长前缀，训练管线吐「 [CONTEXT SUMMARY]: 」，后者只是运行时 至今仍认得的 legacy 格式 。这道"同源"痕迹能证的是格式同源与管线存在， 证不到 线上权重真训过——第二根杠杆的权重级效果是意图、非本仓库可证。
  - [HM] Hermes 的请求其实有 四套 api_mode （chat-completions / codex / anthropic / bedrock），序列化路径各不同，本章讲的是默认 chat-completions 那条；给 GPT-5/Codex 的系统提示词还会以 developer 角色而非 system 角色发出。"一次请求"并不是单一形状。
  - [边界] 压缩触发阈值两家都是 派生值 、不是固定百分比：CC≈有效窗口−13k（200k 模型约 83%），Hermes≈有效预算 50%、地板 64k、小窗退到 85%。数字差别背后是"多晚才肯动手"的不同保守度——Hermes 明显更早出手。
