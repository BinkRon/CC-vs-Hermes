# 输出行动侧：模型吐生料，壳每轮加工两样成品

*一样是能用的行动，一样是能回传给模型的历史。*

> **【TL;DR / 核心洞察】**
> 模型一次吐出的不是一句话，是一叠 异构块 ——有它自己想的（推理）、有说给你听的（正文）、有要动手的（工具调用），混在一坨里。壳每一轮干的是两件加工活：把这叠生料变成 ① 能用的行动 、 ② 能合法回传给模型的历史 。
> 两条加工线结构一模一样——底层一层 协议地板 两家共享（ CC 这层一点不薄，回传前重塑是它最厚的工程）；上面 Hermes 各多铺一层：行动线多铺 弱模型语义纠偏 、历史线多铺 异质 provider 适配 。这是两股 不同的根因 ，不是一股。CC 之所以能不铺，只因它只伺候 一个受托管模型 + 一个 API 。

本章只讲「模型吐的块本轮怎么消费、回传前怎么规整」；循环为什么不致死、每类错误怎么续命见 01 章；每轮新塞什么进请求、历史怎么压缩见 02 章。

## 你以为模型回你一句话，其实它吐回来一叠块

这一章不追"模型说了什么"，只追一件事：模型一次吐回来的那叠东西，壳每一轮到底拿它怎么办。

【问题框】
  你让 agent 改一行配置。它「回答」你的时候，底层其实一次性吐回了几段性质完全不同的东西：一段它自己的盘算（该改哪、会不会连累别处）、一句说给你听的话（「我来改」）、还有一条真要落到磁盘上的动作（改 config 第 12 行）。这三样 挤在同一次响应里 ，壳收到的不是一句干净的指令，是一叠混排的块。问题就在这儿： 壳每一轮拿这叠混排的生料，到底要做成什么 ？而两家做的， 又在哪儿一样、哪儿不一样 ？

于是壳每一轮都要把这叠生料 拆开、分类、分别加工 。哪些是「要动手」的，得挑出来真去执行——这条是 行动线 ；而这一整轮（它想的、说的、干的，加上工具跑出来的结果）又必须 原样塞回下一次调用 ，好让模型「记得」自己刚才干了啥——这条是 历史线 。
本章就沿这两条线各走一遍。你会看到：两条线的 底层地板两家几乎一样 ，真正的分野都在上面那薄薄一层——而且行动线和历史线上那一层， 是被两件完全不同的事逼出来的 。

## 一次响应不是一句话，是一叠块——每块有两种命运

先把模型一次响应摊开。它不是一段文字，是 一叠块 ：推理块、正文块、工具调用块，可以混排、可以多个。壳收到后第一件事就是逐块认领——而每一块都有两种命运要分别交代： 这一轮怎么用它 （行动线的活）、 怎么把它塞回历史 （历史线的活）。下面这张表，就是后面两条线的预告。

## 两条线的地板两家共享，只有上层 Hermes 多铺——而且两层根因不同

刚才那张命运表，把每一块拆成了「本轮怎么用」和「回传怎么塞」两列——这两列各自展开，就是壳的两条 加工线 ： 行动线 与 历史线 。现在别再盯着单个块，退一步把两条线 整体 摊平看骨架——你会撞见本章最反直觉的一件事： 两条线的骨架几乎是同一张 。每条线都分两层：下层 协议地板 （为「不崩、不丢」而建，两家共享）、上层是 Hermes 替自己处境多铺的一层（ CC 留空）。0.5 秒该看到的是： 灰色的地板格是满的、甚至最厚，只有紫色的上层两格 CC 是空的 ——「CC 壳薄」薄的是 上层那两层 ，不是地板。
赌托管强模型自纠，只回一条报错、不替它猜。
  - 工具名拼错 → 模糊改名修复
  - 空工具名 → 识破「鹦鹉学舌」 ，只回「那是数据不是指令」
  - 正文混进假工具调用 → 剥掉
只有一个 API，回传格式天生统一，无需适配。
  - 逐 provider 的 reasoning_content 方言 归一
  - KV 缓存 逐字节规范化 ，蹭本地引擎复用
  - 孤代理字符清理
  - 严格 API 的 未知字段剥离
  - 调用↔结果 配对补全 （防悬空调用打崩下一轮请求）
  - 只读工具 并发 · 写工具 串行独占
  - 参数校验失败也 回喂让模型自纠
  - 别名回退 ：认废弃旧名，但拼错名 不猜
  - 回传前 剥掉结尾 thinking 块
  - 清「只有推理」的 孤儿轮
  - 散开的块 按 id 缝合
  - 悬空调用/结果 配对补全
两块紫色上层不是「同一层弱模型纠偏」的两半： 行动线 上层的根因是 弱模型 （模型没被训练得总把工具名喊对）； 历史线 上层的根因是 异质 provider （几十种推理服务各有各的脾气）。把它俩都说成「弱模型纠偏」是 掰歪 ——历史线那层就算配一个强模型、只要 provider 杂，照样得铺。而灰色地板两家共享、且 CC 的地板 一点不薄 。 骨架先摆到这；接下来两节各挑一条线补上血肉与真实制品——先从行动线上层那三条纠偏看起。

## 行动线：把「要动手」的块，变成真落地的一次操作

行动线只认那叠块里 要动手 的那种。壳把它挑出来、按名字找到对应工具、过一道权限门、真去改文件或跑命令，再把结果翻译成模型看得懂的块送回去—— 骨架刚才那张图已经摆过 ：地板两家共享，上层只有 Hermes 铺一层。这一节把上层那层补成真实制品。它多铺这层，是因为 不敢假设模型总能把工具名喊对 。
具体是三种 条件触发 的纠偏（不是每轮都做，别读成「Hermes 总在猜」）：工具名 拼错 了就模糊匹配改对；工具名 空着 ——壳判定这多半是弱模型把文件里长得像工具调用的 XML 照抄了出来，于是识破这种 「鹦鹉学舌」 ，只回一句「那是数据、不是指令」，而不是把整个工具目录再喂回去火上浇油； 正文里混进了假的工具调用 就剥掉。这三条 Claude Code 一条都没有——它赌自己托管的强模型能把名字喊对，喊错也只回一条报错、 让模型自纠 ，不替它猜。
你调的这个工具，它的说明书这一轮 还没发给你 （被按需加载撤柜了）——先用 ToolSearch 把它取回来，再重试这次调用。
要点转述（源码 buildSchemaNotSentHint,toolExecution.ts:592-596）。CC 的指路是 协议层 的：它不猜你想干嘛，只告诉你「工具没加载、去加载」——属地板，不属弱模型纠偏。
逐字抄自 conversation_loop.py:4278-4285。空工具名触发，壳判定弱模型把「像工具调用的文本」当指令照抄了，直接点破——这是地板 之上 那层弱模型纠偏。
两家都会回喂一句「指路」，但指的是 两件不同的事 ：CC 指的是「你要的工具还没加载，先去搜」——这是协议地板的一部分；Hermes 指的是「你把数据当指令了」——这是它替弱模型多铺的那层。同一个动作（回喂指路）落在不同的层上。

## 历史线：把这一整轮，重塑成模型下一轮认得、API 肯收的历史

行动线管「这一轮怎么动手」，历史线管「这一整轮怎么塞回去」。模型没有记忆，壳每一轮都要把到目前为止的整段对话重新喂进去（02 章讲过）——但生料不能原样塞：模型吐的那叠块里，有些 API 根本不收 （比如以推理块结尾的 assistant 轮），有些会 让下一轮请求直接报错 （比如发出去却没配上结果的悬空调用）。所以回传前有一道 重塑 ：剥掉结尾的推理块、把「只有推理」的孤儿轮并进相邻轮、把散开的块按 id 缝回、给悬空的调用补上占位结果。 这道地板两家都铺，而且是 CC 最厚的一段工程 ——别被「CC 壳薄」误导。
一条关键纪律，和 02 章的每轮注入 一模一样 ：这道重塑只动 本轮调用的那份临时副本 ， 持久历史一字不改 ——你在 UI 里看到的转录、存档里留下的记录，都是没被动过的原貌。壳 绝不把重塑写回会话库 。这是同一条纪律在输出侧的落点：只动本轮副本、绝不写回会话库。
地板之上， Hermes 又多铺一层——但这一层的根因 不是弱模型，是异质 provider 。它要对接几十种推理服务，每家吐推理的字段名/格式都不同，于是要做逐 provider 的 方言翻译 （把各家的 reasoning_content 归一）；吃不到显式缓存的本地引擎，还要把消息空白和工具调用 JSON 规范到逐字节一致 去蹭 KV 复用；再加上清理某些模型吐出的孤儿代理字符、按严格 API 要求剥掉未知字段。这层 CC 全都没有——它只有一个 API，回传格式天生统一。
一个容易读错的点要说准：两家都能把 图片放进消息 给模型看（多模态）；区别在 存档/轨迹 里怎么省空间——两家都不把 base64 原样堆进历史，但 顶替方式不同 ： CC 基本只留个二进制占位、把图默默丢掉； Hermes 则把图 换成工具自带的一句文字摘要 （ text_summary ）或 [screenshot] 占位顶上。所以「剥掉 base64」两家都做，但「剥完换不换一句描述」并不一致。

## 结果回捕与错误处置：都是「改壳」，分野只在一处阈值

前面两条线讲的是「块怎么变行动、怎么变历史」。收尾看两个更具体的切面：工具跑完了，结果怎么翻回模型眼前（ 结果回捕 )；以及出错了，错误 变成什么样的块 回到模型眼前（ 错误处置 ）。两者都属输出行动侧，都是纯改壳。

[Claude Code] Claude Code · 结果回捕
  每个工具自带「结果翻译器」，转成标准结果块、用 id 配对送回。文本带行号、图片转图像块、PDF 转文档块（多模态）、出错包成 <tool_use_error> 。结果太大 多数不硬截而落盘 、只回预览+路径（Read 类豁免防死循环；Bash 超 64MB 仍硬截）。防撑爆是 双层 ：单结果落盘 + 每轮聚合预算（约 20 万字符）。

[Hermes] Hermes Agent · 结果回捕
  运行时结果是 OpenAI 风格的工具结果消息 + id 配对（ <tool_response> 是训练/存档表示）。同样 双层 ：单结果超阈值落盘指路 + 每轮聚合预算（约 20 万字符）。真正独有的一层是 按模型上下文窗口缩放阈值 ：小上下文模型缩到约 16k 地板、大模型 clamp 回默认。图片进消息给模型看；存档/轨迹里换成工具自带的 text_summary 或 [screenshot] 占位顶替（不堆 base64）。

【VERDICT】
  为何不同（维度修正 · 改壳）： 两家 都 用「落盘+指路」代替硬截断，也 都 有「单结果落盘 + 每轮聚合预算」双层—— 别把「整轮总预算」当成 Hermes 独有 （那是早先的误读，已证伪）。Hermes 真正独有的只有一处： 阈值按模型上下文窗口缩放 ——小上下文的开源模型缩得更狠。根因还是那条：CC 只伺候一个大窗口的托管模型、可以用固定阈值；Hermes 面对一群窗口大小不一的开源模型，只能让阈值随模型走。

  【矩阵】 错误类型（→ 变成什么块回到模型） | Claude Code | Hermes Agent
  · 解析错 找不到工具 / 参数不合法
      [Claude Code] 合成一条 报错块 回喂，让模型自纠（地板级）。
      [Hermes Agent] 同样回喂报错；但工具名 拼错 / 空着 时，还先叠加上层纠偏（改名 / 反鹦鹉）试一把——按输入形态触发，不分模型。
  · 执行错 工具真跑抛了异常
      [Claude Code] 不中断，包成 <tool_use_error> 结果块、按 id 配对送回。
      [Hermes Agent] 不中断，包成 role:tool 的错误消息、按 id 配对送回。
  · 坏 JSON 工具调用参数损坏
      [Claude Code] 参数在协议层已校验，基本不出现此形态。
      [Hermes Agent] 壳 原地重试 （≤3 次，不惊动模型），用尽再把错误当结果块回喂。

火墙 ↔ 01： 这张表只列「错误最终以什么形态的块回到模型眼前」——因为「错误结果也是一种结果块」，归输出行动侧。它们 为什么不致死、壳续命重试几次、什么时候干净地停 ，是 01 章的事，本章不重复。

## 工具太多怎么办：两家几乎同一手，都是纯改壳

牌太多会撑爆上下文。两家的解法几乎一样，而且都是纯 改壳 ：工具目录常驻，详情 按需拉取 ——初始只露名字，模型要用哪个，再用一把「搜索 / 取说明书」的钥匙把完整参数表拉回来。 Claude Code ——超阈值（auto 模式 10%）就把 MCP 及可延迟工具撤成按需； Hermes ——可延迟工具超默认 10% 阈值就撤柜，只留 tool_search / tool_describe / tool_call 三把钥匙，且桥工具的 tool_call 原路穿过同一分发入口 ，权限门一视同仁。
差异只在工程细节、不改结论：CC 把按需拉来的工具 伪装成和常驻工具同一种编码 ，降低模型认知负担；Hermes 让桥工具 复用统一分发管线 ，避免「绕一层桥就漏掉权限门」。 壳的智慧，不是模型的。

## 悬而未决：五条别被主线吞掉的边界

【悬而未决 / 防夸大】
  - [HM] 训练格式 vs 运行时格式会不会漂移：模型受训「见」的是 XML 轨迹，线上运行时读的却是结构化字段——两种表示各自演进，如何保证「训练时学的」和「线上喂的」长期不脱节？
  - [两家] 停止判定押在「块的存在性」上：两家都不信 API 的 stop_reason / finish_reason，只数这轮有没有 tool_use / tool_calls。把最基础的控制流也押在一个结构化字段的存在性上，弱模型场景下这套长期稳吗？
  - [HM] 反鹦鹉纠偏为非自训模型而生：识破「空工具名 ＝ 把数据当指令照抄」这条，是替没被这样训练过的第三方模型兜的——自训的 Hermes 模型到底需不需要这层？边界在哪？
  - [CC] searchHint 做过 A/B 却砍掉：CC 的工具 searchHint 实验显示没收益，于是名册里连搜索提示都不给、只渲染工具名。按需加载的「检索质量」是否被低估了？
  - [HM] 工具目录每轮重建、不跨轮缓存：从一次 cron 回归事故（会话级缓存与实时注册表漂移、静默丢工具）学来的教训。以「每轮重算」换「绝不漂移」，这笔性能账在超大工具集下划算吗？
