# Loop 与环境：护栏趋同，分野在闯了祸算谁的

*兜底不是把模型变聪明，是让「模型不聪明」也不致命。*

> **【TL;DR / 核心洞察】**
> 环境这一侧，两家的安全机制 惊人地趋同 ：默认都不把 agent 关进沙箱、动作直落你的真机器；都把「隔离」和「审批」当成互相 替代 的两条路；都留了一道 连全放行都压不过的绝对地板 ；被拒后都塞给模型一段几乎同款的「别耍花招绕过」训诫。趋同不奇怪——兜底要防的是「模型会犯错」这个普适前提，下限设计的答案空间本就窄。 真正的分野不在机制清单，在两处。一是那道绝对地板，两家用了 两套不同的判据 ：Hermes 按「物理可不可挽回」护住 机器 ，CC 按「动没动管着 agent 的那层」护住 规则本身 ——同一个 rm -rf / ，一家无条件拦、一家全放行可放行。二是 闯了祸算谁的 ，以及由此逼出的 对能力边界多诚实 ：越是没有厂商接盘、越是你自担，它越被迫在文档里亲口写明「这不是墙」。

## 00 · 问题：动作落在真机器上，兜底该怎么分类

【问题框】
  agent 干着干着，要删一整个目录、要强推覆盖远端分支、要读你家目录里的密钥、要连一个陌生外网地址。这些动作不在沙盒里排练，是真的落在你的电脑上——删了就是真删了，推了就是真推了。
  于是壳（harness）要替它兜底，而且兜的不是「让模型想得更清楚」（那是 Loop 之内的事）——是 万一模型没想清楚、做错了，能不能保住下限、让代价可控 。这就带出这一章的问题： 如果损失真的发生，会是哪几类？针对每一类，壳把防线摆在了哪一步——动手前拦、动手时隔、还是闯了祸之后兜？
  把两家的答案摆进一张「损失类型 × 防线阶段」的网里，会撞见一件反直觉的事： 两家填格的方式几乎一样，真正的差别只藏在少数几个格子里。

## 01 · 先看总览：防线几乎全压在「事前」，隔离与兜底大多是空格

_纵轴是「祸可能落在哪」的五类损失，横轴是防线站在祸的哪一步。每格标两家 默认形态 下的填充度——不是「有没有这个功能」，是「开箱默认拦不拦」。_

  【覆盖网 · 损失类型 × 防线阶段】列： 事前 · 拦 / 事中 · 隔 / 事后 · 兜
  · 资产毁损 文件／数据没了
      〔事前 · 拦〕CC=●满 弹窗＋删除拦 ; HM=●满 硬地板＋软清单
      〔事中 · 隔〕CC=○空 沙箱默认关 ; HM=○空 默认裸主机
      〔事后 · 兜〕CC=◐半 rewind·仅编辑 ; HM=○空 rollback 默认关
  · 机密外泄 密钥被读走＋传出
      〔事前 · 拦〕CC=○空 近空白 ; HM=◐半 扫描·读无拦
      〔事中 · 隔〕CC=○空 网络闸随沙箱 ; HM=●满 剥密钥·默认开
      〔事后 · 兜〕CC=○空 — ; HM=○空 —
  · 失控·二阶 治理层本身被改写
      〔事前 · 拦〕CC=●满 配置连全放行也拦 ; HM=◐半 改配置在软层·可绕
      〔事中 · 隔〕— 结构上无「事中」对应
      〔事后 · 兜〕CC=○空 — ; HM=○空 —
  · 机器失能 关机／删根／炸弹
      〔事前 · 拦〕CC=◐半 删根软拦·关机无专项 ; HM=●满 硬地板·无条件
      〔事中 · 隔〕CC=○空 — ; HM=○空 —
      〔事后 · 兜〕CC=○空 — ; HM=○空 —
  · 外部世界后果 强推／发消息／发布
      〔事前 · 拦〕CC=○空 仅信息提示 ; HM=○空 发消息零审批
      〔事中 · 隔〕CC=○空 — ; HM=○空 —
      〔事后 · 兜〕CC=○空 已发不可撤 ; HM=○空 已发不可撤

_这张网一眼给出三件事。 其一，防线几乎全挤在「事前」列 ——两家都把赌注押在「动手前把危险动作拦下来问人」。 其二，「事中」列默认几乎全空 ：沙箱、容器隔离两家默认都不开，隔离不是默认防线、是要你自己去开的加固；而且它一开就是 替代 审批而非叠加——CC 开了沙箱，命令反而默认不再逐条问；Hermes 后端换成隔离容器， 整条审批链（含硬地板）直接短路 （碰不到宿主机就不必拦）。 其三，「事后」列几乎全空 ：两家都只有一格半——各自给「自己动笔改的文件」做了原生撤销（CC 的 rewind 默认开、Hermes 的 rollback 默认关），但 bash 层的删除、外泄、已推 commit 一律兜不住，只能回落到 git 或操作系统。_
_趋同到什么程度？连「动作被拒之后对模型说的原话」都近乎逐字重合——一个托管闭源、一个开源自托管，却都在闸门后面补了同一句训诫： 不许换个命令、换个说法把同一件事再做一遍 。_

  ┌─ 制品 · src · 被拒时的话（超时再追加一句）
  │ The user has NOT consented to this action.

_这是趋同的最强证据，也点破了危险治理最脆弱的一环：光有闸门不够，还得堵死模型「被拦一次就换姿势重试」的动机——所以两家都把「拒绝」从一堵冷墙，改成一次有引导的协商。 机制层的趋同到此为止。差别，要到那道「连全放行都压不过」的地板上才显形。_

## 02 · 分野一：同一道绝对地板，两家用了两套判据

_两家都给了一个「我盯着、你放手干」的最高档（CC 的 Bypass Permissions、Hermes 的 yolo），也都留了一道 连这个档都压不过 的绝对地板。有意思的是：拿两个动作去撞这两道地板——一个是删根（ rm -rf / ，物理不可逆），一个是改 agent 自己的治理配置（改 .claude ／改审批规则，元级失控）—— 两道地板恰好落在对角线上 。_
_两块 amber 高亮的硬地板落在 对角线 上，这就是分野： Hermes 的判据是「物理可不可挽回」 ——它的地板只收「没有恢复路径」的动作（删根／格盘／写裸设备／fork 炸弹／关机），护的是 机器 ； CC 的判据是「动没动管着 agent 的那层」 ——它连全放行都压不过的是治理配置，护的是 规则本身 。各自把对方设成绝对地板的那类，恰好留在了自己能被放手档盖过的软层。_

  └─

[Claude Code] Claude Code · 地板为什么护配置：注释写的是防提权

  ┌─ 制品 · src · bypass 免疫路径清单 + 缘由
  │ .git / .claude（自身配置）/ .bashrc·shell 启动脚本 / .vscode·.idea（IDE 配置）

  │ // to prevent VS Code settings manipulation ...
  │ // to prevent shell startup script manipulation
  └─

【VERDICT】
  同一个动作、两种恐惧： 这才是分歧点——不是「谁护谁不护」（灾难命令两家 都 查、治理配置两家 也 都拦），而是 把哪一样升格成连全速前进都不可绕过的绝对地板 。Hermes 怕的是物理不可挽回，CC 怕的是元级失控。两道硬地板刻意都 极小 ：只收「没有恢复路径」的动作；像 rm -rf /tmp/x 、 curl|sh 这种「代价大但可恢复」的都留在软层、放手档一开就放行——别把软清单也当成不可绕过的墙。

## 03 · 分野二：机密外泄这一行，一家做了减法却亲口否认那是墙

_在覆盖网里，「机密外泄」这一行两家的填法正好相反——而这处相反，把「闯了祸算谁的」如何反过来决定「它对自己诚实到什么份上」摆到了明面。_

[Claude Code] Claude Code · 基本敞开：赌你在可信环境
  子进程继承全部密钥 （不做 env 清洗）、 读密钥文件无专项拦 （ ~/.ssh ／ .env 只受「出没出工作区」的通用约束）、 网络外发的闸只在沙箱里成立 ——而沙箱默认关。唯一一处密钥专拦（写团队记忆时扫密钥）还是特性门控。默认形态下，外泄这一行几乎没有防线。

[Hermes] Hermes · 主动剥密钥：却在文档里说这不算隔离
  默认就剥密钥 ：把 API 密钥、网关令牌在传给子进程前先黑掉（GATEWAY 令牌无条件剥）。这是覆盖网「事中」列里两家唯一非空的一格—— 但它自己在安全文档里主动降级了这道护栏 。

  ┌─ 制品 · SECURITY.md · 对自己能力边界的自白
  │ （剥密钥）This reduces casual exfiltration.
  │ It is not containment.
  │ Any component running inside the agent process ... can read whatever the agent itself can read ...
  │ （总纲）The only security boundary against an adversarial LLM is
  │ the operating system
  │ . Nothing inside the agent process constitutes containment ... They are useful.
  │ They are not boundaries.
  └─

【VERDICT】
  诚实度是「风险归谁」的下游： Hermes 既动手剥了密钥、又亲口否认那是边界——不是自相矛盾，是它的处境逼出来的坦白。它开源自托管、从开放渠道吃不可信输入、常常无人值守， 没有厂商接盘、闯祸是你自担 （文档里连 bug bounty 都明说没有、部署加固全列为运维的责任）。正因为残余风险最终落在你头上，它才被迫把「真边界在哪」说死，免得你把一道软门错当成真墙。CC 反过来：多半跑在你自己可信的仓库里、你就在旁边、身后还有厂商，于是它能用更轻的姿态——「 只对你信任的代码用我 」——把外泄这一行的残余风险，交还给「你选对了输入」。

## 04 · 共同盲区：两家的护栏都只盯着「本机」

_覆盖网里有两处，是两家 一起 留白的——不是谁的取舍，是这类产品共同的射程边界，对读者判断「我能把什么托付给它」尤其要紧。_
_盲区一：祸落在别人头上的那一类，两家事前几乎都不设防。 强推覆盖共享分支、往外发消息／邮件、发布 npm／pip 包——这些「外部世界后果」在覆盖网第五行几乎全空。CC 这边，force push 只有一句信息提示（不影响放行）、npm publish 要靠你自己写规则、系统提示里「不主动 push」是 写给模型的纪律、不是代码闸门 ；Hermes 这边，agent 主动发消息／发邮件 零审批 ，publish 无拦，只有 force push 被某条 shell 正则偶然覆盖到。两家的防线都画在「别毁了这台机器、这个仓库」，对「别把祸甩给别人」基本沉默。_
_盲区二：护栏几乎只长在「shell 命令」这一个面上。 那套危险删除拦截、配置保护、沙箱，两家都只覆盖终端命令和文件编辑。 CC 的 MCP／自定义工具直接放行 （passthrough，完全不过环境侧的路径闸和沙箱）； Hermes 的 MCP 调用、发消息、发布 也都绕开命令闸，只靠一个 默认关（opt-in） 的通用钩子兜——插件默认不加载，非 shell 面开箱即无闸。含义很实： 一个改用 MCP 工具去删数据、去联网外传的 agent，能绕开两家几乎全部的环境护栏。_

【VERDICT】
  为什么一起留白： 两家的环境治理，本质是 「本机资产治理」 ——把 shell 能对这台机器、这个仓库干的坏事拦住。一旦动作换个出口（对外发送、发布、经由 MCP 工具落地），或者后果换个落点（砸在协作者而非你自己头上），就掉出了射程。这不是 bug，是这一代 harness 危险治理的形状：它守的是「你的机器」，不是「你的机器能波及的世界」。

## 05 · verdict · 机制趋同，账本分家

【VERDICT】
  把这一章收成一句：环境侧的安全机制之所以惊人趋同，是因为它们都在给「模型会犯错」这个普适前提做下限设计——答案空间窄，就收敛到同一批动作上（拦、隔、兜）。真正把两家分开的，是账本。 第一本账：那道绝对地板护什么。 Hermes 按「物理可不可挽回」护机器，CC 按「动没动治理层」护规则本身。一个防的是物理上收不回的动作，一个防的是能改写游戏规则本身的动作——各自把对方留在软层的那一样，单独升格成了硬地板。 第二本账：闯了祸算谁的。 CC 厂商托管、多半跑在你自己可信的仓库里、身后有 git 兜文件、有企业总闸能远程锁死全放行档；Hermes 开源自托管、你的机器你的账号、没有厂商接盘、责任白纸黑字归运维。风险归属的不同，又反过来逼出了「对能力边界多诚实」的不同：越是你自担，它越被迫把「真边界只有操作系统」说死。 一个共识别漏掉： 两家其实都在对你说「别全信我」，只是指向了不同的出口——CC 说「只对你信任的代码用我」，Hermes 说「进程内没有任何东西算隔离，唯一边界是操作系统」。是「厂商托管、身后有人兜」还是「自托管、你自担」，决定了它把最后那道防线画在哪，也决定了它对自己诚实到什么份上。

## 06 · 边界说明（防「默认→绝对」以偏概全）

【悬而未决 / 防夸大】
  - [两家 · 默认不隔离] CC 命令沙箱 默认关 、开箱即用全靠权限弹窗；Hermes 后端 默认 local ＝ 裸主机 ，容器／远程／云沙箱全是 opt-in。别记成「它们在沙箱里跑命令」——恰恰相反，默认不关。且沙箱一开是 换范式 （用隔离换掉逐条问）不是加一层。
  - [两家 · 事后只兜半格] 「事后兜底全外包给 git」不成立：两家都有原生撤销，但都只覆盖 自己经结构化编辑工具改的文件 ——CC 的 rewind 交互默认开、Hermes 的 rollback 默认关（v2 从开翻成关，注释「多数人从不用」）；bash 层的 rm／dd／重定向、已推 commit、已发外部动作，两家事后都兜不住。
  - [CC · 不对称] 「删根被硬拦」只对 PowerShell 成立：同样删系统关键路径，PowerShell 返回硬拒（bypass 也压不过），Bash 却只是弹窗问（措辞「不能被规则自动放行」，Bypass 档仍可放行）。全仓没有硬编码的危险命令黑名单。CC 护栏强度按「走哪个 shell／碰没碰治理层」排，不按「动作可不可逆」排——这是护栏不一致，不是设计美学。
  - [HM · 诚实漏洞] 既非交互、又非网关、又非 cron 的无人会话，软危险命令会被 直接放行 （本切片主路径连告警都不打，静默放行）——官方 docstring 明文保留这条历史 fail-open。硬地板／sudo 地板／deny 清单在放行前已拦，故缺口只对「可恢复的软危险命令」开。但「无人即更保守」只在被识别的 cron／网关场景成立：cron 无人默认拒，这条非 cron 无人却默认放行，两个无人路径处理不一致。
  - [HM · 别当默认支柱] 内容扫描器 Tirith 默认「挂了就放行」（fail-open） ；smart（小模型审查）与 off 都是 opt-in、审批默认 manual。硬禁地板是全文里 唯一 真正无条件的机制，其余大多是可配的默认值。
  - [CC · 门控能力别当产品] 用 AI 分类器代替人审的 auto 模式、团队记忆密钥拦，均为内部特性门控，外部用户默认不可达；harness 自身的暂存／计划／记忆目录即便在工作区外也 静默可写 （只限这些后台目录、不含任意用户文件，「出圈就问」的大结论仍成立）。
  - [两家 · 子代理不自动隔离] git worktree 隔离两家 都是手动 opt-in ：CC 明确「只有用户说 worktree 才用」，子代理默认共享同一工作目录；Hermes 的 worktree 也要显式开。「并行分身各自隔离」不是默认形态。
  - [不评优劣 · 时间切片] 本篇讲的是危险治理的 形态 与 账本 ，不评两种模式孰优——取决于你是否在场、自托管还是托管、吃的是可信还是开放输入。两家治理都在快速演进，本篇是某一时间切片（CC main / Hermes @8e734810d）。
