## 仪器速览

**行 = 祸可能落在哪（损失类型）**

| # | 损失类型 | 定义 | 特殊性 |
|---|---|---|---|
| 1 | 资产毁损 | 文件/数据没了 | 内分可恢复（repo 内 git 能捞）/ 不可恢复（repo 外、格盘）|
| 2 | 机密外泄 | 密钥/隐私被读走+传出 | 天然不可逆；唯一需「读+传」两步组合成祸的类型 |
| 3 | 失控（二阶损失）| 治理层本身被改写：审批规则、自身配置、shell 启动 | 一旦发生，其它所有防线同时作废 |
| 4 | 机器失能 | 关机、fork 炸弹、删根 | 对无人值守常驻体有乘数：机器死=agent 死=通知通道死 |
| 5 | 外部世界后果 | 强推共享分支、往外发消息、发布包 | 祸落在别人头上，社会性不可逆 |

**列 = 防线站在祸的哪个阶段**

- **事前（拦）**：弹窗问人、允许/禁止清单、硬地板、被拒后训诫
- **事中（隔）**：沙箱、容器、worktree、剥密钥再传子进程
- **事后（兜）**：谁把损失吃下去——git、OS、厂商、还是你自己

## 研究方法纪律（研究员必读）

1. **两名研究员各啃一库**（契约 7：Opus max effort，结构化 JSON 产出）。
2. **先枚举、后归位**：先不带矩阵全量扫一遍环境侧机制（拦/隔/兜任何与「动作落在真机器」相关的），再把每条机制归入矩阵格。**禁止只按矩阵要什么找什么**——矩阵是归位工具，不是搜索过滤器。
3. **塞不进/跨格的机制原样陈列**，标 `misfit`，不修剪事实迁就仪器。矩阵里的错位点是本章没造假的证明。
4. 每条机制必须标注：**默认还是 opt-in**（防「默认→绝对」）、`file:line`、有逐字制品的原文照录。
5. 产出 JSON 字段：`机制名(产品语言)` / `干什么` / `默认|opt-in|门控` / `矩阵行(可多)` / `矩阵列` / `file:line` / `逐字制品(如有)` / `misfit标记与理由`。

## A. 假设验证（写进章前必须过的证据关）

### A1. 「生存前提」假设 ⭐ 本轮最重要
**假设**：两道「连全放行都压不过」的硬地板，各自圈住的是**自己那套治理模式赖以成立的前提**——CC 治理靠「人在场+信任旋钮」，前提是规则层不被篡改，故地板护治理层（第 3 行）；Hermes 治理靠「机器活着+通道通着」，前提是机器不死，故地板护机器（第 4 行）。

- **A1a（CC）**：bypass-immune 清单**完整枚举**——除 .git / .claude / shell 启动 / IDE 配置外还有什么？有没有代码注释或文档解释「为什么这些连 Bypass 也不放行」？
- **A1b（CC）**：CC 对灾难命令（删根类）究竟哪些是**无条件硬拒**、哪些是能被放行档盖过的软问？旧稿称「个别如 PowerShell 删除才硬拒」——核实完整硬拒集合。
- **A1c（HM）**：硬禁地板完整清单重取；**防自尽规则（不许杀自己网关）为何留在软清单**——有没有注释/commit 解释？新版本有没有把它升格进硬地板？
- **证伪条件**：若 CC 地板混入大量与「治理层」无关的路径，或 HM 硬地板含大量与「机器失能/不可恢复毁灭」无关的命令，假设降级为「倾向」或弃用。防自尽留软层本身与假设有张力，必须正面回答而非绕过。
- **用途**：§深潜「两道地板」的升级版结论；若证伪则退回旧稿「各自把对方的软层升格成硬地板」的措辞。

### A2. 「闸门强度∝可逆性、不∝损失大小」
- 两边软/硬清单**逐条**按「可逆性」标注（可恢复/代价大但可恢复/无恢复路径），数反例：不可逆却在软清单的、可逆却硬拒的。
- **证伪条件**：反例占比高则该原则降为「大体趋势」。
- **用途**：矩阵的规律性论断之一（旧稿埋在 §03 verdict 末句，拟升格）。

## B. 空行/空列证伪（严防「没找到→绝对没有」）

### B1. 事后列真的空吗？⭐ 最可能推翻论点的一题
- **B1a（CC）**：有没有 harness 自建的恢复机制？**重点核 checkpoint / rewind / 文件快照类功能**（编辑前自动快照、会话回退）——若存在且默认开，「事后列整个外包给 git」的论断必须改写。
- **B1b（HM）**：有没有任何快照/回滚/undo 机制（包括 opt-in 的）？
- **用途**：论点「harness 做预防不做保险」的生死题。

### B2. 第 5 行（外部世界后果）是真空还是没查？
- **B2a（CC）**：危险模式清单里有没有 `git push --force`、发布类（npm publish 等）、对外发送类命令的专门拦截？系统提示里「不主动 push/commit」是**提示纪律还是闸门**——两者要区分陈述。
- **B2b（HM）**：约 47 条危险模式里有没有指向外部效果的（强推、发消息、邮件、发布）？网关本身能对外发消息——**发消息这个动作过不过审批闸**？
- **B2c（两家）**：MCP / 外部工具调用过不过环境侧的闸，还是只有 shell 命令被治理？（工具面覆盖不全本身是发现。）
- **用途**：若确认近乎空白→「两家的环境治理是机器中心的、对世界后果轻设防」成为新论点；若有料→第 5 行填实。

### B3. 第 2 行（机密外泄）的对称性
- **B3a（CC）**：有没有对子进程剥密钥 / env 清洗（对称于 HM 的剥密钥）？读密钥类文件（~/.ssh、.env、云凭证）有没有专门拦？
- **B3b（HM）**：除剥密钥 + Tirith 内容扫描外，「传出」侧还有什么闸？
- **B3c（两家）**：网络外发的闸各是什么形态（CC 沙箱内触网浮出问人已知；HM 对出网有没有对应物）？
- **用途**：第 2 行填格；「诚实」议题（HM 做了缓解却否认算边界）从这行长出，证据须扎实。
