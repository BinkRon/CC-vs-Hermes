# Harness 产品观

> 状态:初稿(ART-01),供讨论迭代。事实素材来自本项目八章成稿(校正后口径,见 `docs/archive.md`)与 CH-08 侦察;正文只讲产品逻辑,不出现代码细节。

---

## 引子:Harness 即产品

模型只是一个脑子。用户需要的从来不是脑子,而是一种服务——可能是「随叫随到的工程搭档」,可能是「替你盯着一摊事的私人助理」。**Harness,就是把脑子塞进服务框架的那整层设计。你设计什么样的 harness,就等于宣布你要提供什么样的服务。**

这句话有一个推论,贯穿全文:**harness 的每一个默认值和硬约束,都是服务定义的可执行形式**。官网文案可以随便写,但「危险动作默认拦不拦」「人不回话时默认干不干」「记住的东西存在谁手里」——这些设计骗不了人。读一个产品的 harness,比读它的宣传页更接近它的真实产品观。

---

## 一、Harness 分三层:通用层、身份层、定制层

用「人」来类比:

- **通用层**:一个人作为社会意义上的个体,有一些人类普遍具备的能力,以及人们对任何人都有的美好期许——办事靠谱、记得住事、学习能力强。对应到 agent:为了填平 LLM 与这些通用能力之间 gap 所做的 harness,就是通用层 harness。
- **身份层**:同一个人依附某个身份标签存在时——程序员、营养师、音乐人——身份背后是特定的能力组合。Agent 与身份绑定时,需要填平的那部分身份能力 gap,刨除通用层已覆盖的,剩下就是身份层 harness。
- **定制层**:这个人在身份之上,还服务于特定的人(群)——这些人有与自身偏好相关的要求需要 agent 遵守。这部分 harness 属于定制层。这个「人(群)」可以是**使用产品的用户**,也可以是**制造产品的厂商**。

三层不是修辞,可以用两个测试切出来:

1. **换主测试**:把 agent 从现在的主人手里拿走交给陌生人——还成立的部分是通用层+身份层(程序员还是程序员,靠谱还是靠谱),归零重来的部分是定制层。再剥掉身份,剩下的就是通用层。
2. **可关闭性测试**(切定制层内部):用户关得掉的定制是用户的;**用户关不掉的定制,就是厂商的**。厂商的定制永远坐在优先级塔顶——冲突时厂商赢。

三层的物理关系补一句:通用层不只是「一堆基础能力」,它还提供**容器**——「学得会规矩」这条元能力本身是通用层的,身份和偏好都是靠它装进去的。所以身份层有三种实现:**训进权重里、焊死在壳上、装在容器里**——一家产品把身份放在哪种实现里,本身就是重大产品选择。

(另注:有一类机制不属于任何一层——比如把服务过程的数据收回去训下一代模型。它不填任何服务 gap,是附着在服务上的商业模式,本文不把它算进 harness 的服务分层。)

## 二、通用层的普适期许:从「委托」一句话推出来

「聪明、情商高、靠谱、记得住事……」这样列下去是归纳,没有尽头,也永远不知道漏没漏。换演绎:通用层期许 = **「把事委托给任何一个受托者」这件事的结构性要求**。委托的全部结构,一句话装得下:

> **你把「想法」和「行动的权力」,交给一个「呼叫得到」的存在,让事情发生在「你不看着」的时候,而且「不止一次」。**

五个引号,五个结构件:两个**让渡物**(意图、行动权),一个**前提**(通道),两个**参数**(在场与否、单次还是重复)。全部期许从这五件东西机械地长出来。

### 核心:两个让渡物 × 三个流向

任何让渡物跨过「你↔他」这条边界,可治理的位置只有三个:**交到他手上时(承接)、在他手里时(履行)、回到你手上时(回环)**。没有第四个方向。于是:

| 你交出去的 | 交到他手上时 | 在他手里时 | 回到你手上时 |
|---|---|---|---|
| **意图** | 听得懂;理解经确认再动手 | 干到最后不忘初衷 | 歧义时会问;偏离时会报 |
| **行动权** | 干得动;钥匙拿稳不外泄 | 办得妥;失败自己爬起来 | 有分寸,越界先问;闯了祸有交代、有痕迹 |

### 三行注释:前提与两个参数

- **通道前提(叫得应)**:这一切之前,它得先是个呼叫得到的存在。「谁负责让它活着」本身就是产品分野——后文会看到,一家把命交给厂商云,一家自带常驻心跳。
- **在场参数**:上表每格都有两种形态。你在场时,「越界先问」是弹个窗;你不在场时,它必须变成「默认拒绝 + 事后可查」。**不在场形态是同一条期许的困难模式**——谁把困难模式当默认,是价值观的分水岭。
- **重复参数**:委托不止一次时,整张表在更长的时间尺度上再跑一遍——交到他手上=**记得住**(每次相处都被吸收);在他手里=**越用越好**(积累真的用回当下);回到你手上=**它学到的关于你的东西,你看得见、改得动**。

### 这张清单的边界

这个推导的生成器是「委托」本身,不是任何一家产品的功能表,因此没有产品偏见。但要诚实标注它的边界:**它给出的是合同层的下限**——一个合格受托者必须有的。它推不出「令人惊艳的」:主动补位、预判需求、超出交代的惊喜。那些属于品味,不属于合同;也恰恰是产品在「都及格」之后拉开差距的地方。

## 三、回望两家:同一张清单,两种价值观

Claude Code(下称 CC)和 Hermes Agent 都是通用 agent,都在填上面这张清单。有意思的不是谁功能多——大部分格子两家都填了,而且在纯物理约束的格子上(上下文满了怎么压、工具多了怎么装),两家几乎收敛到同一个解。**真正暴露价值观的,是三个问题。**

### 问题一(信任观):哪些期许你用壳保证,哪些交给模型?

看「办得妥」这一格:

- CC 的默认规则是:**模型这一轮不再调用工具,就等于活干完了**。壳不检查、不拦截,把「该不该收工」的判断权整个交给模型;需要更硬纪律的用户,可以自己往预留的插槽里装检查(不装就没有)。
- Hermes 在编码场景的默认规则是:模型说「我修好了」,壳先**扣下**这句话,去查有没有「测试跑绿」的新证据;没有,就悄悄塞回一条提醒,逼它真的去跑测试、看失败、修完再来——催促以两次为限。

同一格,一家几乎留空(信权重),一家焊死(信流程)。反直觉的是:**做模型的公司反而不信模型,只做壳的公司反而全信。** 谜底在处境:CC 押注一个自己完全掌控、持续变强的托管模型,壳做薄,天花板随模型自动抬高;Hermes 活在开源世界,要跑五花八门、良莠不齐的模型群,壳必须替所有模型兜底。**壳的厚度不是能力问题,是信任问题。**

同样的分野出现在「不忘初衷」格:CC 在上下文不得不压缩时,**逐字保留用户说过的每一句原话**——你的意图被当作不可再生资源;Hermes 的压缩策略里,最早的任务锚点会随会话变长逐渐被牺牲——原话被当作可折旧资源,但它把「关于你的偏好记忆」标记为压缩时不可覆盖的权威。两家保的东西不一样:一家保「这次你要什么」,一家保「你是什么样的人」。

### 问题二(在场观):每一格的默认形态,是「你在」还是「你不在」?

这是两家分得最整齐的一刀。

- CC 处处假设**有人在**:危险动作弹窗等你点、键盘上有现场调信任档位的快捷键、命令跑久了自动转后台但用通知把你叫回来、你离开再回来有一份「刚才发生了什么」的摘要。它的世界观是:人会回到会话。真正的无人长跑,外包给厂商的云。
- Hermes 处处假设**没人在**:自带常驻网关竖着三只耳朵(时钟、外部事件、聊天消息),跑完把结果主动推到你的聊天软件里,无事则保持安静;没人点确认时有明文规则——定时任务默认拒绝,并且写下了这个领域里最锋利的一句产品文案:**「沉默不等于同意」**。

于是「谁做得更好」有了诚实的答案:**胜负沿在场契约整齐裂开**。有人在场的那半张表(现场信任调节、协作团队、会话内的顺滑),CC 做得更好;没人在场的那半张表(保活、回音、无人时的确定行为),Hermes 做得更好。没有一边碾压——两家各自把自己押注的那半个世界做到了最好。

(诚实处补一刀:Hermes 为无人场景想得最深,却存在一个官方自认的漏洞——某类没被识别的无人会话里,危险命令会被直接放行。按它自己「必须为没人点确认定义确定行为」的宪法,这是缺陷,不是选择。深想不等于做全。)

### 问题三(权力观):定制层的塔顶归谁?

两家都把 harness 做成了高度可定制的平台,但「谁拥有最终否决权」截然不同:

- CC 是一座**物理上可强制执行的塔**:企业可以下发只读策略锁死员工的配置、远程关掉「全速放行」模式;有一小撮动作连用户开了全放行都压不过——耐人寻味的是,这道铁底护的不是「别删库」,而是**agent 自己的配置层**(版本库设置、启动脚本、agent 自身的规则文件):放手可以,但能改写游戏规则本身的东西永远不放手。这是 agent 时代才有的安全洞察——**防的是自我提权**。
- Hermes 的塔到极限处是**劝告性的**:它也有一道「连 yolo 都压不过」的硬禁清单(删根、格盘、关机——别毁了机器),但产品是开源的,自托管者改一行源码就能拆掉任何地板。所以它的真实塔顶不是厂商,而是**操作系统和你自己**。它对此极度诚实,白纸黑字写着:进程内没有任何东西算隔离,唯一的边界是操作系统。

两种权力观没有对错,是闭源托管与开源自托管两种所有权的必然结果——但它意味着「定制」这个词在两家指的不是同一件事:**CC 的定制是在厂商围墙内配置;Hermes 的定制是你自己就是围墙。**

### 两幅画像

把三个问题的答案叠起来,两家的 harness 价值观各自收拢成一句话:

- **CC:「天才在你身边工作。」** 给他自由(壳薄,信权重),守住底线(防自我提权的铁底,你的原话逐字保存),相信他随模型一代代变强——而你,就坐在他旁边。
- **Hermes:「管家独自在你家当值。」** 不许谎报(验证门),事事回音(主动推送),规矩白纸黑字(连自己的漏洞都写进文档),家是你的(你就是围墙)——而你,通常不在家。

## 四、这张清单怎么用

对不同的读者,这套「三层 + 期许清单」是三种工具:

- **评一个 agent 产品**:先用换主测试和可关闭性测试把三层切开;再拿期许清单逐格问「填没填、困难模式做了没有、填在壳里还是信了模型」。格子会替你把「功能对比」升维成「价值观对比」。
- **造一个通用 agent**:这十来条期许就是需求清单的下限。尤其别漏三个容易被吞的格子:钥匙拿稳不外泄(凭据治理)、闯了祸有痕迹(可追责)、学到的东西可见可纠(记忆透明)。
- **造一个垂直 agent**:先别写需求,先问基座——它的容器允许你改写到多深?营养师需要自己的记忆结构、自己的汇报节奏、自己的危险动作定义;如果基座只允许「加工具」,你要的不是插件,是另起一个 harness。

最后留一个开放问题,它是这张清单上两家都没填的洞,也可能是下一代 harness 的产品机会:**两家的「办得妥」都只在有机器可验证的领域(代码——测试会跑绿)敢用壳来保证;一到没有裁判的领域(订餐、沟通、判断——管家型任务),就都退回「信模型」。** 程序员的靠谱有编译器作证,管家的靠谱无人作证。谁先为「无法机器验证的靠谱」造出壳内机制——比如把「回音—确认—留痕」做成聊天任务的验证门等价物——谁就填上了通用层的最后一格。
