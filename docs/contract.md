# 契约层（CONTRACT）—— 所有调研与呈现的强制规则

> 这是"怎么干"的宪法。每一次调研 + 渲染都必须遵守下面每一条。
> "什么/为什么"见 `framework-and-plan.md`；实时进度见 `progress.md`。

---

## 契约 1 · 调研性质：产品逻辑，不是技术细节（最高优先级）

**受众是产品经理，不懂代码。** 每一篇都为这个受众而写。

- **要什么**：这个机制在"做产品"意义上如何运作、解决什么用户/产品问题、为什么这么设计、两家的取舍差异各是什么。
- **不要什么**：数据结构、函数签名、类名、语言特性、代码片段、实现技巧、框架名词。
- **代码只允许出现在页脚的 `file:line` 溯源里**，正文一律不出现代码。
- **自检标准**：一个不懂代码的 PM 读完，能复述出"它在做什么、为什么这么设计"。
  - 若某句话必须懂代码才能理解 → 删掉或改写成产品语言。
  - 若一张图是在画代码结构而非产品流程 → 重画。
- 宁可少讲一个技术点，也不要牺牲产品可读性。**深度体现在"想清楚了产品逻辑"，不体现在"堆了多少实现细节"。**
- **附注：真实文案与数字不是"代码"**。机制注入的提示词原文、真实错误消息、阈值/预算/重试次数等具体数字，属于**证据**，PM 读得懂也最该看到——契约 3 的深潜甚至强制要求展示它们。本条禁的是函数名/类名/数据结构这类"必须懂编程才能读"的东西，别把"具体"错杀成"技术"。

## 契约 2 · 输出格式

- **单章仍是自包含 HTML**（便于单独 review/重生成），复用 `findings/_design-system.html` 的整段 `<style>` 与 `<script>`，沿用其 stepper 约定（`id="stepper"` + 脚本里的 `const CAPTIONS`）——这样装配脚本能正确识别改造。
- **stepper 组件化约定（2026-07-02 起，升级已落地）**：新章一律用**组件化约定**：每个 `.stepper` 内嵌 `<script type="application/json" class="captions">[...]</script>`（条数=帧数）、按钮用 `.prev`/`.next` class，自带状态互不干扰，一章可放 1–3 个（契约 3 深潜配额）。`_design-system.html` 的 demo 与脚本已改为该约定（含双 stepper 演示），照抄即可；其脚本与 `build.py` 的 `SHARED_JS` 是同一段逻辑，**改一边必须同步另一边**。存量 7 章（01–07/SYN）保留旧约定（唯一 `id="stepper"` + 全局 `CAPTIONS`），`build.py` 继续自动转换，不回改。`build.py` 有构建期守卫：混用两种约定、残留旧标记、缺 `.prev`/`.next`、缺/坏 captions JSON 一律**构建报错**（防死按钮静默出货）。另注意：内联脚本的注释里不能出现字面量的 script 闭合标签（HTML 解析会掐断脚本）。
- **成品是合订本 `index.html`**：`build.py` 把 `findings/` 各章装配成一份带**左侧栏切章**的文档（组件化 stepper 互不干扰、mermaid 切到该章才渲染、设计系统只此一份）。**每当一章过了 review 门，跑 `python3 build.py` 重新生成 `index.html`；加新章改脚本顶部 `CHAPTERS` 清单即可。** 交付物是 `index.html`，勿手改。
- **设计系统两层制（2026-07-04 修订，DSN-01；细则见 `docs/design-principles.md` §3）**：
  - **集中层（单一真相不变）**：tokens、正文排版、轻/中档组件 CSS 只写在 `findings/_design-system.html` 的 `<style>` 里；`build.py` 构建时抽取它，只额外叠加"合订本专属"的侧栏/多章框架 CSS（`SHELL_CSS`）。**会被 ≥2 章复用的样式一律加进 `_design-system.html`**，别再往 `build.py` 里拷一份（防两份漂移）。单章因需自包含各带一份拷贝，用到新组件时同步补进该单章即可。
  - **bespoke 层（每章 ≤1 处）**：本章重档可视化允许章内 `<style class="bespoke">`，必须 scoped 在本章根 class 下（如 `.syn2`）、只准引用全局 tokens 变量；`build.py` 的 `extract_bespoke_css` 收进合订本 head（综合②先例）。bespoke 手法产生复用价值时提炼进集中层，禁止复制第二份。
- **合订本离线守卫**：`SHARED_JS` 里所有 mermaid 调用必须裹 `MERMAID_OK`（`typeof mermaid!=='undefined'`）守卫——否则 CDN 掉线时抛 ReferenceError 掐死整段脚本，导致章节停在 `display:none`、正文全空白。加守卫后：图退化为文字、正文照常可读。
- **单章导读不链接未完成章节**：单章页脚导读里指向尚不存在的章节一律写**灰字（非链接）**，避免单章 review 时点开 404。合订本由 `build.py` 剥掉单章 `<nav>`，不受影响。
- **两段式流水线**：研究员产出结构化 JSON（纯产品逻辑 + 图规格 + `file:line` 溯源）→ 渲染员套设计系统出单章 HTML → `build.py` 并入合订本。
- **流程图规范（2026-07-04 修订）**：一律 `flowchart TD`（纵向）；两图对比用 `.flow2` 两列；精简主干、短标签、边缘分支移到正文。**新增强制：语义配色**（判断=amber、终点=green、动作=中性）+ **两图对比必须高亮差异节点**（强调色+粗描边；自检：遮住标题 3 秒指出两图差异，指不出=白画）；图内字号 ≥15px。mermaid 非唯一：机制形状不是"流程"时允许手工排版主图（细则见 `docs/design-principles.md` §1.3）。
- **排版与视觉重量宪法（2026-07-04 新增，替代原"重可视化、少字"）**：形式跟随论点——每处内容先定 轻/中/重 档再选形式（重档每章 1 个、中档 ≤2 组、轻档禁用盒子）；叙事正文裸文、**全书统一 1080 版心且一章内所有块同一左右边缘（禁窄列破栏，2026-07-04 PM 裁定）**、粗体每段 ≤2 处、标题必须是论断。全量规则见 `docs/design-principles.md` §0–§2（**调研+渲染前必读**）。
- **出稿机械检查（强制，血泪教训）**：HTML 写出后必须两关——① 抽出内联脚本 `node --check` 过语法（中文正文里的英文双引号 `"` 会掐断整个 `<script>`，导致 mermaid 退回浅色默认主题、stepper 报 `step is not defined`）；② 浏览器 smoke：查 console 0 error，并**真的点一次 stepper 的"下一步"**确认 `step()` 可用、图为深色主题。只截静态首帧不算验收。字符串里要强调就用中文「」或全角引号，别用英文 `"`。

## 契约 3 · 每篇骨架（默认可偏离）+ 深潜规则（2026-07-04 再修订）

**骨架降为默认**（2026-07-04，DSN-01：原"骨架不变"是形式主义根因之一——模板槽位绑架了论证）：结论摘要（先给判断）→ 问题（具体故事）→ 核心机制 → **深潜**（1–3 个，规则见下）→ 对比 → 悬而未决 → 页脚 `file:line` 溯源，仍是无特殊理由时的默认；但**叙事顺序由该章视觉脚本决定**（`docs/design-principles.md` §4），按论证逻辑排、不按槽位排（综合②先例：场景→主图→根因→收纳）。各节标题必须是论断/问题，禁止组件名当标题。

**深潜三条规则**（修订动机：原"每章恰好一个 Hero + 固定 6 帧双盒 stepper"是格式预算而非内容判断——记忆/错误恢复/后台并发因此被饿成对比块；模具唯一又导致形式与机制形状错配。对标 learn-claude-code 后的结论：清晰感来自**具体性**，不来自动画）：

1. **名额跟随分歧密度**：每章**至少 1 个、至多 3 个**深潜。入选标准两条都要满足：① 该机制是本章两家分歧最戏剧化的点之一；② 手里有真实制品可展示（见第 3 条）。落选的分歧点仍进对比维度；若"够格的深潜多到放不下"，那是拆章信号，不是砍深潜的理由。
2. **形式匹配机制形状**：stepper 是默认形式之一，**不是唯一**。选形式前先问"这个机制的形状是什么"——空间型（窗口/上下文被挤压）→ 布局演变图；时间型（重试/恢复/生命周期）→ 时间线或分叉轨迹；协作型（多体/消息往来）→ 泳道/消息流。形状和形式错配时，宁可不用 stepper。
3. **必须含真实制品（硬要求）**：每个深潜至少展示一份真实材料——注入提示词的**原文**、真实错误消息、阈值/预算/重试次数的具体数字表，或两家同类文案**并排**（我们独有的王牌：如 Hermes verify 门的 nudge 原文 vs CC 压缩提示的"只准输出文本"原文）。**纯转述不算深潜**："壳合成一条系统消息"是转述，贴出那条消息的原文才是深潜。真实制品不违反契约 1（见契约 1 附注）。

## 契约 4 · 每个阶段性成果完成后，立即更新运行态（强制）

**每完成一篇 finding 或一篇综合，在提交前必须同步更新三处（缺一不算完成）：**
1. **`progress.md`**：把该项状态改为 ✅，更新"最后更新"日期与"当前焦点"；在章节清单里带上**需求 ID**（契约 8）；综合篇额外把该轴的"一句话设计思路"写进结论区。
2. **`backlog.csv`**：把该需求的 `状态` 列推进（`done` 则从活跃区移走——见契约 8）。
3. **`handoff.md`**：更新该需求 ID 块的"最新一棒"（新棒覆盖旧棒），写清完成摘要 + 下一步 + 阻塞项。

**跨 session 但本篇未做完**（换手/搁置）时：progress + backlog 状态照改，`handoff.md` 那一棒的"下一步"必须明确到让下一个 session 1–2 分钟内无歧义接手。未更新这三处，不算完成。

## 契约 5 · 贯穿的分析仪器

- **轴一（Loop 之内）每篇**：追问「**改模型 vs 改壳**」——这个行为是靠壳里的工程保证的，还是甩给模型（尤其 Hermes 自训模型可训进权重）？
- **轴二（Loop 之外）每篇**：追问「**什么世界/人/部署现实逼出了它**」（此处"改模型vs改壳"失效）。
- **每轴收尾必须综合成一个完整故事**，而不是把单篇拼在一起：轴一 →「一个 Loop 是怎么设计的」，轴二 →「Loop 之外是怎么设计的」。
- **综合篇只准编织、不准开采（2026-07-03）**：综合篇只允许引用**已过三道门章节**的材料与结论，不得引入未经核验的新机制断言——否则它就是一个没过门的隐形章节。脊椎所需材料若某章缺失，回该章补齐并过门，再进综合。配套前置：**06/08 的 verdict 必须显式回答"在场问题"**（如轴一各章 verdict 显式回答信任问题），综合②的"在场契约"脊椎只从各章 verdict 取料。

## 契约 6 · 大顺序（2026-07-03 修订：串行只约束"有依赖者"）

**逻辑依赖仍是硬约束**：先整条轴一（按轴内优先级）→ 综合① → 再整条轴二（按轴内优先级）→ 综合②。综合篇必须在其输入章全部 `done` 之后（综合②等 CH-06 + CH-08）；轴内有先后依赖的按序。

**但"没有依赖关系"的需求可并行**（配合契约 8 的 `依赖` 列）：互不相挡的需求（如章节 CH-06 与边角回改 FIX-02/FIX-03）允许不同 session 同时做。领取判据见契约 8.1（**`依赖`全 `done` 且 `状态=draft`**——不只看依赖，还要没被别人认领），与其它在做的需求无关。**不跳轴仍成立**（不因为能并行就把综合②提到输入章之前）。

## 契约 7 · 一篇一篇来 + 逐篇 review 门（强制）

**约束的是"单个 session 内不铺开、不攒篇"，不是禁止多 session 并行**（并行规则见契约 6 修订 + 契约 8）：**一个 session 一次只推进它绑定的那一个需求**，该需求必须过 review 门才交棒；单需求内部 ≤3 个 agent、串行单篇。不同 session 各自绑不同的独立需求同时做是允许的。下面的门对"每个被推进的需求"逐一适用：

1. **单篇流水线（2026-07-04 修订：渲染前加"视觉脚本"工序）**：每个代码库 1 个研究员（Opus max effort）出结构化 JSON（各啃一库、互不分心）→ **主 session 收敛论点后写"视觉脚本"**（五问：一句话论点 / 最大卡壳点 / 叙事顺序 / 重量分配表 / 重档草案——模板见 `docs/design-principles.md` §4；判断密集，不外包渲染员）→ 1 个渲染员按视觉脚本 + 设计系统出 HTML。单篇 ≤3 个 agent、且**串行单篇**。
2. **自审**：我通读产出，检查①产品逻辑是否清楚（契约 1）②设计系统是否合规（契约 2）③结论是否有 `file:line` 支撑。
3. **Codex 交叉核实（soundness 门）**：把该篇的事实性论断交给 Codex，对着**真实代码**核验"claim 是否属实"（Codex 读代码，我保产品可读性，互补）。
   - 调用：`codex exec -C /Users/luohuibin/cursor_project --sandbox read-only --skip-git-repo-check -o <out> "<prompt>" </dev/null`（不传 `--model`；`</dev/null` 必加；`-o` 文件为真相源，stdout 仅诊断）。
   - 对 Codex 的反馈**先形成自己的判断再采纳**（不照单全收）：属实的错误才改，误报驳回并记录理由。
   - **Codex 提示词必须额外问一句 completeness**：本维度代码里有没有"重要机制在这些论断里缺失或被低估"（不止查我写的对不对，还要查我漏没漏）。
4. **完整性/忠实度门（强制 · ch-07 血泪教训）**：Codex 只查"我写的是否属实"（soundness），**查不出"我漏掉/扭曲了机制"**（completeness）——综合环节为追求"一个干净故事"会把不合主线的机制压掉（subagent 模式就是这样被吞进"底座"的，靠用户才发现）。故每篇出稿必须再过一道**独立**核验：一个独立 agent 读"该维度真实代码 + 成稿"，把每个机制逐条标记「如实表达 / 忠实合并 / 静默丢弃 / **以偏概全**（子情形当全部）」，凡"会改变读者结论"的遗漏或扭曲必修。**根因是我对'清爽叙事'的系统性偏好——不能靠'下次更仔细'，必须靠这道独立门。**
5. **收敛论点前先自查**：从结构化 JSON 收敛成产品论点前，先把研究员列出的机制**逐条**过一遍"有没有被主线吞掉 / 有没有以偏概全"，再定稿。
6. **设计门（第四道门，2026-07-04 新增，DSN-01）**：渲染完成后、build 之前，按 `docs/design-principles.md` §5 的 checklist 逐条过（10 秒重点测试 / 三档配额 / 粗体与标题抽查 / 两图差异 3 秒测试 / 列宽 / 与重量分配表对照防机制静默丢失）。内容三道门保真实，这道门保表达——**过不了一样打回**。
7. **修正 → 更新 `progress.md` + `backlog.csv` + `handoff.md`（契约 4 + 契约 8）→ 才交棒。**
8. 整条轴一做完（含综合①）后**停下来交给用户看**，不擅自进轴二。

## 契约 8 · Backlog + 任务绑定 + Handoff（2026-07-03 新增）

**三份文件各司其职**：`backlog.csv`=需求清单与状态（该做什么、能不能领）；`progress.md`=运行态看板（一屏读懂全局，见契约 4）；`handoff.md`=跨 session 接棒台账（每需求只留最新一棒）。

### 8.1 领活必须绑定需求（推进交付物的 session，开工第一步）

**适用范围**：**推进交付物**（章节 / 回改 / 专题）的 session 必绑。**纯讨论 / 流程 / 元层维护 session 不绑不留棒**（如改契约、调研、sweep）——其产出直接落权威位置（contract / archive / commit），与 8.4 末条一致。

一个 session 只推进它绑定的**一个**需求（契约 7）。开工分两种入场，**判据不同**：

- **① 领新需求**（该需求 `状态=draft`）：判据 = **`依赖`列全部 `done` 且 `状态=draft`**。"依赖全 done"不够，必须同时是无人认领的 draft——否则会抢到别人正在做的在制品。优先级高者优先；同级选能立即动手的。想并行请**换 session**，别在一个 session 里同时领两个。
- **② 接在制品的棒**（该需求 `状态` 非 draft、非 done，即已有人做到一半）：**只读 `handoff.md` 该 ID 最新一棒的"下一步"续做**，不重新领。这是 handoff 机制的本职。

**领取＝占位（防并行抢单）**：领新需求的动作，**当场把 `backlog.csv` 的 `状态` 从 `draft` 切到下一档**（章节→`researching`，回改/专题→`rendering`）+ 在 `handoff.md` 该 ID 块起一棒（至少"本 session 已领取 · 下一步=…"）。此后它不再是 draft，别的 session 不得再当新需求领——**`状态` 列就是"谁占了"的信号**，我们的 progress.md 按轴/章节组织、不设"执行中"状态看板，故不在 progress 另标在制。

### 8.2 `backlog.csv` 字段协议

列：`ID,标题,类型,优先级,复杂度,状态,依赖,描述`

| 列 | 取值 | 说明 |
|---|---|---|
| `ID` | `{类型缩写}-{序号}` | 章节 `CH-06`/`CH-08`/`CH-SYN2`；回改 `FIX-01…`；专题 `EXT-01…`；其它按需增类 |
| `类型` | 章节 / 回改 / 专题 | — |
| `优先级` | P0–P3 | — |
| `复杂度` | S / M / L | 章节多为 L，回改多为 S |
| `状态` | 见 8.3 状态机 | 唯一状态真相源 |
| `依赖` | ID（多个用 `+` 连）/ `无` | **并行判据**：`依赖`全 `done` 才可领 |
| `描述` | 一句话 | **禁用英文逗号**（CSV 分隔符）——用顿号/加号/分号；一行一需求不换行 |

- **只装活跃 + draft**；需求切 `done` 后原则上**从 CSV 移走**（结论一句话记进 `docs/archive.md`），保持 backlog 清爽。存量已完成的 01–07/SYN1 不回填 CSV，其结论在 `archive.md`。
- **例外·依赖完整性**：一个 done 需求**只有在没有任何活跃需求的 `依赖` 列还引用它**时才可移走；只要它仍是别人的未消化依赖（如 CH-06/CH-08 被 CH-SYN2、CH-08 被 MEM-09 引用），就**保留其行、`状态` 置 `done`**，让"`依赖`全 done 才可领"这条判据始终能在 CSV 内自查、不必翻 archive。（`依赖=无` 者无人引用，done 即可移，如 FIX-01。）
- **正文不进 CSV**：CSV 只承载索引，机制正文/结论在 `findings/`（成稿）与 `archive.md`（结论）。

### 8.3 需求状态机（`状态` 列）

`draft`（已识别未开工）→ `researching`（双库研究员产 JSON）→ `rendering`（收敛论点 + 渲染单章）→ `gating`（过三道门：自审 / Codex soundness / 完整性）→ `done`（过门 + `python3 build.py` 并入合订本）。

- 回改/专题类无需研究员的，可 `draft → rendering → gating → done`，跳过 `researching`。
- 需 PM 决策时不切状态，在 `handoff.md` 那一棒的"阻塞项"写明，PM 解除后再推进。

### 8.4 `handoff.md` 规则（每需求只留最新一棒）

- **一需求一块**，块头是 `## {ID} {标题}`；**只保留最新一棒**，新棒覆盖旧棒（旧棒 git 考古）。棒计数按需求递增；棒数异常高（≥8）是该需求该拆分/止损的信号。
- **一棒三字段**：完成摘要（≤3 点，可选）／下一步（必填，明确到 1–2 分钟无歧义接手）／阻塞项（需人工动作的事；无则写「无」）。
- **写稳定事实，不写推理过程**；质量结论只写一句引用（如"三道门见 archive"），不贴测试日志。
- **session 内闭环、或无需求归属的调研/sweep** 不需要 handoff 棒——产出直接落权威位置（backlog 卡 / archive / commit）。
- **done 后**：该 ID 块块头标 `✅ done`、保留最后一棒作完成记录（不留悬空 draft 块）；整体归档时随 progress 一并沉 archive，不让死块长期堆积。

### 8.5 三处一致性不变量（分文件方案的安全带）

我们把"状态看板 / 叙事 / 接棒"拆给三个文件，代价是可能漂移。故立此不变量：同一需求的状态在三处必须一致，**任一变更在同一 session 内同步三处**——
- `backlog.csv` 的 `状态` 列（唯一状态真相源）
- `handoff.md` 该 ID 块的最新一棒
- `progress.md` 章节清单里该项的标记（可领 / 🟡 进行中 / ✅ 完成）

**可 grep 自查**：`grep '{ID}'` 三个文件，状态口径不得打架。**done 不在 progress/handoff 逗留**——验收后随该需求一起沉 archive，不留"已 done 待清理"的中间态。

### 8.6 并行 session 的共享产物串行化

章节 HTML 各写各的（不同 `findings/` 文件）互不冲突；冲突只在**合订本装配这一步**——`build.py` 的 `CHAPTERS` 清单、`python3 build.py`、`index.html`。故并行 session 共享同一工作树时，**改 CHAPTERS / 跑 build / 提交 index.html 这一段必须串行**（一个先并入合订本，另一个再并），或各自 git worktree 隔离最后合并。
