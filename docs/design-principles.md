> **为什么有这份文件**（2026-07-04，DSN-01）：PM 判定 v1 产出形式主义——一套固定组件模板套所有章节，
> 导致全章等重盒子海、正文进框、粗体轰炸、流程图灰阶淹没差异点、组件名（"Hero 机制"）当章节标题。
> 根因不是缺组件，是**缺"什么时候用什么、什么时候不用"的使用规则**，以及流水线里没有"设计"这道工序。

> 与契约的关系：契约 2/3/7 的排版相关条款以本文件为准绳；冲突时改契约对齐本文件。

## 0. 总纲：形式跟随论点

**设计预算是稀缺品。** 重档**没有固定数量上限**（2026-07-06 PM 裁定，覆盖旧「至多 2 个」硬卡）——但每一个都要**独立挣来**：过『不画就看不懂』这道测试（承载一个真实的读者卡壳点、且含真实制品或语义配色差异高亮）。拿不准就降档。反面是 v1 的病——全在中档、平均用力。**重档多 ≠ 设计好；数量由"这一章有几个真卡壳点"决定，不由额度决定。** 一章里出现多个重档时，视觉脚本必须逐个点名各自的卡壳点、证明不是凑数。
判断任何一处该投入多少，只问一个问题：**这个内容不精心画，读者会不会误解或卡壳？**

- 会 → 它是本章的"重头"，配得上定制可视化（bespoke）。
- 不会 → 从简：轻档甚至纯文字。**简单的东西简单画，是尊重读者，不是偷懒。**

反面判据（v1 血泪）：如果一章从头到尾每个块视觉重量相同，等于没有重点——**什么都强调 = 什么都没强调**。

## 1. 全局层（强制 · 所有章一致）

### 1.1 Tokens（不变，沿用 v1）

色彩语义：`--blue`=Claude Code、`--purple`=Hermes、`--green`=结论/正向、`--amber`=判断/警示、`--red`=危险。
背景三阶：`--bg` / `--surface` / `--surface-2`；文字三阶：`--text` / `--muted` / `--faint`。
**bespoke 样式只准引用这些变量派生，不得引入新的品牌色**（防 8 章漂移）。

### 1.2 正文排版规范（新增 · 治"挤成一坨"；列宽规则 2026-07-04 PM 裁定后修订）

- **列宽：全书统一一条版心（`--maxw` 1080），一章内所有块共用同一左右边缘**——正文、图、矩阵、制品
  全部对齐同一条右缘，**禁止窄列/破栏混排**（血泪：v2 初版把 33/38em 窄列与全宽组件混在一页且左贴不居中，
  宽屏下右侧大片空白、各块右缘参差，比 v1 还乱；综合②同病已一并修正）。
  长行可读性**靠短段（渲染后 ≤4 行）、1.8 行距、裸文无框、段间留白**保证，不靠缩短列宽。
- **裸文优先**：叙事散文一律直接呼吸在页面背景上，**不进盒子**。轻强调用左侧 2px 色规（综合② `.scene` 先例），
  不用四边描边框。盒子只留给三类东西：图、真实制品（提示词原文/错误消息/数字表）、判词 callout。
- **一段一事**：一段只推进一个论点；超过 4 行（渲染后）必拆。说明文字和引文/制品分离，不粘连。
- **粗体配额：每段 ≤2 处**，只标题眼（读者扫读时必须撞见的那个词），不标动词串。
  两家名字着色（`.cc`/`.hm`）不算粗体配额。
- **标题必须是论断或问题**，不是栏目名/组件名。「Hero 机制 · 分步演示」这类立即打回；
  「Hermes 为什么不信模型的'我做完了'」这类才合格。
- 分节标题采用编辑式：小字 kicker（栏目定位）+ 大标题（论断本身）——综合② `.sec` 先例。

### 1.3 Hero 规范（修订 · 治"装饰性 chips"）

Hero 可以是固定配方（learn-cc 证明零变体反而降低读者导航成本），但**每个字段必须是硬信息**：

- 章号徽标 + 轴分组色标签 + **论断式 H1** + 一句副标（金句/祈使对偶句，如「挂在旁边，别写进去」）。
- **一行硬指标**（可选但推荐）：本章最硬的 2–3 个数字/事实（对照站用 `102 LOC · 1 个工具`；我们可用如「CC 5 种权限模式 · Hermes 默认拒绝」）。
- 核心判断的载体（**PM 2026-07-06 裁定，CH-A1R-01 验收反馈，覆盖此前"大字 lede"方案**）：回到**thesis 框**（绿左规判词框——判词 callout 本就是 §1.2 允许进框的三类之一），框内必须分段（一段一事，≤2 段）；裸大字 lede 与正文连排会"挤成一坨、字号层次乱"，不再作默认。
- **开篇"问题/故事"段允许 story 虚线框**（同一裁定）：thesis 框＋story 框构成开篇两个视觉锚点；story 框内必须用 `<p>` 分段（渲染后每段 ≤4 行），禁止 `<br><br>` 连排长文。此为**开篇专属**例外——正文其余叙事仍裸文优先。
- **砍掉**：装饰性关键词 chips 排（v1 的「while(true) 引擎/终止信号/…」——视觉噪音 > 信息量）。

### 1.4 流程图规范（修订 v1 · 治"灰阶淹没差异点"）

- 仍然 `flowchart TD` 纵向、短标签、主干精简。
- **语义配色（新增强制）**：判断菱形=amber 描边、终点=green 描边、普通动作=中性。
- **两图对比必须打亮差异**：两图并排存在的唯一理由是让读者 0.5 秒看到差异点，
  差异节点必须用强调色+粗描边高亮（mermaid `classDef` 即可），其余节点保持安静。
  自检：遮住标题，能否 3 秒内指出两图哪里不同？不能 = 白画。
- 图内字号 ≥15px（v1 的 13.5px 偏小）。
- mermaid 不是唯一选项：当机制形状是"谱系/脊椎/链条/对照"而非"流程"时，
  手工 HTML 排版往往更准（综合②"在场之脊"、"所有权链"先例）——见 §3 重档。

### 1.5 详情入口与弹窗规范（新增 · 2026-07-06，CH-A1R-02 教训）

一段内容想给「想追溯细节的读者」留一个深入口（点开看完整对照/原文），先过一道**准入**，再定**形态·摆位·权重**。

**准入：论点 vs 收据。** 只有「收据」才准放进入口/弹窗——
- **论点**＝读者不看就会漏掉或误解的东西 → 必须在主线正文里，不许藏在点击后面（藏起来＝对略读读者不存在＝以偏概全，撞忠实度门）。
- **收据**＝读者不看结论不变、只是「信不信、查不查得到」变了的佐证（提示词逐字原文、完整清单、超长对照）→ 放进入口/弹窗正是它该在的地方，避免用参考材料把主线撑肿。
- 自检：把入口后面的内容永久删掉，读者带走的那句话会不会变？会变→是论点，拉回主线；不变→是收据，可入口。

**形态：跟随载荷（形式跟随论点的推论）。**
- **集合型入口**——揭示「更多同类项」（示例只露一部分、点开看全部）：入口＝集合里的**第 N+1 格、排末尾、待在容器内**，继承集合的视觉语汇（是 chip 就当 chip）。见 `_design-system.html` 的 `.more`（CH-A1R-02 `.cl .more` 先例）。
- **证据型入口**——揭示「某一步/某论断的完整证据」：入口＝**紧贴它所解释的那个块的一条档三文字链**（`↳ 描述 →`），不是浮在缝里的按钮（CH-A1R-02 `.receipts` 先例）。

**摆位：锚定原则。** 一条深入口只有两种合法姿势——
- **贴着主体**（相邻即锚定，不需额外标识）：优先。单条注解直接放在它解释的那一步正下方/旁边。
- **远离主体但打标**（脚注式：主体上打 `*`/`ⓘ` 标识、别处按标给内容）：只在「必须放远」或「同一处有好几条」时才值得起这套标识。
- **反面**（CH-A1R-02 血泪）：既不贴主体、又不打标 → 入口飘在版面缝里成孤儿，读者不知道它属于谁。

**权重：档三·弱。** 深入口按定义次要于它伴随的内容——文字链或幽灵 chip，**不用带框按钮**（带框＝档二，与真内容抢眼）。读者的眼应先落正文、后落入口。

**弹窗（modal）载荷**：给「长到内联会撑爆主线」的收据用。要求——① 触发器标签＝弹窗标题（知道点开了什么）；② 统一 anatomy（标题+关闭×、点遮罩/Esc 关、深色主题、内部滚动）；③ 弹窗内含 mermaid/图时同样受合订本离线守卫（`MERMAID_OK`）约束。
- **反诱饵**：入口文案不许把「本节正努力弱化的差别」当钩子吊出来（CH-A1R-02：原「…9 vs 13 输出结构」诱导读者重视一个本节判为『非分野』的差别，改成中性的「看两家 prompt 逐字原文」）。文案该强化本节论点，不与它反着劲。

## 2. 三档视觉重量（每处内容先定档，再选形式）

| 档 | 适用 | 形式 | 配额 |
|---|---|---|---|
| **轻** | 一句话能说清的差异/事实 | 行内对照（`CC：… / Hermes：…` 一行带色点）、裸文一句、左色规引文 | 不限，**禁用盒子** |
| **中** | 结构性对比、多维对照 | `.compare` 双列卡、`.matrix` 矩阵、灰阶极简小图 | **每章 ≤2 组**（v1 曾一章 4 组 compare+verdict = 平行墙）|
| **重** | "不画就看不懂"的核心机制（读者卡壳点） | 定制可视化（语义配色主图 / stepper / 泳道 / 时间线 / 手工排版），允许 bespoke scoped 样式 | **由『不画就看不懂』测试定，无固定数量卡**（2026-07-06 PM 裁定）；拿不准降档；多个重档时视觉脚本逐个点名卡壳点、证明非凑数 |

**降档优先**：拿不准放哪档，放低的那档。v1 的病是全在中档；宁可显得朴素，不要显得平均。

**重档的特权与义务**：特权——通栏宽度、前后大留白（让它成为"这一屏的主角"）、bespoke 样式；
义务——必须承载本章视觉脚本里认定的"读者最大卡壳点"，且含真实制品或语义配色的差异高亮。

## 3. 两层制（样式归属规则 · 契约 2 修订对应）

- **集中层（强制）**：tokens、正文排版、轻/中档组件 CSS 只写在 `findings/_design-system.html`（单一真相不变）。
- **bespoke 层（每章 ≤1 处）**：重档可视化允许章内 `<style class="bespoke">`，
  **必须 scoped 在本章根 class 下**（如 `.syn2`，防泄漏别章），**只准引用全局 tokens 变量**。
  `build.py` 的 `extract_bespoke_css` 会把它收进合订本 head（机制已落地，综合②先例）。
- 判断"进集中层还是 bespoke"：会被 ≥2 章复用 → 集中层；为一个论点定制、无复用价值 → bespoke。
  bespoke 里长出复用价值的手法，由后续章节提炼进集中层，不许复制粘贴第二份。

## 4. 每章工序：视觉脚本（渲染之前的"设计"这道工）

**谁做**：主 session（判断密集，与"综合"同级；不外包给渲染员）。
**何时**：综合论点收敛之后、派渲染员之前。
**产物**：一份视觉脚本（写在渲染员的任务提示里即可，不必单独成文件），回答五问：

1. **一句话论点**：本章读者只能带走一句话，是哪句？
2. **最大卡壳点**：哪个概念不精心画，PM 读者会误解或读不下去？——这决定唯一的重档给谁。
3. **叙事顺序**：问题→机制→差异→判词是默认，但**允许每章不同**（综合②就是场景→主图→根因→收纳）。
   按这一章的论证逻辑排，不按模板槽位排。
4. **重量分配表**：把本章每个论点/机制标上 轻/中/重 档（对照 §2 配额自查）。
5. **重档草案**：画什么、高亮什么、读者应该 0.5 秒看到什么、用哪件真实制品。
   有思路后可查 §6 手法库找参照（借鉴手法，不临摹样式）。

**警示（与契约 7.4 同源）**：视觉减负最容易顺手丢机制——"不好画"不是降档更不是删除的理由，
是换形式的理由。重量分配表必须覆盖研究 JSON 里的全部机制，谁都不许静默消失。

## 5. 设计门（第四道门 · 与三道内容门并列）

渲染完成后、build 之前，逐条过（PM 视角，不是工程视角）：

- [ ] **10 秒重点测试**：快速滚一遍全章，能说出"本章最重要的一张图/一个论断"吗？说不出 = 没有视觉重点，打回。
- [ ] **重档自查**：每个重档都在视觉脚本里过了『不画就看不懂』测试、逐个点名了卡壳点（非数量卡、是逐个挣来）；中档 ≤2 组；轻档无盒子。
- [ ] **粗体抽查**：任抽 3 段，每段粗体 ≤2 处。
- [ ] **标题抽查**：每个 h2/h3 是论断或问题，无组件名/栏目名。
- [ ] **差异高亮测试**：两图对比遮住标题，3 秒指出差异点。
- [ ] **列宽**：叙事段落没有撑满版心。
- [ ] **完整性挂钩**：对照视觉脚本的重量分配表，确认没有机制在改版/排版中被静默丢弃（与忠实度门衔接）。
- [ ] **词条可解码（PM 2026-07-06 新增，CH-A1R-01 教训）**：重档图里高度概括的短语级词条（如「tick 保活」「别交桩」），逐个自问"不看正文能懂吗"——不能懂的必须配 hover 一句话解释（`data-tip`，01 章先例）或图下附注；图注注明"悬停可见解释"。压缩出来的黑话没有解码通道＝白画。
- [ ] **详情入口合规（2026-07-06 新增，CH-A1R-02）**：页面里每个「点开看详情」的入口过 §1.5 四问——① 后面装的是收据不是论点（删掉它读者结论不变）？② 贴着它解释的主体、或已用标识锚定，没浮在缝里？③ 档三弱权重、非带框按钮？④ 入口文案没拿「本节要弱化的差别」当诱饵？
- [ ] 机械两关照旧（契约 2：node --check + 浏览器 smoke 真点交互）。

## 6. 手法库（收割自 learn-claude-code，2026-07-04；借鉴手法、不临摹样式）

> 逐页明细见收割存档 `assets/harvest-learn-cc/`（s01–s20 + 版本对比/架构层，22 页逐页手法分析）；
> 本节是提炼后的可复用原则，已按"我们何时用"改写。写视觉脚本时拿不准图式，先查这里再查存档。

### 6.1 用色纪律（最值得学的一条）

- **颜色 = 内容坐标**：全局固定色板贯穿始终（我们已有：蓝=CC、紫=Hermes、绿=结论、amber=判断）。
- **页内颜色只编码"读者此刻该分辨的那一个维度"，其余一律灰**。对照站 s01 整张流程图压成灰、
  只留一条蓝标"此刻活跃"；s03 用绿/黄/红直接承载 allow/ask/deny 档位——色即义、无需图例。
  → 我们的两图对比：全图安静灰，唯一强调色给差异节点。
- 概念自带"档位/等级/状态"时，让颜色去承载它（权限档位、待办状态、来源归属），不要另配图例文字。

### 6.2 重型可视化的选型判据（图式与难点类型一一对应）

先判断"这一章最难的是哪一类"，再挑对应图式：

- **过程/状态随时间变化** → 可 step 的分步件（图 + 状态面板联动，步进时只点亮当前节点）。
- **时序/并发**（谁先谁后、谁等谁）→ 带时间轴的多泳道图。
- **关系/依赖**（谁挂着谁）→ 拓扑/DAG 图。
- **一对多映射**（分发/扇出）→ 中心节点 + 放射分支 + 每支一句白话。
- **档位路由**（什么落到哪档）→ 泳道流水线 + 颜色编码档位 + 真实样本卡。
- **容量/占用**(不可见资源) → 拟物量表（量杯/进度条）+ 分色图例。
- **隔离/派生**（主线 vs 支线）→ 双泳道，只在末端一根回传箭头；**空间隔离** → 分叉文件树。
- **分层存留**（哪层丢、哪层留）→ 上下分层图，一层蒸发一层沉底。
- **动态装配**（多源拼成品）→ 分色积木拼装图。
- **范式转变**（两种世界观）→ 双模式并置心智图。
- **谱系/链条/对照** → 手工 HTML 排版（综合②"在场之脊"先例），不硬套 mermaid。
- **难点其实在工程实现** → 放弃大图，重心给真实制品走读（对照站 s19 先例：不硬画）。
- 难点不是过程/结构，一句话能说清 → **别上重型**（轻档裸文）。

交互原语**收敛到一种**（我们的组件化 stepper），到处复用、不造新控件——对照站全站只有一个编号 stepper 原语。

**可点性靠语境，不靠边框**（CH-A1R-02 反讽）：`.more` chip 用虚线边却明确读作可点，靠的是「排在 chip 集合末尾＋箭头＋『查看全部 N 种』计数」三个语境信号，不是边框粗细。曾有一版规矩把「虚线不够可点」矫正成「一律实心按钮」，反把入口做成抢眼的孤儿按钮——**修可点性该加语境信号（位置/箭头/计数/hover），不是加边框重量**。详情入口的完整规则见 §1.5。

### 6.2b 对比表达（版本对比页专记 · 与我们全书同构，最直接可用）

- **对比的原子单元 = 三列表 `维度 | Claude Code | Hermes`**（我们的 `.matrix` 正是这个骨架，保留为对比默认形式）。
  先立维度、再同行并置两方取值——差异靠"横扫同一行"获得，冷静、维度多时耐读。
- **差异高亮要克制**：同行并置本身就是差异表达，不满屏红绿；仅当某行是"一方有/一方无"的**质变**时，
  才用一个语义色徽章（如绿「有」/灰「无」）点一下。
- v1 的教训对号入座：`.compare` 双列卡是**弱形式**（两坨散文并排，读者得自己找对应点），
  只用于"两边哲学根本不同、没法逐维对齐"的场合；**能逐维对齐的一律用 `.matrix` 三列表**。
- **密度延迟释放**：维度很多时先分组/聚焦，读者选中某组再展开，不一屏压人。
- **两种尺度各司其职**：三列表讲逐维差异（微观），定位矩阵/色带讲全局格局（宏观），一章可以两者都有、但别混用。

### 6.3 成对概念共用隐喻

相邻/对立的概念用同一套图形语言（对照站 s08 压缩 ↔ s09 记忆共用"层/容量"隐喻），
让"差异"落在同一坐标系里可比。→ 我们全书 CC/Hermes 天然成对：**同一章的两侧必须用同一套图形词汇**，
差异靠高亮不靠换画法（两边画法一变，读者就没法对齐比较了）。

### 6.4 文案与证据

- 副标/金句用**祈使对偶句**把产品主张一句立住（「挂在旁边，别写进去」「Load Only When Needed」）；
  机制越简单，副标越该短到像口号。
- **每个论点配一个可验证的真实制品**（真命令/真数字/真文案原文），抽象主张必须被具体样本钉死
  （与契约 3 深潜"必须含真实制品"同源，扩展到全文案层面）。
- 卡片只给"需要并列对比的同类项"，卡内只放标签级短语（一两行），解释放裸文短段（130–180 字/段）。
- "反例开场"（没有 X 会怎么翻车）比正面说教更快让读者认同必要性。
- 复杂度统一下沉：实现细节压进页脚/折叠区，正文与图只讲"为什么/是什么"
  （与契约 1"代码只在页脚 file:line"同构，对照站独立验证了这个分层）。

### 6.5 骨架的一致与弹性（诚实标注一处张力）

对照站**锁死**七段式模板（问题→解决→原理→变更→试一下→接下来→源码），靠重复降低导航成本——
这与我们刚把骨架从"不变"降为"默认可偏离"（契约 3）方向相反。**不照单全收，两边各取一半**：

- **段落功能层面保持一致**（读者学一次就知道"判断在头、深潜在中、诚实边界在尾"）——骨架仍是强默认，
  10 章里预计 8 章照默认走；
- **偏离必须有论证理由**且写进视觉脚本（综合②那种"场景→主图→根因→收纳"是为'编织'类章节定制的，
  不是风格自由）。一致性服务导航，弹性服务论证——两个都要，冲突时论证优先。

### 6.6 综合/总章的收束方式

总章**不重画**各分章的图——用"定位表 + 回指"收束（对照站 s20 用「机制 × loop 位置」定位表收 20 个机制）。
我们的综合①/②同理：一张「维度 × 轴/位置」定位矩阵把散点收成一张网，每格一句话回指原章。
（综合②的"在场之脊"已经是这个思路的变体：脊 = 定位轴，五行 = 回指五章。）
