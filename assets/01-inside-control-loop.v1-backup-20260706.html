核心 Agent Loop

同样是「调模型 → 跑工具 → 回喂 → 再问」的引擎，两家在「什么时候停、信不信模型」上分道扬镳。

while(true) 引擎 终止信号 失败回喂 改模型 vs 改壳 验证才能停
两家用的是同一台引擎，分歧只在一个词——信任。Claude Code 默认把「该不该继续」交给模型（模型不点工具 = 收工，壳默认不判断任务是否完成）；Hermes 则不信任自训练模型的「我做完了」，把可靠性焊进壳里（改了代码没测试就不许停）。一个默认赌模型，一个补模型。（CC 也有两个 opt-in 的壳侧覆盖——停止钩子、token 预算续跑，见 §05；只是默认不用。）

01问题：agent 为什么需要一个「循环」

用产品语言讲：一次模型调用干不完的活，靠什么把它串成一件完整的事。

用户说「帮我修好这个登录报错」。模型不可能一句话搞定：它得先读代码、再搜日志、改一处文件、跑一次测试、看到测试还红再改第二处……每一步都要先看到上一步的真实结果，才能决定下一步。

单次问答做不到这件事——模型说完就结束了，没有「拿着工具结果接着想」的能力。于是需要一个循环：把「模型想用什么工具」变成真实动作，把动作结果塞回对话，再问一次模型，如此往复。这个循环要回答三个产品问题：什么时候算干完了？（终止）、中途出错怎么办？（恢复）、怎么防止它失控烧钱或干坏事？（约束）。两家的全部差异，都藏在这三个问题的答案里。

02一轮 loop 的数据流

左蓝 Claude Code，右紫 Hermes。主干只保留骨架——两条流走到「模型没点工具」之前几乎一模一样； 差别全在出口：Claude Code 直接收工，Hermes 头上多了一道验证门，会把「自认为做完」的模型推回循环。

对照：Claude Code 的 Hero —— 「无工具即停止」

Claude Code 默认不带「验证门」这类逻辑。循环里没有内置代码判断任务是否完成，它只看一个二元信号：模型这轮回复里有没有「我要用某个工具」。有→跑下一轮，无→收工。「该不该继续」的判断权默认交给模型——所以它的能力天花板主要由模型决定，而非 harness 的 if-else。（注：CC 并非结构上没有「壳覆盖模型停止」的能力——用户配置的停止钩子能拦收工把模型推回循环、opt-in 的 token 预算也会续跑，只是都非默认；详见 §05。）这与 Hermes 的差别，是默认姿态：一个默认把决定权交出去，一个默认收回来。

04对比：四个维度看清分歧

终止方式、失败恢复、控制侧约束、过程纪律——同一台引擎，答案却指向两种哲学。

① 终止方式：谁说了算「做完了」

Claude Code

默认下：模型这轮没点任何工具 = 结束。壳默认不判断任务完成度。交互式 REPL 默认没有轮数上限，防失控靠上下文压缩与用户随时中断；轮数上限仅 SDK/子 agent 才设。（注意别把 token 预算当"防烧钱天花板"——它是 opt-in 的"催多干地板"：用户写"+500k"后，模型没点工具、只要没到目标，壳会注入"keep working"续跑。见 §05。）

Hermes

「无工具调用」只是候选答案。还要过停止门：空/仅思考 → 重来；改了代码却无新通过证据 → 注入「先去验证」并 continue。模型无权单方面宣布完成。加上 90 轮预算 + 一次 grace call 兜底。

为何不同（改模型 vs 改壳）：Claude Code 把终止权交给模型，信任它自己知道何时收手；Hermes 不信任自训练模型的「完成」声明，把「停止纪律」焊进壳。前者赌模型足够聪明，后者假设模型会偷懒谎报。

② 失败恢复：出错之后怎么办

Claude Code

分层回喂，原则是「能回喂的都回喂，让模型自纠」：单工具错→标记成错误塞回历史；模型过载→切备用模型重跑；输出截断→偷偷抬 token 上限/多轮续写；上下文 413→压缩后重跑。真抛异常前，先补齐所有悬空的工具结果。

Hermes

两层，同样以「回喂优先，不轻易 abort」为主：工具异常→变「Error executing tool…」字符串回喂；坏工具名→对目录自动修复 + 3 次自纠；API 失败→内层重试+backoff+自动压缩+故障转移；空/仅思考输出→专门恢复路径。

为何不同：哲学一致（错误格式化成模型能读的结果、让它像人一样「看到报错再改」），差别在信任的下限。Claude Code 假设模型能产出合法工具调用；Hermes 不假设——连工具名拼错、JSON 参数损坏、干脆空输出，都得靠壳先修好，模型才配拿到结果。

③ 控制侧约束：防失控的护栏

Claude Code

偏「资源与安全护栏」：权限门 allow/ask/deny（deny 变错误回喂而非执行）、并发上限（读 10 并行 / 写串行）、可选轮数上限、输入校验 + 清洗（防伪造已授权字段）、停止/PreToolUse 钩子、全程中断监听、结果体积预算。默认不猜任务完成度——但停止钩子（用户配置）与 token 预算续跑（opt-in）能让壳覆盖模型的停止决定。

Hermes

多了一层「行为纪律」：迭代预算（父 90 / 子 50，cheap execute_code 退款）、原始调用上限 + grace call、每调用超时（推理模型 600s）、权限/护栏门、verify-before-done 门、无效工具重试上限 3、API 重试上限、429 速率守卫（针对自托管栈）。

为何不同（改模型 vs 改壳）：Claude Code 的约束几乎全是「别烧钱、别干坏事」的护栏，把「做得好不好」留给模型；Hermes 在护栏之外还加了一层「逼你做对」的纪律（验证门、think 块剥离、工具名修复）——因为自训练开放模型的行为不如托管前沿模型可靠，壳只能补得更厚。

④ 过程纪律：计划清单谁来盯

Claude Code

给模型一个「任务清单」工具（活儿超过 3 步就该用、开工前把当前项标「进行中」、干完标「完成」），还有个先出计划、批准了再动手的「计划模式」。关键是壳会主动催：清单搁久没动，壳周期性往对话里塞一句轻提醒——「任务清单工具最近没被用。若手头的活值得追踪进度，考虑用它……这只是温柔提醒，不适用请忽略。」（特性开启时）你把一张 3+ 项清单全标完、却没有一项是「验证」，壳还会补一句提醒你补做验证。

Hermes

也有一个功能相当的「待办」工具（拆解任务、追踪进度、长对话里保持专注），但刻意不挂钩子——它的设计写死「不改系统提示、不改工具返回，行为引导全部只放在工具说明里」。壳在待办这件事上不主动插话；它多做的一层是让待办清单熬过上下文压缩（压缩后自动把清单再灌回去，计划不被冲淡）。Hermes 把「每隔约 10 轮主动催一次」的额度，留给了记忆蒸馏与技能自造（见轴二·Loop 与自己），不花在催你维护待办上。

为何不同（改壳的两种脾气）：两家给了同一件规划工具、也都不强制你做计划——差别在壳把「催」的力气花在哪。Claude Code 让壳主动盯着待办卫生（该更新了、别忘了验证），但都是「软提醒、你可忽略」；Hermes 让待办工具保持安静（引导只写进说明）、转而保证计划扛过压缩，把主动催促省给「记忆/技能」的自我改进。这恰好呼应本章主线：CC 的过程纪律是可忽略的软提醒，而 Hermes 真正的硬纪律在别处（验证门那种结构性覆盖）——唯独在「待办」这块，它反而是最不插手的一方。

05悬而未决 · 意料之外

  • [CC] 默认循环里没有内置代码判断「任务是否完成」——默认终止等价于「模型这轮没点工具」，能力天花板因此主要由模型决定。但 CC 并非结构上没有"壳覆盖模型停止"的能力，只是把它做成两个非默认口子：① 用户配置的停止钩子返回 blockingError 时，壳把理由塞回历史、continue 而非 break（结构上与 Hermes verify 门同族）；② 用户在 prompt 里声明 token 预算（如"+500k"）后，模型没点工具、只要没到目标 90%，壳会注入"Stopped at X% — keep working"并续跑——这是"催多干的地板"，不是"防烧钱的天花板"。默认 REPL 下两者都不触发，故正文的"默认交给模型"成立；但别据此以为 CC 结构上做不到二次判断。
  • [CC] 工具能在模型还没说完话时就开始执行（边流式输出边执行），是用户无感的延迟优化，非模型行为。
  • [CC] API 错误被刻意排除在停止钩子之外——团队踩过「错误→钩子注入更多内容→重试→又错误」烧掉上千次调用的死亡螺旋。
  • [CC] 子 agent 不是另写一套循环，而是用完全相同的 query() 引擎递归调用；上下文压缩对模型完全透明，它无从知晓自己「失忆」过。
  • [HM] 自训练模型被信任得更少而非更多：harness 裹了异常厚的可靠性壳（工具名修复、JSON 清洗、空/仅思考恢复、消息交替修复），暗示权重本身产不出可靠的合法工具调用。
  • [HM] 「没工具=完成」不是编码任务的真正终止符——verify 门能悄悄推翻模型的停止决定，即使它自认为干完了，循环仍可继续。
  • [HM] 迭代预算是按 agent 计的：父(90)+每个子(50)，系统总迭代可远超父级 90；且预算耗尽后还有一次 grace call 收尾。
  • [HM] 纯文档改动（README/markdown/LICENSE）故意豁免验证门；门在 Telegram/Discord 等聊天场景默认关闭，避免把验证细节泄露给人——一条产品上下文感知的可靠性规则。
