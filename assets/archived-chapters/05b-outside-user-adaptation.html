它如何适应你

第二次找它，还要从头解释你是谁吗？

上半问「谁拥有代码」，把两家劈成两极。到了「它如何记你」这一侧，两个所有权相反的产品反而被同一道天花板逼到一处：谁都没把「懂你」训进权重——Claude Code 结构上不能、Hermes 有管线却没接你——只能退回壳里，用文件记你、趁你不在提炼、还都不敢全信这份记忆。共识是这副「壳内适应」的骨架；真正的分道，是把用户建模做多重：Hermes 建独立用户档案、开机主动提议建档；CC 把它揉进通用记忆、还能跨人在团队间共享。

01第二次找它，你还要从头解释你是谁吗？

一个 agent 会不会「长记性」，决定了它是每次归零的工具，还是越用越懂你的伙伴。

你昨天花十分钟跟它交代过：你是后端出身、第一次碰前端；你讨厌它擅自 mock 数据库；这个项目的怪规矩是「所有改动合成一个 PR、别拆」。今天再开一个新对话——它还记得吗？

这一篇拆的就是这件事：它跨会话保留什么、趁你不在怎么整理这些笔记、以及——它会不会把一次临时的坑，硬记成一条永远拒绝干活的规矩。

02「适应你」，其实是分时间尺度在壳里给你记笔记

壳内那套「记你」，落到实处不是玄妙的自我进化，而是在不同时间尺度上、趁你不在的时候做记忆家务。四个尺度从最细（边聊边记）到最粗（跨人共享），左蓝 CC、右紫 Hermes 并置。

尺度 1 · 会话之内 —— 边聊边记，不等结束
CC · 会话笔记
fork 一个子 agent 写下「你纠正了什么 / 哪些走法别再试」，上下文被压缩时重新注入回来
Hermes · 当场写盘
记忆立即落盘、持久保存；但为保前缀缓存，系统提示的快照要下次开会话才刷新
尺度 2 · 一轮结束 —— 答完你之后，后台悄悄提炼
CC · 后台提炼子 agent
这一轮你没顺手写、才由回合末的子 agent 补写，工具被锁死：只能读 + 只写记忆
Hermes · 复盘 fork
约每 10 轮起一个复盘分身回看整段对话，出厂发行版默认打开
尺度 3 · 跨天 / 机器空闲 —— 像睡觉时整理白天
CC · 夜间做梦
距上次 ≥24 小时且攒够 ≥5 个别会话才触发，四阶段巩固：去重 / 删被推翻的 / 修剪索引
Hermes · 空闲策展人
定期把长期没用到的技能归档，只归档、不删除
尺度 4 · 跨人 —— 一个人纠正，别人不必再纠正
CC · 团队记忆同步
同组织成员共享记忆，一人纠正、全队受益
Hermes · 无对应
自托管单机 · 记忆留在你自己机器上、不跨人
四个时间尺度，越往下越慢、越少人在场；只有最粗那层是 CC 独有

这台机器怎么记你，逐字看它给自己写的提示词。CC 的会话笔记有专门一栏问「你纠正了什么」；夜间做梦是一套编了号的四阶段流程；Hermes 的每轮复盘则被要求「主动学、别空手而归」。

CC 会话笔记模板 · src/services/SessionMemory/prompts.ts:27-28# Errors & Corrections _Errors encountered and how they were fixed. What did the user correct? What approaches failed and should not be tried again?_
CC 夜间做梦的四阶段 · consolidationPrompt.ts:26-61Phase 1 — Orient … Phase 2 — Gather recent signal … Phase 3 — Consolidate … Phase 4 — Prune and index
Hermes 每轮复盘被要求「要 ACTIVE」 · agent/background_review.py:171-175Be ACTIVE — most sessions produce at least one skill update … A pass that does nothing is a missed learning opportunity, not a neutral outcome.
一个非对称，别以偏概全：跨会话的记忆册（CC 四类记忆、Hermes 记忆文件）都是当场写、下次读——中途落盘立即持久，但要下个会话才被读进去用；CC 的会话笔记是个例外，它在同一会话被压缩时就重新注入回来。只有 Hermes 那条外接的用户建模是又一种时效：每轮活注入，请求模型之前现取现注一份画像。一句话给你记：跨会话记忆册当场写、下次读；外脑画你那条当场读、当场用——别混成一种。

03用户建模：做成一等子系统，还是揉进记忆？

这是下半最大的真分野，别抹平：Hermes 把「你是谁」做成一个独立子系统，CC 把它当记忆里的一类揉进去。

Hermes 单独给你开一本 用户档案（跟 agent 记自己教训的册子是两本、两套字数预算、系统提示里分两块渲染）；更关键的是开机就主动建档——你有史以来第一条消息，就会触发它主动提议建一份画像，且明确带一道「先问你同不同意」的门。它还留了 8 家可插拔的外接建模接口。

Hermes 开机主动建档指令（直击隐私顾虑） · agent/onboarding.py:157-183This is the user's very first message ever … OFFER — do not assume — to build a short profile … Never read their connected accounts (email, calendar) silently — ask each time … Save each confirmed fact with memory target='user'

呈现层的适应也做进一等公民：把你的挫败信号当头等事看待，把教训写进「管这一类任务的技能正文」里——而且规定必须写成声明、不写成命令。

Hermes 挫败信号 + 「声明非命令」写法规范 · background_review.py:180-238 · prompt_builder.py:162-167捕捉到的挫败信号（first-class）：'this is too verbose' / 'just give me the answer' 写法规范：'User prefers concise responses' ✓ — 'Always respond concisely' ✗

CC 没有独立画像文件，也没有开机主动访谈。它把「记你的偏好」和「记你反馈」当成通用记忆的两类，混在四类记忆里；每一轮用一个小模型从所有记忆标题里挑不超过 5 条注入当前对话。诚实说清（GAP-6）：CC 没有一条开机主动给你建档的通路（你手动造技能的流程里确会访谈你，但那是你触发、且为造技能而非画你这个人）；会话内的适应是「即时写 + 按需召回」的副产品。

CC 「用户」类记忆的用途原话 · src/memdir/memoryTypes.ts:121 / :136用户类：tailor your future behavior to the user's preferences … collaborate with a senior software engineer differently than a student 反馈类：so that the user does not need to offer the same guidance twice
CC 自动记录「该记你什么」的清单 · src/memdir/memdir.ts:350-355User corrections and preferences ('use bun, not npm'; 'stop summarizing diffs') … Facts about the user, their role, or their goals
分野一句话：Hermes 把「用户建模」做成一等公民——独立档案 + 主动建档 + 每轮活注入；CC 把它当记忆的一类，靠刻意分层（记忆明令不许存 CLAUDE.md 里已经写死的东西）保持「配置层 vs 适应层」不重叠。这正呼应上半——CC 自己就画了「你配置的」和「它适应的」之间那条界。

04会长记性，也会越用越固执

这是本章最非显然、也最值得单拎出来的一点：一个会自己攒记忆的 agent，不只越用越懂你，也会越用越固执——把一次临时故障硬记成永久规则，几个月后还拿它拒绝干活。两家都撞上了，但姿态相反。

Claude Code · 预防式：把「不信任记忆」焊进出厂设计

系统提示里专门有一段召回纪律：记忆提到某文件就先确认它还在、要据此行动就必须先核实；「记忆说 X 存在」不等于「X 现在存在」。夜间做梦也内建「删掉被现场推翻的旧事实」。它把「记忆天然会过期」当默认前提来设计。

用前必核实的原话 · memoryTypes.ts:253A memory that names a specific function, file, or flag is a claim that it existed *when the memory was written* … "The memory says X exists" is not the same as "X exists now."

Hermes · 事后式：被真实投诉后才补的护栏

后台复盘出过真实事故：把「这工具没装好」误记成「这工具坏了」，几个月后还拿这条错误结论拒绝干活——用户抱怨了。整改两手：复盘指令加一段「不要捕获」黑名单，再加一道记忆写入的审批闸。

复盘「不要捕获」黑名单（直击事故） · background_review.py:250-269Do NOT capture … Negative claims about tools or features ('X tool is broken'). These harden into refusals the agent cites against itself for months after the actual problem was fixed.
产品真理：一个会自己长记性的系统，必须同时有一套「忘记 / 不信」的纪律，否则记性就是负债。差别只在这一课在哪学会——CC 焊进实验室（出厂预防），Hermes 合在真实投诉里（事后补救）。

06差异在控制，共识在适应

同骨架：两家都把「适应你」落在壳内文件记忆 + 后台趁你不在提炼 + 不敢全信记忆这三件事上——因为谁都没把「懂你」训进权重。CC 结构上不能、Hermes 没接你的对话，为什么见『变强』章。

分道：差别退成把用户建模做多重——Hermes 独立档案 + 主动建档 + 活注入（自托管单用户陪伴，README 第一句就是 grows with you）；CC 揉进通用记忆 + 刻意分层 + 跨人团队（托管多租户）。

合上双向：差异在控制（上半 · 谁拥有代码，劈成两极），共识在适应形态（下半 · 同一道天花板，逼成同一副骨架）。上半你居高临下改它，下半它趁你不在记你——这两半合起来，才是完整的人机关系。

07悬而未决 · 防夸大红线

  • [CC · 别夸成会访谈你] CC 没有会话内的主动访谈建模——它的「会话内适应」是即时写 + 按需召回的副产品，不是「它会主动问你、给你建档」。
  • [两家 · 非对称别混说] 写盘的记忆是当场写、下次读；Hermes 外接 provider 是每轮活注入——两种时效不是一回事，别混成一种。
  • [Hermes · 默认的两副面孔] 出厂发行版默认开（复盘 fork、策展人）；但裸库缺省是防御性关，USER.md、外接建模、写入审批闸各有各的 opt-in 面，别把门后的功能当默认支柱。
  • [Hermes · 别撞名] 命令行里的 hermes profile 是多实例隔离（配置方向），跟这里说的用户档案名字撞车、概念相反，别混为一谈。
  • [不评优劣 · 时间切片] 本篇讲「适应的形态」，不评两条路孰优——取决于你要的是托管多租户产品，还是自托管的长期单用户陪伴。切片：CC main / Hermes @daf4f1a7a。
