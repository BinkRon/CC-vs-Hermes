# learn.shareai.run s01–s10 表达手法库

> 调研方法：`curl` 抓 10 页 SSR 源码分析真实标记（hero 字段、class 语义、SVG 结构、语义配色分布、表格/代码/卡片计数），并用浏览器实渲染 s01/s03/s08 三页（含逐步交互与切 tab）验证"重型可视化"的真实构造。未逐页渲染的页面，其重型组件按已验证的通用范式 + 该页 hero/配色/计数证据推断，下文如实标注"直接看到" vs"据范式推断"。

---

## 通用骨架（先讲清，逐页只讲差异）

全站 Tailwind + zinc 灰阶底 + 暗色模式，是一套**教材式模板**，10 页共用同一套骨架：

- **左侧全局目录**：整门课 20 讲按 5 大类分组，每类一个**彩色圆点**（工具与执行=蓝／规划与协调=绿 emerald／记忆管理=紫／并发=琥珀 amber／多 Agent=红）。这套"分类色"贯穿到每页 hero 的分类标签底色——**颜色即分类坐标**。
- **hero 固定配方**（下详）。
- **顶部常驻"重型交互件"**：紧跟 H2 的第一个大组件，永远是一个**分步模拟器**（图 + 状态面板 + 讲解框 + 步进器 N/总数）。
- **四标签页**：`学习 / 模拟 / 源码 / 深入探索`。学习=概念分步；模拟=可播放沙盘（Play/Step + 0.5x–4x 速度）；源码=真实代码；深入探索=另一张执行流程图。
- **页脚**：上一版/下一版链式导航。

## s01 · Agent Loop（直接渲染看到）

1. **主题**：最小 agent = 一个"调模型→跑工具→喂回结果"的 while 循环。
2. **Hero**：`s01`灰底 mono 徽标 + H1 + 蓝色分类标签｜英文副标"One Loop Is All You Need"｜元信息行`102 LOC · 1 个工具 · Minimal model/tool loop`｜灰色左边框斜体金句。极克制，一屏内交代完"是什么+多小+一句话精髓"。
3. **最重可视化**：顶部**双栏分步器**——左是 loop 流程图（Start→API Call→stop_reason?→Execute Tool→Append Result 回环），右是实时`messages[]`状态面板。步进时**当前节点点亮为蓝、其余全部压暗成灰**，右侧消息数组同步逐条增长，下方蓝框讲解同步换词（1/7）。凭什么重投入：循环的"随时间演化"是本页最难在静态图里说清的概念，只有让读者亲手 step 才懂。手法=分步交互 + 图/状态双栏联动 + 单一语义高亮。
4. **刻意从简**：流程图本体全灰（#cbd5e1/#e2e8f0），只留**一条蓝色高亮**标"活跃路径"——简单结构不上色，把唯一的颜色预算花在"此刻在哪"。
5. **排版节奏**：段落均长约 130 字、短促；卡片仅 4 个；粗体 23 处、适度。图重、字轻、留白足。
6. **可借鉴**：①"一张灰图 + 一处语义高亮"胜过满图配色——颜色只标读者此刻该看的那一处。②难懂的是"过程"就别画静态图，做成可 step 的状态机，让读者的手代替脑补。

## s02 · Tool Use（SSR + 内联 SVG 看到）

1. **主题**：给循环加能力 = 往"分发表"里注册工具，循环本身不动。
2. **Hero**：同配方，副标"Add a Tool, Add Just One Line"，元信息`Tool dispatch map`。
3. **最重可视化**：内联 SVG 是一张**分发扇出图**——`dispatch(name)`一点扇出到 bash / read_file / write_file / edit_file 四个工具，每个配一句真实描述（"Execute shell commands"…）。配合顶部分步器演示"注册即接入"。凭什么重投入：本页核心是"稳定的循环 vs 可增长的工具表"这层解耦，扇出图最能表达"一对多映射"。
4. **刻意从简**：工具描述用一行裸文带过，不做卡片；蓝色几乎独占（emerald/amber 各 1 次点缀），色板极窄。
5. **排版节奏**：段落均长 127 字；3 张卡片、3 个对照表；粗体 25。
6. **可借鉴**："一对多"关系用**扇出图**（中心节点 + 放射分支 + 每支一句白话）远比表格直观；把"改动多小"直接写进副标（Add Just One Line）当卖点。

## s03 · Permission（直接渲染看到，全站配色最讲究）

1. **主题**：危险动作在真正执行前，必须过 harness 的一个判定关口。
2. **Hero**：同配方，`180 LOC · 5 个工具 · Permission gate`，金句"Dangerous actions need a harness decision point"。
3. **最重可视化**：**三泳道流水线** `Tool requests → Permission desk → Outcome`（三栏各带图标：红警示／盾牌／播放）。左栏三张真实请求卡，**卡片底色/边框颜色 = 判定结果**：`read_file README.md`→绿卡→allow；`rm -rf ./tmp/build-cache`→琥珀卡→ask；`sudo rm -rf /`→红卡→deny。凭什么重投入：本页最难的是"哪条命令落到哪档"这套路由规则，用**颜色直接编码风险档位**后一眼可扫。手法=泳道流水线 + 语义配色即语义本身 + 真实命令原文当证据。
4. **刻意从简**：判定理由每条一句短语（"read-only workspace file"/"forbidden root delete"），不展开；右侧 Outcome 初始只留"select a request route"占位，不抢戏。
5. **排版节奏**：段落偏长（均 167 字，因要解释规则）；粗体最密之一（43 处，用来标 allow/ask/deny 等关键词）。
6. **可借鉴**：当概念本身有"档位/等级"时，**让颜色去承载那个等级**（绿=放行/黄=询问/红=拒绝），色即义、无需图例；用**真实危险命令**（`sudo rm -rf /`）当样本，比虚构例子更有说服力。

## s04 · Hooks（SSR 看到）

1. **主题**：横切行为应挂在循环"周围"，而非塞进循环内部。
2. **Hero**：同配方，副标"Hang on the Loop, Don't Write into It"，`Lifecycle hooks`。
3. **最重可视化**：据范式推断为**生命周期时点图**——沿循环标出若干挂载点（前/后/错误等），演示"钩子在不改主干的前提下被触发"。语义色 emerald 显著（12 次），应是用绿色标"被挂载/激活"的时点。凭什么重投入：本页卖点是"关注点分离"，需要空间上把"主干"和"挂点"画开。
4. **刻意从简**：语义卡片虽多（7，全站最多）但仍是小卡；红/紫仅各 1 次点缀。
5. **排版节奏**：段落均长 181 字（全站偏长，规则解释多）；代码块 9、Python 例 8，靠真实钩子代码兜细节。
6. **可借鉴**：把设计主张直接写成副标的**祈使对偶句**（挂在旁边、别写进去），一句话立住产品观点；"包在外面不改里面"这类架构主张，用"主干 vs 挂点"的空间分离图最好懂。

## s05 · TodoWrite（SSR 看到）

1. **主题**：没有显式计划的 agent 会跑偏；用待办清单让长任务可见可纠偏。
2. **Hero**：分类切到**绿色**（规划与协调），副标"An Agent Without a Plan Drifts Off Course"，`Todo manager`。
3. **最重可视化**：据范式推断为**状态流转的待办板**——emerald(17)+amber(9) 主导，应是用绿=已完成、琥珀=进行中给每条待办**着色标状态**，步进演示计划随执行推进而更新。凭什么重投入：本页核心是"计划是活的、随执行变化"，静态清单说不清，需分步展示状态迁移。
4. **刻意从简**：仅 1 张表、3 段 Python，把重心全压在状态板上。
5. **排版节奏**：段落均长 136 字；粗体 30，卡片 5。
6. **可借鉴**：**用状态色给列表项分档**（做完/在做/待做）把"进度"变成可扫描的颜色带；"反例开场"（没计划就跑偏）比"正面说教"更快让人认同必要性。

## s06 · Subagent（SSR 看到）

1. **主题**：把大任务拆成子任务，各自一份干净上下文，主线不被污染。
2. **Hero**：绿色分类，副标"Break Large Tasks into Small Ones with Clean Context"，`Isolated subtask context`。
3. **最重可视化**：据范式推断为**主线/支线上下文对照图**——蓝(18)为主，演示主线派生出隔离的子上下文、子任务完成后只回传结论。凭什么重投入：卖点是"上下文隔离"，必须在空间上把"主线消息历史"和"子线干净历史"画成两条泳道才说得清。
4. **刻意从简**：卡片仅 2（全站最少），不堆概念；4 个对照表承载"主 vs 子"差异。
5. **排版节奏**：段落均长 168 字；粗体仅 20（全站最低），克制。
6. **可借鉴**："隔离/派生"关系用**双泳道**（主线一条、支线一条、只在末端有一根回传箭头）表达"清爽上下文"最直观；对比性主张多用表格，把"主线 vs 子线"逐行摆平。

## s07 · Skills（SSR 看到）

1. **主题**：专项知识按需注入——只在任务真需要时才加载。
2. **Hero**：绿色分类，副标"Load Only When Needed"（全站最短副标之一），`On-demand skill loader`。
3. **最重可视化**：据范式推断为**按需加载触发图**——蓝/绿各 10 均衡，演示"触发词命中→技能才被拉进上下文"。凭什么重投入：核心张力是"能力很多 vs 上下文有限"，需要动态展示"平时不加载、命中才加载"。
4. **刻意从简**：段落数最少（20）、卡片仅 3；把"按需"这个简单机制用极短文案（Load Only When Needed）一句钉死。
5. **排版节奏**：段落均长 178 字但总量少；粗体 24。整页信息密度最低、最"轻"。
6. **可借鉴**：机制越简单，**副标越该短到像口号**（四个词讲完"何时加载"）；"按需 vs 常驻"这类懒加载概念，用"触发→加载"的时序动画比罗列条件清楚。

## s08 · Context Compact（直接渲染看到）

1. **主题**：上下文会被填满；用压缩让对话在窗口拥挤时仍可用。
2. **Hero**：分类切到**紫色**（记忆管理），副标"Context Will Fill Up"，`414 LOC · 9 个工具 · Context compaction`。
3. **最重可视化**：顶部分步器里一个**"Token 量表"复合视图**——左边竖直"Context Window"容器像量杯（30K/100K），顶部横向 Token usage 进度条（绿）配图例（user=蓝/assistant=灰/tool_result=绿三色点），右侧三张策略卡 Micro/Auto/Manual 各带`waiting`状态徽标 + 两行说明。步进时量杯灌满、进度条前进、策略依次激活。凭什么重投入：本页最难的是"什么时候、按哪种策略压缩"，量表把抽象的"窗口占用"变成可见的液面，色点图例让"谁占了多少"一眼分。
4. **刻意从简**：三张策略卡各只两行（"old tool_result / shrink bulky outputs"），不展开；量杯内部不画每条消息，只给液面高度。
5. **排版节奏**：段落最多（38）但均长 151、仍短句；粗体 29。
6. **可借鉴**：抽象的"容量/占用"用**拟物量表（量杯/进度条）+ 三色图例**，把不可见的资源变成可见液面；"多种策略"用**并列小卡 + 状态徽标**表现"谁在待命、谁被触发"。

## s09 · Memory（SSR 看到）

1. **主题**：有些事实要能扛过摘要压缩、跨 session 存活——留一层持久记忆。
2. **Hero**：紫色分类，副标"Keep a Layer That Doesn't Lose Details"，`Durable memory layer`。
3. **最重可视化**：据范式推断为**分层持久化图**——紫(9)主导，应是把"易失的对话上下文"与"持久记忆层"画成上下两层，演示"压缩时上层丢、下层留"。SVG 计数偏高(10 含图标)。凭什么重投入：与 s08 成对，核心是"哪层会丢、哪层不丢"的**层级对照**。
4. **刻意从简**：靠 5 张语义卡 + 4 张表分担，主图只表达"层"的关系不塞细节。
5. **排版节奏**：段落 35、均长 153；粗体 31。代码块 8，用真实记忆文件格式当证据。
6. **可借鉴**："持久 vs 易失"用**上下分层**（一层随时间蒸发、一层沉底不动）表达最省口舌；与相邻章（s08 压缩／s09 记忆）**共用一套视觉隐喻**，让成对概念可对照记忆。

## s10 · System Prompt（SSR 看到）

1. **主题**：系统提示是运行时**装配**出来的产物（策略+工具+技能+上下文），从不硬编码。
2. **Hero**：绿色分类，副标"Assembled at Runtime, Never Hardcoded"，`Runtime prompt assembly`。
3. **最重可视化**：据范式推断为**装配流水线/拼装图**——SVG 计数全站最高(14 含图标)，蓝(21)+绿(15) 双色最丰富，应是把多个来源（策略/工具/技能/上下文块）用不同色块**拼装成最终 prompt**，演示"运行时组装"。凭什么重投入：核心主张是"提示是被生成的、不是写死的"，必须展示"多源→拼装→成品"的组合过程。
4. **刻意从简**：段落均长回落到 136（短），把复杂度交给拼装图，文字只做旁白。
5. **排版节奏**：粗体最密之一（38，标各装配来源名）；代码块 9。
6. **可借鉴**："由多部分动态组装"用**分色积木拼装图**（每种来源一个色块，最后拼成一整条）表达"生成而非硬编码"；把"从不硬编码"这种反直觉主张直接放进副标做钩子。

## 横向共性：10 页反复出现的手法规律

**1. Hero 是一套零变体的"身份证"配方，克制到像填表。** 每页 H2 之前只有：`sNN`灰底 mono 徽标 + H1 + 分类色标签 + 一句英文副标 + 一行硬指标（`LOC · N 个工具 · 一句话定性`）+ 一句灰边框斜体金句。没有导语、没有配图、没有"本章你将学到"。指标行把"这套机制多小/多大"量化成可比数字（102 LOC vs 414 LOC），金句用作者原话一句话立论。**借鉴**：给每篇定一套固定 hero 字段表，逐篇只换值不换结构；用"一句英文精髓 + 一行硬指标"替代冗长引言。

**2. "重投入"永远砸在同一个位置、同一种形态：紧跟标题的分步模拟器。** 10 页无一例外，H2 后第一个大组件是"中心视觉隐喻 + 支撑卡/状态面板 + 同步讲解框 + 步进器 N/总数"的四件套，而且概念越"随时间演化/有状态迁移"（loop、permission 路由、token 填充），越用它。它不是插图，是让读者亲手 step 的状态机。**借鉴**：判断该不该给某概念"上重型交互"的标准是——它的难点是不是"过程/状态变化"；是，就做可 step 的联动件；不是，别浪费。

**3. 颜色是一套受纪律约束的语义系统，不是装饰。** 全局：五大类各一个固定色（蓝/绿/紫/琥珀/红），一路贯穿目录圆点→hero 分类标签，**颜色 = 内容坐标**。局部：颜色直接编码概念自身的档位/状态——permission 的绿 allow／黄 ask／红 deny、todo 的完成/进行、token 的三色来源。**同一页里绝不无意义用色**：s01 流程图整张压成灰，只留一条蓝标"此刻活跃"。**借鉴**：先给你的分类体系分配固定色并全局贯穿；页内的颜色只用来编码"读者此刻该分辨的那一个维度"（档位/状态/归属），其余一律灰。

**4. 盒子/卡片是稀缺资源，用得起就压得住。** 每页语义卡片仅 2–7 张，卡内文案压到一两行短语（"read-only workspace file"、"shrink bulky outputs"），从不把卡当段落容器。大量说明是**裸文短段**（均长 127–181 字），粗体只用于标关键词（allow/deny、来源名），不通篇加粗。**借鉴**：卡片只给"需要并列对比的同类项"（三种策略、三档判定），一屏别超过 3–4 组；卡内只放标签级短语，解释放裸文。

**5. 真实制品当证据，不用占位符糊弄。** 危险命令用真的`sudo rm -rf /`，工具描述用真的英文原文，token 用真的`30,000 / 100,000`，代码 tab 直接给真实 Python 源码。抽象主张（"危险动作要过关"）总被一个具体样本（一条真命令 + 它的判定）钉死。**借鉴**：每个论点配一个可验证的真实制品（真命令/真数字/真文案），"以实例证主张"是这套页面可信度的根。

**6. 结构极强的复用 + 成对概念共用隐喻。** 20 讲共用一套模板骨架（同 hero、同四标签、同页脚链导航），读者学一次导航一劳永逸；相邻/成对概念（s08 压缩 ↔ s09 记忆）刻意共用"层/容量"视觉隐喻，便于对照。**借鉴**：合订本各章共用一套骨架和交互约定（读者只需学一次怎么看）；成对/对立的概念用同一套图形语言呈现，让"差异"落在同一坐标系里可比。
