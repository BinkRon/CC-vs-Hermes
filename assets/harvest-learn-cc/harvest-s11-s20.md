# learn.shareai.run s11–s20 + compare + layers —— 表达手法库收割

> 说明：站点是 Next.js + Tailwind（zinc 灰阶为底 + 语义色点缀）+ dark 模式全覆盖 + hljs 代码高亮 + lucide 线性图标。sNN 章节页共用**一套极强的固定模板**，正文观察均已剔除左侧全站章节导航栏。overview 类主图多为外链 `.svg` 文件（文件名语义化，如 `team-topology.svg`），少量时间线类为内联 SVG。

## 通用模板（先记一次，后面各页只记差异）

- **固定骨架**：`问题 → 解决方案 → 工作原理（多个 H3 小机制）→ 相对 s(N-1) 的变更 → 试一下 → 接下来 → 深入 CC 源码`。七段式雷打不动，读者第二页起就形成肌肉记忆。
- **Hero 极克制**：眉标（所属层，如「多 Agent 平台」）＋ H1 中文标题 ＋ 一行英文副题（金句，如 *A task graph turns vague goals into ordered, observable work*）＋ 一行元信息徽章（`297 LOC · 8 个工具 · Task board`）＋ 一句斜体设计原则 ＋ `学习/模拟/源码` 三标签切换条。到第一个 H2 之前不放大图。
- **语义色小盒**：`问题`＝琥珀（amber-50/border-amber-200/text-amber-800），`提示/注记`＝蓝（blue-50），`解决/成功`＝翠绿（emerald）。都是 `rounded-md border p-3` 的轻量卡，不是重块。
- **变更表**：清一色 3 列 `组件 | 之前 (sN-1) | 之后 (sN)`，纯边框表、无单元格变色——靠「前/后」文字同行并置来表达差异。
- **深入源码**：折叠式密排代码块（hljs 高亮）＋ `file:line`，是全页信息密度最高、但视觉最"素"的区。

## s11 · Error Recovery
1. **主题**：Agent 遇 API/输出故障时，按错误类型分流到不同恢复路径并重试。
2. **Hero**：眉标+标题+英文副题+元信息（LOC/工具/主题）+原则句，无大图，克制。
3. **最重可视化**：一张"三恢复路径"概览图（输出截断 / 上下文超限 / 临时故障 三条并列路径，各带触发条件与恢复动作）。凭什么重投入——本章核心难点正是"不同故障要不同解法"，一张图把三分支一次讲清；配底部 1/6 分步 stepper 逐路径走。
4. **刻意从简**：两张对照表（恢复模式表、s10→s11 组件变更表）用纯文字网格；重试常量/阈值只在深入源码里以代码带过。
5. **排版节奏**：盒子少、裸文＋表格多；段落短，粗体只标术语；留白主要在 Hero 与每个 H2 之间。
6. **可借鉴**：① 当一个概念天然是"多条互斥分支"时，用并列路径图＋分步 stepper 让读者一条条走，胜过堆叙述。② 故障类问题先给"分类表（触发→动作）"当索引，再逐类展开。

## s12 · Task System
1. **主题**：把大目标拆成带依赖的持久化任务图（DAG），支撑跨 session 的多 Agent 协作。
2. **Hero**：元信息 `297 LOC · 8 工具 · Task board` ＋金句副题 ＋ 引用式原则句"大目标拆成小任务，排好序，持久化"。
3. **最重可视化**：**Task DAG 依赖图**＋ **Kanban 看板**（blocked→ready→working→done 四列）。凭什么重投入——"依赖决定可执行性"是本章最难直觉，DAG 把抽象依赖画成节点连线，看板把状态机落成可见泳道列。
4. **刻意从简**：TodoWrite vs Task System 的定位差异用 6 行对照表；TaskRecord 的 9 个字段用 `字段|类型|用途` 三列表——数据结构不画图，用表。
5. **排版节奏**：图—表交替，工作原理下 8 个 H3 小机制各配一小段＋一段代码；粗体密度中等。
6. **可借鉴**：① "状态"用看板列（泳道）表达，"依赖"用有向图表达——两种关系用两种专用图，不混在一张。② 同类概念对比（新旧两套机制）永远先上一张"维度×两方案"表定调。

## s13 · Background Tasks（对应我们 04 章）
1. **主题**：慢操作丢到后台线程异步跑，主 Agent 不空等、继续推理。
2. **Hero**：副题 *The agent can keep reasoning while slow work completes elsewhere*＋元信息，克制。
3. **最重可视化**：**Background Task Lanes**——一张**内联 SVG 时间线**，横轴 `t=0→time`，纵向三条并行泳道（Main Thread / Background 1 / Background 2）＋一条 Notification Queue，画出"主线程不停、后台并行、结果异步回注"。凭什么重投入——"异步并行＋结果何时回来"是纯文字最难讲清的时序概念，时间线泳道是它的天然表达。
4. **刻意从简**：同步 vs 后台仅用 4 维对照表；回调/通知细节留给深入源码。
5. **排版节奏**：一大图定乾坤，其余为短文＋表；留白突出那张时间线。
6. **可借鉴**：① 凡涉及"谁在什么时间点并行做什么"，上带时间轴的多泳道图（我们 04 章后台/定时几乎必用）。② 通知/结果回注画成一条独立泳道汇入主线，比箭头乱连清楚。

## s14 · Cron Scheduler（对应我们 04 章）
1. **主题**：用 cron 表达式让 Agent 任务按时自动创建，调度与执行解耦。
2. **Hero**：原则金句"周期性工作应由 harness 创建，而非靠模型记住"＋`645 LOC · 11 工具`。
3. **最重可视化**：**四层模型概览图**（Scheduler 守护线程 → cron_queue 队列 → Queue Processor → agent_loop 消费）。凭什么重投入——本章核心正是"调度/执行解耦"这条产品主张，四层管线图把"谁触发、排到哪、谁消费"一图讲透。
4. **刻意从简**：手动 vs 定时触发用 3×3 小表；cron 五字段匹配、jitter、过期等 9 个细节全压进深入源码的 9 个小节。
5. **排版节奏**：主图后 H3 密集（17 个），但每个只一小段；靠源码分节吸走复杂度，正文保持轻。
6. **可借鉴**：① "解耦"类主张最好落成一条**分层管线图**（每层一个职责盒＋箭头），让"解耦"从口号变成可见的层。② 大量实现细节统一下沉到"深入源码"分节，正文只留产品逻辑——正是我们 CLAUDE.md 的铁律。

## s15 · Agent Teams（对应我们 07 章）
1. **主题**：用持久化队友线程＋文件收件箱做多 Agent 协作，并行而不撑爆上下文。
2. **Hero**：立论"一个 Agent 不够"＋核心机制词"文件收件箱＋队友线程"，无大图。
3. **最重可视化**：**Team Topology 拓扑图**（Lead 居中，N 个队友环绕，画出 mailbox 路由与异步消息流向）。凭什么重投入——多智能体最难的是"谁跟谁怎么通信"的**结构**，网络拓扑图是通信结构的天然载体。
4. **刻意从简**：`s06 子 Agent vs s15 队友` 用 4 维对照表（生命周期/通信/上下文/规模）；权限冒泡等用短文。
5. **排版节奏**：拓扑图＋对照表＋一个 1/8 分步 stepper（Lead 同时 spawn Alice/Bob→各自干→回报 inbox）演示并行时序。
6. **可借鉴**：① 多 Agent 章必备两图：**拓扑图**（讲通信结构）＋**分步 stepper**（讲一次协作的时序）——静态结构与动态时序分开画。② "新旧两代机制"对照永远用维度表，把差异钉在同一批维度上。

## s16 · Team Protocols（对应我们 07 章）
1. **主题**：多 Agent 之间要显式的消息契约（请求-响应协议），而非"凭感觉"。
2. **Hero**：金句 *Multi-agent systems need explicit message contracts, not vibes*，直接抛出两个场景（优雅关停、计划审批）。
3. **最重可视化**：**交互式协议卡片 stepper**（4 卡轮播），逐步演示 shutdown 与 plan-approval 两条 `pending → approved/rejected` 状态机的流转与关联。凭什么重投入——"请求与响应如何配对、状态如何流转"是动态过程，静态图讲不动，用可点的分步卡片让读者亲手推进状态。
4. **刻意从简**：协议类型用 `类型|方向|用途` 表；overview.svg 只做架构总览一带而过。
5. **排版节奏**：一张总览图＋一个重交互 stepper＋两张表；粗体标协议名。
6. **可借鉴**：① **状态机**用"可点击推进的卡片 stepper"表达（每卡＝一个状态），比画静态状态图更让读者"体感"到流转。② 抽象契约先用两个具体场景锚定，再讲通用机制。

## s17 · Autonomous Agents
1. **主题**：Agent 从共享任务看板自主认领工作，无需 Lead 逐个派发。
2. **Hero**：问题直击"看板有 10 个未认领任务，Lead 就得手动派 10 次，不 scale"。
3. **最重可视化**：**Autonomous Agent Cycle 图**（A/B/C 三个 Agent 各带 idle 计时器，共同盯一个 4 任务看板，标注"没人直接派活"）。凭什么重投入——本章的核心范式转变是 **push（Lead 派发）→ pull（Agent 自取）**，这张图把抽象范式一眼画清，是最难也最中心的概念。
4. **刻意从简**：`阶段|行为|退出条件`（WORK/IDLE/SHUTDOWN）用三列表；轮询、身份重注入等用短文＋代码。
5. **排版节奏**：范式图定调＋生命周期表＋1/8 stepper 强化"从中央调度器到工作看板"的心智切换。
6. **可借鉴**：① 讲"范式转变"（A 模式→B 模式）时，画一张把两种模式并置对照的**心智模型图**，并用一句反问点破旧模式的痛点。② 生命周期用"阶段×行为×退出条件"表，读者一眼看全状态。

## s18 · Worktree Isolation
1. **主题**：用 Git worktree 给并行 Agent 各分独立目录，避免文件互踩。
2. **Hero**：金句"各干各的目录，互不干扰"＋点明 task 管分工、worktree 管目录、以 ID 绑定。
3. **最重可视化**：**Worktree Overview 图**（`.worktrees/auth-refactor/`、`.worktrees/ui-login/` 两个独立分支目录，从共享 `main` 仓库分叉隔离）。凭什么重投入——"隔离"是空间概念，用文件树/分支分叉图直观展示"各自的目录空间"。
4. **刻意从简**：`run_git()` 只当一行伪函数带过；事件日志用极简 JSON；s17→s18 变更用 6 行表。
5. **排版节奏**：一图＋一表＋1/6 stepper（从"单一工作区的痛"演进到隔离执行）；代码全留纯文本、不加装饰。
6. **可借鉴**：① "隔离/空间"类概念用**分叉的文件树/分支图**表达（共享根 → 各自枝），比文字描述边界清晰。② stepper 第一帧先演"不隔离会怎样撞车"，再演隔离方案——痛点前置。

## s19 · MCP Tools
1. **主题**：用 MCP 协议标准化接入外部工具，无需为每个服务重写封装代码。
2. **Hero**：问题描述"每接一个 Jira/部署/知识库都要手写 wrapper"，副题点出 discover+invoke。
3. **最重可视化**：本页**最重的是"深入源码"区**——`assembleToolPool()` 代码走读＋8 小节真实实现（Transport 类型、权限检查、OAuth 流程）。凭什么——MCP 的价值在工程实现细节，故重心放在密排代码而非概念图。概念层用一张 client-server 架构图＋4 行概念表即可。
4. **刻意从简**：1/6 stepper 刻意用最小视觉复杂度，只对比"内置单工具箱 vs 多服务器工具箱"的前后态。
5. **排版节奏**：轻概念（图＋小表）＋重实现（长代码分节），是全批里"重心在源码侧"的少数页。
6. **可借鉴**：① 当一个特性的难点在"工程实现"而非"概念直觉"时，允许把重投入放到代码走读区，概念图反而从简——**重投入跟着难点走，而非千篇一律都画大图**。② 集成类特性用"前：单一 / 后：多源"的极简对比 stepper 一句话讲清增量。

## s20 · Comprehensive Agent Turn（多机制串联的表达范本）
1. **主题**：把前 19 章机制整合进同一个 loop，呈现一次生产级 Agent 回合的全貌（1708 LOC · 32 工具）。
2. **Hero**：*All Mechanisms, One Loop* ＋收束金句"最终 harness 仍是一个 loop，只是被一圈让它生产就绪的系统包围"。
3. **最重可视化 / 多机制串联手法**：三件套配合——
   - **System Architecture 流程图**：`UserPromptSubmit 钩子 → 上下文压缩 → LLM → tool_use 分支 → 工具处理器 → PostToolUse 钩子 → 回到 loop`，一条主回路串起所有机制。
   - **Turn Packet Stepper（1/7）**：把一次回合切成 7 个有序阶段 `Intake → Guardrails → Route → Execute → External → Recover → Append`，每阶段列出输入/决策/输出——**用编号 stepper 承载"许多机制按序触发"，强调复杂度来自编排顺序而非单个机制**。
   - **Component Positioning 表**：把每个机制**映射到 loop 的具体位置**（pre-input / pre-LLM / during / post / stop），让"20 个机制装在一个 loop 的哪一环"落地。
4. **刻意从简**：各子系统（记忆/压缩/后台/worktree/MCP）在此只用一小节回指，不重画——整合章靠"定位表＋回指"复用前文，不重复造图。
5. **排版节奏**：全书最长页（144KB，20 个 SVG），但结构仍是"一主图定回路 + 一 stepper 走时序 + 一表定位置"三支柱。
6. **可借鉴**：① **多机制串联的黄金三件套**：主流程图（讲回路）＋编号 stepper（讲一次执行的先后）＋"机制×loop 位置"定位表（讲各归其位）。② 整合/总章不重画细节图，用"位置表＋回指前文"把复杂度组织起来——这正是我们全书"综合①/②"该学的收束手法。

## compare · 版本对比（"两对象对比"的表达，与我们全书同构）
1. **主题**：任选两个版本（s01–s20），查看它们之间的变化。
2. **Hero 极简**：标题「版本对比」下只有一行说明"查看任意两个版本之间的变化"＋两个下拉框「版本 A」「版本 B」。留白极大、信息密度极低——**对比页把"选择"做成唯一入口，正文由选择动态生成**。
3. **对比呈现机制**：diff 内容**完全客户端渲染**（服务端 HTML 只有两个下拉与一句提示；每个版本的完整源码被打进 JS chunk 供前端算差异）。⚠️ 诚实标注：受浏览器被占用限制，我**未能抓到最终渲染后的 diff 视觉样式**；但可确证站点**通篇的"原子对比单元"**就是每个 sNN 页那张 **3 列 `维度 | 之前 | 之后` 纯边框表**——无单元格变色，靠"前/后同行并置"表达差异。compare 页在概念上是这张表的动态化版本（选 A/B → 生成维度对照）。
4. **信息密度**：入口页极空（airy），把密度延迟到"选完之后"才释放——渐进式，不一上来就压人。
5. **排版**：下拉框＋提示，无盒子无图，是全站最"空"的页。
6. **可借鉴**：① 对比的**原子单元是"维度行 × 两方并列列"**，差异靠"同行左右并置"而非红绿高亮就已足够清楚（我们全书 CC vs Hermes 可直接沿用这套三列表骨架）。② 若对比维度多，可学它把"选择/聚焦"做成入口、密度延迟释放，避免一屏塞满。

## layers · 架构层（分层信息架构的表达）
1. **主题**：五个正交关注点（L1–L5）组合成完整的 Agent。
2. **Hero**：立论句"五个正交关注点组合成完整的 Agent"，直接进入分层。
3. **最重可视化 / 分层手法**：**5 条色彩编码的横向层带**（L1 Tools&Execution / L2 Planning&Control / L3 Memory / L4 Concurrency / L5 Multi-Agent），每层一个专属主色（约为 红 / 翠绿 / 蓝 / 紫 / 琥珀，light 用 `bg-100`、dark 用 `bg-900`、`-500` 作强调点）。每条层带内是一排**模块卡片**（20 个 s01–s20 模块各一卡，卡上带 `LOC · 工具数` 徽章，可点进章节）。共 26 个 SVG＝每个模块的小缩略图标。
4. **刻意从简**：无表格、无长文——整页就是"色带＋卡片网格"，靠颜色和空间分组承载全部信息架构。
5. **排版节奏**：盒子（卡）占绝对主导，色彩＝分类，留白在层带之间；粗体只在层标题与徽章数字。
6. **可借鉴**：① **用"颜色＝类别"给一组同级对象分层**：每个类别一个稳定主色，卡片继承该色，读者靠颜色即完成归类（我们轴一四侧/轴二四议题若做总览页可套用）。② 每张卡挂一两个关键量化徽章（LOC/工具数），让"总览"同时携带可比的量纲。

## 横向共性（反复出现的手法规律 + 对比表达专记）

**1. 一套模板贯穿到底，靠"重复"降认知负荷。** 十页共用 `问题→解决方案→工作原理→变更→试一下→接下来→深入源码` 七段式。读者第二页起就知道每段在哪、该期待什么——**结构本身成了导航**。启示：我们全书各章也该锁死一套段落骨架，让读者一次学会、通篇复用。

**2. Hero 极度克制，是全站的呼吸点。** 眉标（层）＋标题＋一行英文金句＋一行量化徽章（`LOC · 工具数 · 主题`）＋一句斜体原则，到第一个 H2 前绝不放大图。把"最重的图"留到"解决方案/工作原理"里，Hero 只负责定位与立论。启示：章首不堆图，先用一行金句＋几个关键数字把"这章解决什么"钉住。

**3. "重投入跟着难点走"，不是每页平均用力。** 每页只有**一张真正的重头图**，且必然对准该页最反直觉的概念：时序难点→**带时间轴的多泳道图**（s13）；关系/依赖难点→**图/拓扑**（s12 DAG、s15 topology）；状态流转难点→**可点击推进的分步 stepper**（s16 协议、s11 路径）；范式转变→**双模式并置心智图**（s17）；空间隔离→**分叉文件树**（s18）；而当难点其实在工程实现（s19 MCP），就**放弃大图、把重心给代码走读**。启示：先判断"这维度最难的是时序/关系/状态/范式/空间/实现哪一类"，再挑对应图式——**图式与难点类型一一对应**。

**4. 语义配色是克制的三色制，不是彩虹。** 灰阶（zinc）打底，只用琥珀＝问题/警示、蓝＝提示/注记、翠绿＝解决/成功 三种语义色的浅底小盒；layers 页则升级为"每类一主色"的分层色带。颜色永远承载语义（状态/类别），从不为装饰。启示：我们的语义色也应固定映射（如"改模型 vs 改壳"、"CC vs Hermes"各一主色），全书一致。

**5. 交互只用一种原语：编号 stepper（1/N 轮播）。** 凡"一次执行的先后顺序"或"状态如何流转"都用同一个可点推进的分步卡片，从不发明新控件。s20 更用 7 步 stepper 承载"多机制按序触发"的全景。启示：交互组件收敛到一种（我们的 stepper），到处复用、不造新轮子——正合我们"点一次 stepper"的验收铁律。

**6. 复杂度统一下沉到"深入源码"，正文只留产品逻辑。** 大量实现细节（阈值、算法、OAuth 流程）压进末尾折叠的密排代码分节，正文与图只讲"为什么/是什么"。启示：与我们 CLAUDE.md"正文不出现函数名、代码只在页脚 file:line"完全同构——可放心照此分层。

**7. 整合/总章不重画，用"定位表＋回指"收束。** s20 面对 20 个机制，不重画每个，而是用"机制×loop 位置"定位表把它们各归其位＋一句话回指前章。启示：我们的"综合①/②"两篇同理——不重复各分篇的图，用一张"侧/议题 × 位置或轴"的定位矩阵把散点收成一张网。

### 「版本对比」页的对比表达 —— 专记

- **对比的原子单元 = 3 列表：`维度 | 之前 | 之后`。** 这是站点通篇（每个 sNN 的"相对变更"表 + compare 页动态版）唯一的对比骨架。左列钉维度，右侧两列并置两方取值。
- **差异靠"同行左右并置"表达，不靠红绿高亮。** 变更表**无单元格变色**——把"前值/后值"放在同一行相邻两格，读者横扫即得差异。这比满屏红绿更冷静、也更耐读（红绿高亮易在维度多时变噪）。
- **compare 页把"选择"做成入口，密度延迟释放。** 入口只有两个下拉（版本 A / 版本 B）＋一句提示，极空；diff 正文选完才客户端生成。渐进式披露，避免一屏压人。
- **对我们全书（CC vs Hermes 同构）的直接搬用**：
  1. 每个对比点用 **`维度 | Claude Code | Hermes` 三列表**当骨架，先立维度、再并置两方——不必发明花哨的并排卡。
  2. **差异优先用"同行并置"承载**；只有当某行确实是"一方有/一方无"这种质变时，才用一个语义色徽章（如翠绿"新增"/灰"无"）点一下，**克制使用高亮**。
  3. 维度多时，考虑"选择/聚焦"式入口或分组，把信息密度**延迟到读者聚焦某组后**再释放，而非一次全摊开。
  4. 对比表与"分层色带 / 定位矩阵"配合：**表讲逐维差异，色带/矩阵讲全局格局**——两种尺度各司其职。
