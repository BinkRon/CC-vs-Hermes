轴一 · 控制侧Loop 之内

核心 Agent Loop

模型每次只活一次调用；所谓连干六十轮，是壳一次次把它重新唤醒。

模型一次调用只活一轮  ·  续命重试几乎清一色上限 3  ·  CC 高压纪律全在条件层  ·  按模型加压是 Hermes 独有轴（名单 8 家）

「连续工作」不是模型的能力，是壳用三层防线撑起的一种幻觉：机械层一次次把模型重新唤醒，纪律层用写给模型的话拦住它偷懒，强制层用代码兜住它出错或想停。

最大的分歧藏在上面两层——Claude Code 默认赌模型自觉，Hermes 用壳去补模型的不足。一个默认赌模型，一个补模型。

本章只讲「循环为什么转下去」；上下文怎么拼、压缩怎么做见 02 章，输出块怎么处置、每类错误怎么办见 03 章。

The illusion

模型说完即死，『连续工作』从何而来

一个大模型的本质是一问一答：给它一段文字，它吐出一段文字，然后这次调用就结束了——它没有后台进程，也没有留在原地等下一步的能力。可我们看到的 agent 却能连着干六十轮：读代码、改文件、跑测试、看到报错再改。这不是模型进化出了持久力，是壳制造出来的一种幻觉。

机关很朴素。模型这一轮想调什么工具，壳就替它把工具真的跑一遍，把结果拼回那段对话历史，然后带着整段加长的历史再发起一次全新的调用。模型所谓的「记忆」，其实就是每一轮被重新喂进去的那份历史——两家做法一模一样：上一轮的历史、模型这轮说的话、工具的执行结果，三段接龙成下一次调用的输入。

于是「能一直工作」可以拆成三个互不相同的问题：壳凭什么再调用一次（机械层）？模型凭什么不偷懒、不提前收工（纪律层）？模型想停下、或者要崩的时候，壳怎么办（强制层）？这三个问题的答案，就是本章要拆的三层防线。

两个补充。其一，物理上「agent 在工作」，是壳在一段段地把一个生成器往前摇——正因如此，中断、压缩、恢复才能干净地插在两段调用之间。其二，会话越滚越长，迟早撑爆模型的上下文窗口；让长会话不爆窗口靠的是上下文压缩——那是维持循环能转下去的地基，细节留给 02 章。

Layer 1

第一层·机械续命：继续还是收工，只看一个开关

先看最底层的机械。两家的判断同构到近乎重合：模型这一轮只要调用了任何一个工具，壳就判定「话还没说完」，再发起一轮；一个工具都没调、只吐了一段纯文字，壳才认为它真的说完了，走收尾。工具调用的有无，就是「继续还是停」的唯一开关——Claude Code 的注释直接把它称作 the sole loop-exit signal。

Hermes 在每一圈重发之前还多设了四道闸：这一回合的调用数没到上限、跨回合的迭代预算还有余额、没有收到用户中断、预算耗尽时是否还留着一次收尾调用——任何一道不满足，循环当场跳出。但主干上的那个开关，和 Claude Code 是同一个。

Layer 2 · Deep dive

第二层·提示词纪律：都看人下菜，只是看的不是同一个『人』

模型会偷懒。它会交出一份计划就想收工，会说一句「我这就去做」然后停手——这些是壳设计者反复见到的失败模式。两家的对策都是往系统提示里写「别罢工」守则，但关键在于：没有一家对所有场合用同一份守则。

Claude Code 的默认守则其实相当温和——失败别急着放弃、危险动作先确认（更硬的「如实汇报别谎报完成」只进了内部构建）。真正的高压话术全部锁在条件层：人不在场时才有「偏向行动别问」「终端在后台就大胆自主」；用户亲手开了功能才有「token 目标是硬下限」。Hermes 的常驻守则是一句朴素的「把活干完、别交桩」；更狠的「必须立刻调工具、别用一句承诺结束回合」只喂给一份弱模型名单（gpt、codex、gemini、gemma、grok、glm、qwen、deepseek 八家），而 Claude 这类强模型反倒不注入。

四格摆在一起，差别不在「加不加压」——两家都按场景和功能分层加压（CC 有自主模式，Hermes 有编码岗守则和无人值守协议）——而在 Hermes 比 CC 多出一根加压轴。Claude Code 只面对一个自家托管的模型，「模型行不行」根本不是变量，它的条件层主要跟着「人在不在看」走：终端前台还是后台、交互还是自主。Hermes 要跑一整群第三方模型，「模型行不行」就成了它独有的第一变量——纪律按模型名单发放，弱的多压、强的少压。

连催促待办的方式也照这条线分。Claude Code 的壳会在待办搁置太久时周期性塞一句软提醒（旧版待办工具还在每次回执后附一句「请继续推进」）；Hermes 刻意不让待办挂任何提示——引导只写在工具说明里，把主动催促的额度留给记忆蒸馏和技能自造（见轴二）。

诚实边界：Claude Code 生态里最锋利的那段话术——「回合结束前，检查你最后一段是不是一句没兑现的承诺」——并不在 CLI 代码库里。它是 Agent SDK／多智能体运行时另外叠加的一层，详见文末悬而未决。

Layer 3 · Deep dive

第三层·壳的强制：防死是共识，防懒是分歧

前两层管「该不该继续」，第三层管「拦不拦得住意外」。模型随时可能「死」：工具抛错、吐出空响应、话说到一半被截断、API 调用失败。两家的壳都不让它真的死——把错误包成一条工具结果回喂给模型，让它自己看到报错再改，然后重试、切备用模型、压缩后重跑。Hermes 兜得更低：连空输出、只思考不说话、工具名拼错、参数 JSON 损坏、流被网络掐断，都各有一条专门的恢复路径——弱模型的种种毛病，壳都得替它治。

每一类错误具体怎么处置，是 03 章的事。这里只需记住一件事：在两家眼里，错误都不是终点，是下一轮的输入。

分歧出现在「防懒」。模型想收工——不调工具、交出一段纯文字答案——Claude Code 默认放行，一道完成度检查都没有；Hermes 默认设一道验证门。这道门最能说明「可靠性焊进壳、而不是托付给权重」，值得逐帧拆开看：模型说「我修好了」，只是一个待核实的声明，不是结论。

The brakes

反面：每个续命机制都自带刹车

无限续命等于无限烧钱。真正的产品智慧不是「一直续」，而是分清两种失败：可恢复的失败（空输出、网络断、工具名拼错）值得猛续，确定性的失败（安全拒答、鉴权错误、压缩也救不回）就该干净地停——重试一个确定性失败，只会把它原样复现一遍。下面这张表把两家的每个刹车并在一起看。

The verdict

谁在防它罢工：一个赌模型自觉，一个用壳补模型

三层防线上的投入分布：机械层同构；纪律层都看人下菜，但看的变量不同——一个看场景（人在不在场），一个看场景加模型（还看名单）；强制层防死是共识、防懒是分歧（Claude Code 默认 0 门、Hermes 默认 1 门）。归因还是那条信任线：Claude Code 押注一个可控的托管模型——模型够强，纪律就能温和，防懒就能交给模型自觉；Hermes 面对的是一群良莠不齐的第三方模型——纪律得按名单发放，防懒只能焊进壳里。

一个反直觉的落点：做模型的那家，反而更不信任模型说的「我做完了」。越清楚权重的脾气，越不敢把「完成」的判断权交出去。

往下一章看：Claude Code 加压的那个变量——「人在不在场」——不是本章的终点。这根线在轴二「Loop 之外」会长成一整条脊椎：壳怎么随人的在场与否，动态调整它对模型的信任与放手。

Loose ends

悬而未决：九条别被主线吞掉的边界

轴一 · 输入侧Loop 之内

每轮那个请求怎么拼出来

壳每一轮要现拼一个发给模型的请求包——这个包分三层，你以为每轮全新，其实八成逐字复用。

一次请求 = 静态前缀 + 工具 + 历史与每轮注入  ·  两家都冻前缀吃缓存  ·  CC 面对 1 个模型 / Hermes 前缀按模型名分叉  ·  压缩都「先删后摘」外包给 LLM  ·  训回权重是 Hermes 独有

拼同一个请求包，两家的手艺八成是重合的——都为吃缓存把稳定内容冻在最前，都把每轮变化的东西塞进消息窗口、变了才补，都跑几乎一样的「先删后摘」压缩阶梯。

真正的分叉都挤在同几条缝上，而每条缝都在泄露它到底是什么：Claude Code 首先是个产品（动态层塞满 IDE、终端、任务的信号）；Hermes 首先是个面对一群自家开源模型的拥有者（连冻在最前的『静态』前缀都得按模型分叉，且只有它能把压缩训回权重）。

本章只拆「每轮那个请求包怎么拼、分几层」；循环为什么能一直转下去见 01 章；记忆怎么写、怎么蒸馏、跨会话怎么演化的完整生命周期见 08／05 章。

The illusion

你以为每轮是一份新输入，其实八成是逐字搬过来的

直觉里，模型每轮"看到"的，应该是一份为这一轮现写的输入。实际恰恰相反——壳每一轮都在拼同一个请求包，而这个包的绝大部分，是上一轮逐字不变地搬过来的。

原因是钱和速度。给模型发一次请求，最贵的成本是让它把长长的前缀重新"读"一遍；只要前缀一个字节都没变，服务端就能"认出这叠没动过"、直接命中缓存、又快又省（这就是 prompt cache）。于是两家做同一件事：把几乎不变的东西（身份守则、工具目录）死死钉在最前面当"冻结前缀"，把每轮要变的东西全赶到后面去——为的就是别惊动缓存。

所以"每轮那个请求怎么拼"，可以干净地拆成三层：最前面冻起来吃缓存的静态层，后面每轮现塞进去的动态层，以及当整个包快撑爆窗口时才启动的压缩应急阀。这一章就沿这三层各走一遍——你会看到两家在每一层都先高度趋同，再在同一处分叉。

The anatomy

一次请求长什么样：两家共享的三层装配骨架

先把一次请求摊平成一张图。最该看清的不是每一格写了什么，而是两家的装配骨架几乎是同一张——差异不在骨架，而在骨架的每一层内部（后面三节逐层拆）。自上而下就是发给模型的顺序：冻结的静态层在最前吃缓存，动态层每轮补在后面，压缩阀只在快撑爆时才动。

Layer 1 · 静态层

冻结吃缓存是共识，但 Hermes 连『静态』前缀都得看模型下菜

两家都把稳定内容冻在最前面吃缓存，逻辑一模一样。Claude Code 用一个字面分界符 __SYSTEM_PROMPT_DYNAMIC_BOUNDARY__ 把系统提示词切开，只有它之前的部分（身份→任务政策→执行守则→工具用法→语气）跨会话冻结进全局缓存；Hermes 干脆把系统提示词显式设计成三层——stable / context / volatile，拼成一整串、每会话只建一次、之后逐字重放，代码注释直说这是「唯一能让上游缓存保温的办法」。到这里，两家是同一套缓存经济学。

分叉在于——Claude Code 只面对一个自家托管的模型，这段"静态"前缀就是真·统一的。Hermes 要伺候一群异质开源模型，它的"静态"前缀根本不是一整块：命中 gemini/gemma 就注入一段 Google 模型执行守则，命中 gpt/codex/grok 换一段，遇到阿里的模型还要专门写一句「你其实是某某模型」去纠正 API 返回的错名字，弱模型名单上的还要再加一段工具强制纪律。两个不同模型，拿到的"静态层"实质是不同的。

Google 系（gemini / gemma）
注入一段专属执行守则——自主干到彻底解决、别给个计划就停
OpenAI · Grok 系（gpt / codex / grok）
换一段执行铁律——工具结果空就换姿势重试、别早停
阿里系（alibaba provider）
专门写「你其实是 X 模型」——纠正该 provider 返回的错误模型名
弱模型名单（tool-enforcement）
再加一段工具使用强制——必须同一条回复里立刻调工具

更彻底的一层：连"缓存"这件事，都得挑模型/provider。显式的 cache_control 断点主要给 Anthropic 系、外加少数几个经特定网关的兼容 provider（如 Qwen / MiniMax / 阿里）开着；更多本地/开源模型吃不到，只能退而求其次——靠把消息空白和工具调用的 JSON 规范成"逐字节一致"，去蹭底层推理引擎（llama.cpp / vLLM / Ollama）的 KV 复用。同一套省钱机制，不同模型享受的档位不同。

接回轴一：这正是 01 章那根「按模型加压」的独有轴，在装配层的第二次现身——CC 只面对一个模型，前缀能真·统一；Hermes 面对一群模型，连冻在最前面的料、连吃不吃得到缓存，都得看是哪个模型。所有权与异质性，从纪律层一路渗到了装配层。
Layer 2 · 动态层

每轮现塞什么：都塞进消息窗口、变了才补——但塞的东西暴露了产品 vs 底座

冻结前缀之外，才是每轮真正变的东西。两家的机制惊人地一致：都不把这些新内容写回那个冻结的前缀（否则缓存全废），而是塞进消息窗口——Claude Code 包成一个个 <system-reminder> 的"合成 user 回合"（模型看得见、你在对话里看不见），Hermes 直接追加到当轮 user 消息的尾部。而且两家都只在内容变了才补：CC 用一本 readFileState 台账记住每个文件上次的样子、只发变化的，还给这些信号限流（待办/任务每 10 轮、计划/自主模式每 5 轮才补一次）；Hermes 在回合开场一次性预取、整轮复用、不重复查。

分叉在于——塞进去的都是些什么。Claude Code 每轮可注入的信号有四十多种，绝大多数是产品面信号：被外部或 linter 改动过的文件、IDE 报来的 lint/类型诊断、你在编辑器里选中的行、打开的文件、待办/计划/自主模式状态、钩子输出、伴生宠物、队友信箱……Hermes 每轮的主注入只有八种左右，都是底座信号：记忆召回、插件 hook、/steer 打断、几个网关注入。

Memory

记忆每轮怎么进请求：一个当便签，一个当底稿

记忆也是每轮装配的一部分，但两家塞法不同，正好照出两种记忆观。Claude Code 每轮对你这句话做语义召回、最多挑 5 条相关记忆，包在会标注"存于几天前"的 <system-reminder> 里（而且这条每轮召回本地默认是关的，要服务端开旗才生效），并反复叮嘱"用前先读当前文件验证"——它把记忆当成会过期的便签。Hermes 则把整本 MEMORY.md 和 USER.md 原样冻进系统提示词的易变层、在压缩交接里标为 authoritative（不被摘要覆盖），外部记忆插件召回的片段也裹着一句「Treat as authoritative reference data」——它把记忆当成不该被冲淡的底稿。一个防陈旧，一个保权威。

这里只讲"记忆怎么进这一轮请求"这一个切面；记忆怎么写、怎么蒸馏、怎么跨会话演化的完整生命周期，是轴二的事——见 08「Loop 与自己」／05「Loop 与用户」。本章不重复展开，以免与后文双写。

The verdict

装配是共享手艺，分叉泄露身份

三层都先趋同、再在同一处分叉：静态层都冻结吃缓存，动态层都塞进消息窗口且变了才补，压缩层都跑"先删后摘、外包 LLM"的同一条阶梯。请求装配这件事，八成是两家共享的工程手艺——别把它想成两套哲学。

但三条分叉缝，是两个不同的根因，不是一个：静态层的分叉（前缀按模型分叉、缓存挑模型家族）和压缩层的分叉（只有它能把压缩训回权重），根子都是模型的所有权与异质性——呼应轴一脊椎；而动态层的分叉（四十多 vs 八种注入），根子是产品 vs 底座，是另一回事。我没有把三层硬收进一个漂亮的单一根因——那样会把动态层那条缝掰歪。

所以两家在输入侧的"身份"是：Claude Code 首先是个产品（它的动态层为 IDE/终端/任务而生），Hermes 首先是个面对模型动物园的拥有者（它的静态层为异质模型而分叉、它的压缩层为自家权重留了训练接口）。同一个请求包，装法八成一样；不一样的地方，恰好各自暴露了自己是什么。

Loose ends

悬而未决：八条别被主线吞掉的边界

  • [边界] "CC 约 44 种注入"是满配上限，外部默认 build 远没这么多——snip、cached-microcompact、每轮语义记忆召回、验证代理等为内部灰度或默认关；Hermes 的 8 种也含 MoA/prefill/网关注入等只在网关模式出现的项。别把"能注入的种类"当成"每轮都注入"。
  • [HM] Hermes 那层叫"volatile 易变"的其实名不副实：记忆快照/USER.md/时间戳虽叫易变，却冻在缓存的系统提示词内部，只在压缩/重建时才刷新，并非每轮变；真正每轮变的在消息窗口。所以"静态/动态"并不等于"系统提示词/volatile 层"。
  • [CC] "静态前缀"也不是整个系统提示词：分界符之后的记忆索引/环境/语言/输出风格虽在同一个 system 数组里，却按会话变、不进全局缓存。CLAUDE.md 更是根本不在系统提示词正文里，而是搭在一条合成的首条 user 消息上；git 状态又落在系统提示词尾部——三样东西分居三处。
  • [CC] prompt cache 其实有三套模式（自家全局缓存 / 含 MCP / 第三方），断点位置和块数都不同；而且"缓存断点检测器"只是事后监控 break 原因、不负责放断点。"缓存断点打在哪"没有单一答案。
  • [边界·压缩] CC 的压缩不止"整段摘要"一种：默认的 auto-compact 确实丢中段原文，但还有 partial 压缩（保留首尾原文消息）、session-memory、context-collapse 等路径（多为手动或特性门后）。"CC 只会整段摘要"只在默认路径成立。
  • [HM] 别把运行时压缩器和训练管线当成同一套：它们是两个独立模块、代码不共用，也不吐同一个字符串——运行时如今实际吐「[CONTEXT COMPACTION — REFERENCE ONLY]」长前缀，训练管线吐「[CONTEXT SUMMARY]:」，后者只是运行时至今仍认得的 legacy 格式。这道"同源"痕迹能证的是格式同源与管线存在，证不到线上权重真训过——第二根杠杆的权重级效果是意图、非本仓库可证。
  • [HM] Hermes 的请求其实有四套 api_mode（chat-completions / codex / anthropic / bedrock），序列化路径各不同，本章讲的是默认 chat-completions 那条；给 GPT-5/Codex 的系统提示词还会以 developer 角色而非 system 角色发出。"一次请求"并不是单一形状。
  • [边界] 压缩触发阈值两家都是派生值、不是固定百分比：CC≈有效窗口−13k（200k 模型约 83%），Hermes≈有效预算 50%、地板 64k、小窗退到 85%。数字差别背后是"多晚才肯动手"的不同保守度——Hermes 明显更早出手。

轴一 · 输出行动侧Loop 之内

模型吐生料，壳每轮加工两样成品

一样是能用的行动，一样是能回传给模型的历史。

一次响应 = 多块混排（推理 / 说给你听 / 要动手）  ·  壳每轮开两条加工线  ·  底层协议地板两家共享  ·  上层 Hermes 多铺两层（且两股根因不同）

模型一次吐出的不是一句话，是一叠异构块——有它自己想的（推理）、有说给你听的（正文）、有要动手的（工具调用），混在一坨里。壳每一轮干的是两件加工活：把这叠生料变成 ① 能用的行动、② 能合法回传给模型的历史。

两条加工线结构一模一样——底层一层协议地板两家共享（CC 这层一点不薄，回传前重塑是它最厚的工程）；上面 Hermes 各多铺一层：行动线多铺弱模型语义纠偏、历史线多铺异质 provider 适配。这是两股不同的根因，不是一股。CC 之所以能不铺，只因它只伺候一个受托管模型 + 一个 API。

本章只讲「模型吐的块本轮怎么消费、回传前怎么规整」；循环为什么不致死、每类错误怎么续命见 01 章；每轮新塞什么进请求、历史怎么压缩见 02 章。

The raw material

你以为模型回你一句话，其实它吐回来一叠块

你让 agent 改一行配置。它「回答」你的时候，底层其实一次性吐回了几段性质完全不同的东西：一段它自己的盘算（该改哪、会不会连累别处）、一句说给你听的话（「我来改」）、还有一条真要落到磁盘上的动作（改 config 第 12 行）。这三样挤在同一次响应里，壳收到的是一叠混排的块，不是一句干净的指令。

于是壳每一轮都要把这叠生料拆开、分类、分别加工。哪些是「要动手」的，得挑出来真去执行——这条是行动线；而这一整轮（它想的、说的、干的，加上工具跑出来的结果）又必须原样塞回下一次调用，好让模型「记得」自己刚才干了啥——这条是历史线。

本章就沿这两条线各走一遍。你会看到：两条线的底层地板两家几乎一样，真正的分野都在上面那薄薄一层——而且行动线和历史线上那一层，是被两件完全不同的事逼出来的。

Anatomy A

先把模型一次响应摊开。它不是一段文字，是一叠块：推理块、正文块、工具调用块，可以混排、可以多个。壳收到后第一件事就是逐块认领——而每一块都有两种命运要分别交代：这一轮怎么用它（行动线的活）、怎么把它塞回历史（历史线的活）。下面这张表，就是后面两条线的预告。

Line 1 · Action

行动线：把「要动手」的块，变成真落地的一次操作

行动线只认那叠块里要动手的那种。壳把它挑出来、按名字找到对应工具、过一道权限门、真去改文件或跑命令，再把结果翻译成模型看得懂的块送回去。这条线的地板（找不到工具/参数不合法怎么办、并发怎么排、结果太大怎么办）两家几乎一样，下一节的主图会并排摊开。这里先交代一件事：地板之上，Hermes 多铺了一层——它不敢假设模型总能把工具名喊对。

具体是三种条件触发的纠偏（不是每轮都做，别读成「Hermes 总在猜」）：工具名拼错了就模糊匹配改对；工具名空着——壳判定这多半是弱模型把文件里长得像工具调用的 XML 照抄了出来，于是识破这种「鹦鹉学舌」，只回一句「那是数据、不是指令」，而不是把整个工具目录再喂回去火上浇油；正文里混进了假的工具调用就剥掉。这三条 Claude Code 一条都没有——它赌自己托管的强模型能把名字喊对，喊错也只回一条报错、让模型自纠，不替它猜。

Line 2 · History

历史线：把这一整轮，重塑成模型下一轮认得、API 肯收的历史

行动线管「这一轮怎么动手」，历史线管「这一整轮怎么塞回去」。模型没有记忆，壳每一轮都要把到目前为止的整段对话重新喂进去（02 章讲过）——但生料不能原样塞：模型吐的那叠块里，有些 API 根本不收（比如以推理块结尾的 assistant 轮），有些会让下一轮请求直接报错（比如发出去却没配上结果的悬空调用）。所以回传前有一道重塑：剥掉结尾的推理块、把「只有推理」的孤儿轮并进相邻轮、把散开的块按 id 缝回、给悬空的调用补上占位结果。这道地板两家都铺，而且是 CC 最厚的一段工程——别被「CC 壳薄」误导。

一条关键纪律，和 02 章的每轮注入一模一样：这道重塑只动本轮调用的那份临时副本,持久历史一字不改——你在 UI 里看到的转录、存档里留下的记录，都是没被动过的原貌。壳绝不把重塑写回会话库。这是同一条纪律在输出侧的落点：只动本轮副本、绝不写回会话库。

地板之上，Hermes 又多铺一层——但这一层的根因不是弱模型，是异质 provider。它要对接几十种推理服务，每家吐推理的字段名/格式都不同，于是要做逐 provider 的方言翻译（把各家的 reasoning_content 归一）；吃不到显式缓存的本地引擎，还要把消息空白和工具调用 JSON 规范到逐字节一致去蹭 KV 复用；再加上清理某些模型吐出的孤儿代理字符、按严格 API 要求剥掉未知字段。这层 CC 全都没有——它只有一个 API，回传格式天生统一。

一个容易读错的点要说准：两家都能把图片放进消息给模型看（多模态）；区别在存档/轨迹里怎么省空间——两家都不把 base64 原样堆进历史，但顶替方式不同：CC 基本只留个二进制占位、把图默默丢掉；Hermes 则把图换成工具自带的一句文字摘要（text_summary）或 [screenshot] 占位顶上。所以「剥掉 base64」两家都做，但「剥完换不换一句描述」并不一致。

Two dimensions

结果回捕与错误处置：都是「改壳」，分野只在一处阈值

前面两条线讲的是「块怎么变行动、怎么变历史」。收尾看两个更具体的切面：工具跑完了，结果怎么翻回模型眼前（结果回捕)；以及出错了，错误变成什么样的块回到模型眼前（错误处置）。两者都属输出行动侧，都是纯改壳。

Claude Code · 结果回捕

每个工具自带「结果翻译器」，转成标准结果块、用 id 配对送回。文本带行号、图片转图像块、PDF 转文档块（多模态）、出错包成 <tool_use_error>。结果太大多数不硬截而落盘、只回预览+路径（Read 类豁免防死循环；Bash 超 64MB 仍硬截）。防撑爆是双层：单结果落盘 + 每轮聚合预算（约 20 万字符）。

Hermes Agent · 结果回捕

运行时结果是 OpenAI 风格的工具结果消息 + id 配对（<tool_response> 是训练/存档表示）。同样双层：单结果超阈值落盘指路 + 每轮聚合预算（约 20 万字符）。真正独有的一层是按模型上下文窗口缩放阈值：小上下文模型缩到约 16k 地板、大模型 clamp 回默认。图片进消息给模型看；存档/轨迹里换成工具自带的 text_summary 或 [screenshot] 占位顶替（不堆 base64）。

为何不同（维度修正 · 改壳）：两家都用「落盘+指路」代替硬截断，也都有「单结果落盘 + 每轮聚合预算」双层——别把「整轮总预算」当成 Hermes 独有（那是早先的误读，已证伪）。Hermes 真正独有的只有一处：阈值按模型上下文窗口缩放——小上下文的开源模型缩得更狠。根因还是那条：CC 只伺候一个大窗口的托管模型、可以用固定阈值；Hermes 面对一群窗口大小不一的开源模型，只能让阈值随模型走。

Tool exposure

工具太多怎么办：两家几乎同一手，都是纯改壳

牌太多会撑爆上下文。两家的解法几乎一样，而且都是纯改壳：工具目录常驻，详情按需拉取——初始只露名字，模型要用哪个，再用一把「搜索 / 取说明书」的钥匙把完整参数表拉回来。Claude Code——超阈值（auto 模式 10%）就把 MCP 及可延迟工具撤成按需；Hermes——可延迟工具超默认 10% 阈值就撤柜，只留 tool_search / tool_describe / tool_call 三把钥匙，且桥工具的 tool_call 原路穿过同一分发入口，权限门一视同仁。

  • [HM] 训练格式 vs 运行时格式会不会漂移：模型受训「见」的是 XML 轨迹，线上运行时读的却是结构化字段——两种表示各自演进，如何保证「训练时学的」和「线上喂的」长期不脱节？
  • [两家] 停止判定押在「块的存在性」上：两家都不信 API 的 stop_reason / finish_reason，只数这轮有没有 tool_use / tool_calls。把最基础的控制流也押在一个结构化字段的存在性上，弱模型场景下这套长期稳吗？

一个 Loop 是怎么设计的

三侧看完，两家的 Loop 其实只由一个变量决定——你有多信任模型。

信任决定壳的厚度 单杠杆 vs 双杠杆 改模型 vs 改壳 模型作者反而更不信任
控制侧、输入侧、输出侧看似三件事，其实是同一个选择的三次显形——这个 harness 有多信任模型，就决定了壳做多薄。Claude Code 押注"一个可控、持续变强的托管模型"，于是三侧都选择信任模型、把壳做薄；Hermes 面对"自训 + 第三方、良莠不齐的开源模型群"，于是三侧都选择不全信、用壳（有时连同训练）加厚兜底。

01两句话，两种哲学

Claude Code

信任模型，壳做薄。能交给模型判断的就交给它（何时停、怎么总结、怎么表达意图），壳只做管道——执行、翻译、守资源。好处：能力天花板随模型升级自动抬高。代价：模型犯浑（跑偏、做错）时，薄壳基本不纠正——护栏只挡「烧钱 / 闯祸」，不挡「做得对不对」。

Hermes Agent

谁都不全信，模型和壳一起加固。既然自己能训模型，就把一部分可靠性训进权重；又因为要跑参差不齐的开源/第三方模型，还得在壳里加一层防线。两根杠杆一起用。代价：壳复杂、要持续维护。

02同一个哲学，在三侧的三次显形

控制、输入、输出行动——三个不同的技术问题，暴露出 Hermes 的两根杠杆：控制侧、输出侧是信任驱动的"改壳"（越不敢信模型，壳越厚）；输入侧是所有权驱动的"改模型"（拥有模型，才能把能力训进权重）。Claude Code 只有前一根、还基本不用。

2.1控制侧 · 何时算做完

Claude Code

默认模型说了算——这轮没点工具＝完成。壳默认不判断"活干完没"（停止钩子、token 预算续跑是两个 opt-in 覆盖口，默认不用）。

Hermes Agent

壳设一道 verify-before-done 门：模型改了代码没测试证据就被推回循环，无权单方面宣布做完。（此门 surface-aware：编码/程序化入口默认开、聊天平台默认关，且只对代码改动触发。）

判词——信任问题，动的是壳杠杆：CC 默认信任模型的自述→壳默认不设门（但留了停止钩子/预算续跑两个 opt-in 口）；Hermes 默认不信任任何模型（含自家）的"我做完了"→在编码入口加验证门（聊天入口默认关）。（改壳）

2.2输入侧 · 上下文满了谁来压

Claude Code

把压缩的智能外包给通用模型（壳只给一段提示词让它自己写摘要）。

Hermes Agent

把压缩形态既训进模型、又写进壳（同源管线造训练数据，模型见过被压的历史）。

判词——这一侧不是"信不信"，是"谁拥有模型"，动的是模型杠杆：CC 不拥有模型，只能给通用模型写提示、临场发挥（改壳）；Hermes 拥有模型，用同源管线造"压缩过"的训练数据、让模型见过被摘要的历史（意图把这形态训进权重——"已训进线上权重"非本仓库可证，但这条改模型的路只有模型作者走得通）。这正是 Hermes 多出的第二根杠杆。

2.3输出行动侧 · 怎么解析意图、结果怎么回

Claude Code

读模型原生结构化格式，壳几乎不设防。

Hermes Agent

运行时同样读结构化，但壳额外加一层可靠性纠偏（工具名修复 / 空名防"鹦鹉学舌" / 剥掉正文里的假工具调用），以兜住弱模型。

判词——信任问题，动的是壳杠杆：CC 信任那一个托管模型的格式→壳不设防；Hermes 不敢假设第三方弱模型可靠→壳加纠偏层。（改壳）

03反直觉：做模型的人，壳反而做得更厚

直觉会以为：Nous 自己训模型，应该最信任它、壳做得最薄。实际恰恰相反——它的壳更厚。关键是看清它不信任的到底是谁：

  • 对任何模型（含自家）的"我做完了"自述——不信，所以有 verify 门（这一层对自家模型也设防；不过它 surface-aware——编码入口默认开、聊天平台默认关）。
  • 对第三方开源弱模型的格式可靠性——不敢假设，所以有纠偏层（这一层主要不为自家模型）。

根源是处境不同：Anthropic 只发一个自己完全掌控、持续变强的模型，Claude Code 可以放心押它；Nous 活在开源世界——模型不止一个、质量参差、还要自托管——它无法假设可靠，于是两头下注：能训进权重的就训进去（改模型），兜不住的用壳挡（改壳）。

"改模型 vs 改壳"的答案：Claude Code＝单杠杆（信任那一个托管模型）；Hermes＝双杠杆（拥有模型→能改模型；面对异质生态→又必须改壳）。

04对产品经理意味着什么

  • 若你基于一个强托管模型做 agent：学 Claude Code——壳做薄、把判断权交给模型，天花板随模型涨；但要接受"模型犯浑时壳兜不住"。
  • 若你要跑多种/开源/自托管模型：学 Hermes——壳必须厚；如果你能训模型，尽量把可靠性训进去；代价是壳复杂、要持续维护。
  • 一句话记住：壳的厚度不是能力问题，是信任问题。你越不能假设模型可靠，壳就越得厚。

同样要让 agent「跨越单次对话而存在」，两家被相反的部署现实推着走。分水岭不是「谁功能多」——两家都有定时、后台、远程触发、推送——而是一个隐藏变量：谁替 agent 保活。Claude Code 是厂商托管产品，答案是「厂商云替它保活」，于是一切围着人的注意力缝隙编排；Hermes 是自托管开源 agent，答案是「没人替它保活」，于是它被逼着自己变成那个常驻活体。

01问题：两次触发之间，谁让它活着

用产品语言讲：模型是个「被叫一次算一次」的函数，两次调用之间它根本不存在。谁来填补空窗？

一个具体场景：凌晨 3 点，你的代码仓库来了个新 PR 需要 review；或你想让 agent 每天早上自动跑一遍冒烟测试。可你本人在睡觉、在上班，不在终端前。这时 agent 要能存在、被叫醒、干活、再把结果送到你眼前。

但模型本身是个无状态、被叫一次算一次的函数——两次调用之间它根本不存在。谁来填补「两次触发之间」的空窗？谁在你不在时让它活着、按时醒来、干完还能找到你？这就是「时间与触发」的题眼。两家给出的答案，暴露了它们对「agent 到底活在谁的地盘上」的根本不同假设。

铸好的壳，谁被允许改写哪一层？（这是双向关系的上半；下半「它如何适应你」见下篇）

谁拥有代码 固定插槽 vs 可换 provider 市场 vs 策展目录 厂商扛信任 vs 你自己扛 企业只读总闸
两家都把 harness 做成了高度可编程的平台——但开放的是两种不同的东西，而且「信任由谁来扛」正相反。分水岭还是那个隐藏变量的延伸：谁拥有代码。Claude Code 碰不到核心，只能在封死的引擎上凿一批固定插槽，并被迫自建一整套信任与治理；Hermes 把源码给你，几乎一切做成可替换 provider，而它不必自建那种「企业市场式」治理墙——「你拥有它」已替它回答了大半个信任问题（剩下用 opt-in、显式授权、安全扫描等轻边界兜底）。但这只是人机关系的上半——你居高临下地改写它；下半反过来：它趁你不在，把你记进文件、越用越懂你（见下篇）。

01问题：让它做设计者没预料过的事

用产品语言讲：可编程性不是「功能多少」，而是「你想让它长成它出厂时没有的样子，路走不走得通」。

你想让 agent 做一些它的设计者从没预料过的事：强制「不过测试不许合并」、把它接到你公司内网的 API、灌进你团队的编码规范、换一个别家的模型来跑。怎么做到？

答案取决于一件事：你能不能碰源代码。这一个事实，就把两家的设计劈成了两半——一边你只能用官方替你凿好的开口，另一边你可以把手伸进任何一层。

01第二次找它，你还要从头解释你是谁吗？

一个 agent 会不会「长记性」，决定了它是每次归零的工具，还是越用越懂你的伙伴。

你昨天花十分钟跟它交代过：你是后端出身、第一次碰前端；你讨厌它擅自 mock 数据库；这个项目的怪规矩是「所有改动合成一个 PR、别拆」。今天再开一个新对话——它还记得吗？

这一篇拆的就是这件事：它跨会话保留什么、趁你不在怎么整理这些笔记、以及——它会不会把一次临时的坑，硬记成一条永远拒绝干活的规矩。

这是本章最非显然、也最值得单拎出来的一点：一个会自己攒记忆的 agent，不只越用越懂你，也会越用越固执——把一次临时故障硬记成永久规则，几个月后还拿它拒绝干活。两家都撞上了，但姿态相反。

Claude Code · 预防式：把「不信任记忆」焊进出厂设计

两家在「环境」这一侧惊人地趋同：默认都不把 agent 关进沙箱，动作直接落在你的真机器上；都把「隔离」和「审批」当成互相替代的两条路——关进笼子，就不再逐条问你；都设了一道连「全速前进」都压不过的绝对地板；被拒时都塞给模型一段几乎同款的「别耍花招绕过」训诫。真正的分野不在机制清单，而在这套治理默认假设谁在场、谁为闯的祸兜底：CC 厂商托管、跑在你自己装满真实资产的机器上，把治理做成一个你亲手转的信任旋钮，原生假设终端前坐着能随时点确认的人；Hermes 开源自托管、常以一个被消息叫醒的常驻实体跑着、常常无人值守，必须为「没人点确认」定义确定行为，并因为没有厂商兜底、闯祸是你自担，索性白纸黑字告诉你「进程内没有任何东西算隔离，唯一的边界是操作系统」。一句话：CC 为「有人盯着」设计，Hermes 为「没人盯着」设计。

01问题：它动的是你的真机器，闯的祸是真祸

这一侧不谈玩具环境。agent 编辑的是你磁盘上真实的文件、跑的是你机器上真实的命令行——没有中间的模拟层。

agent 干着干着，要 删一整个目录、要 强推覆盖远端分支、要 读你家目录里的密钥文件、要 连一个陌生外网地址。这些动作不是在沙盒里排练，是真的落在你的电脑上——删了就是真删了，推了就是真推了。

于是每个 agent 都要回答同一组问题：这些动作落在哪个范围能碰、哪个范围碰不得？危险的那些，谁来拦、怎么拦？真闯了祸，谁来兜底？而最关键的一层追问是——拦的时候，假设有没有人在旁边点头？

03深潜一 · 连「全速前进」都压不过的地板，各自护着什么

两家都给了一个「我盯着，你放手干」的档位（CC 的 Bypass Permissions、Hermes 的 yolo）。但两家也都留了一道连这个档位都压不过的绝对地板——有意思的是，两道地板圈住的不是同一样东西。

Claude Code · 地板圈住「管着 agent 自己的那一层」

哪怕你切到 Bypass（全放行、红色警示），有几类动作仍被强制拦下要你确认：改 .git/、改 .claude/ 自己的配置、改 shell 启动脚本（.bashrc 等）、改 IDE 配置。逻辑是：放手可以，但改动「能自我提权、能改写游戏规则本身」的东西，永远不放手。全放行不等于无法无天。

真实清单 · 连全放行都拦的路径.git/ · .claude/（自身配置） · ~/.bashrc / shell 启动脚本 · .vscode/ / .idea/（IDE 配置） · 以及任何用户显式标了「禁止 / 要问」的规则

Hermes · 地板圈住「别把这台机器毁了」

有一小撮命令被定为无论如何都不许跑：删根 / 删系统目录 / 删家目录、格式化磁盘、往裸块设备写、fork 炸弹、杀掉所有进程、关机 / 重启。这道地板在 yolo、关审批、cron 放行之下——原话逻辑是「开 yolo 是你信任它动你的文件和服务，不是信任它去把磁盘擦了、把机器关了」。（它另有一组「防自尽」规则不许杀掉自己的常驻网关，但那属可绕的软清单、不是这道无条件的硬地板。）

真实原文 · 硬禁地板被拦时给 agent 的话BLOCKED (hardline): {description}. This command is on the unconditional blocklist and cannot be executed via the agent — not even with --yolo, /yolo, approvals.mode=off, or cron approve mode. If you genuinely need to run it, run it yourself in a terminal outside the agent.
同一个动作、两种恐惧：两家真正的差别不在「护不护」，而在把哪一样设成连全速前进都不可绕过的绝对地板。CC 的 bypass-免疫地板护的是「管着 agent 自己的那层配置」（.git / .claude / shell 启动 / IDE）；灾难命令它也查，但多是「问一下」这种能被放手档盖过的软拦（个别如 PowerShell 删除才是硬拒），而非无条件地板。Hermes 的 yolo-免疫地板正相反，护的是「别毁了这台机器」（删根 / 格盘 / 关机等）；改自己的安全配置它也拦，但没放进这道硬禁地板。各自把对方留在软层的那一样，单独升格成了硬地板——这，而非「谁护谁不护」，才是分歧点。两道硬地板刻意都极小：只收「没有恢复路径」的动作；像 rm -rf /tmp/x、curl|sh 这种「代价大但可恢复」的都留在软清单、yolo 一开就放行，别把软清单也当成不可绕过。

04深潜二 · 被拒之后：两家几乎同款的「别耍花招」训诫

这是我们独有的一张王牌——把两家在「动作被拒之后对模型说的原话」并排。一个托管闭源、一个开源自托管，措辞却近乎逐字重合：都不许模型用歪招绕过这次拒绝。

Claude Code · 给模型的绕行准则原文

src · 拒绝后随附的行为准则IMPORTANT: You *may* attempt to accomplish this action using other tools ... e.g. using head instead of cat. But you *should not* attempt to work around this denial in malicious ways ... If you believe this capability is essential ... STOP and explain to the user ... Let the user decide how to proceed.

Hermes · 命令被拒 / 超时给模型的原文

src · 被拒时的话（超时再追加一句）The user has NOT consented to this action. Do NOT retry this command, do NOT rephrase it, and do NOT attempt the same outcome via a different command. Stop the current workflow and wait for the user to respond ... （超时追加：）Silence is not consent.
为什么两家不谋而合：危险治理最脆弱的地方从来不是「拦不拦得住第一次」，而是「拦住之后，模型会不会换个命令、换个说法、假借跑测试之名把同一件事再做一遍」。两家都发现：光有闸门不够，还得在闸门后面放一句给模型的训诫，堵死绕行的动机。把「拒绝」从一堵冷墙，变成一次有引导的协商——「你可以正当地换条路（用 head 代替 cat），但不许耍花招绕过我的本意；真觉得非做不可，就停下来问人」。Hermes 那句「沉默不等于同意」尤其点题：它常常无人值守，没人回答不能被模型解读成默许。

Hermes 的常态恰恰是没人在场。它常以一个被时钟 / webhook / 消息叫醒的常驻实体跑着，跑完把结果推给不在场的你（见第 04 篇）。于是它必须为「没人点确认」定义确定行为——cron 默认拒、沉默不等于同意、硬禁地板永远在；能触达你时（推到 Telegram / Slack 等你 /approve）才把决定权交回人。

更深一层：两家其实都在对你说「别全信我」——但指向了不同的出口。CC 说「只对你信任的文件用我」（把残余风险推给你选对输入，因为它多半在你自己可信的仓库里、你就在旁边）；Hermes 说「进程内没有任何东西算隔离，唯一的边界是操作系统」（把残余风险推给你在 OS 层做隔离，因为它是个从开放渠道吃不可信输入、又常常无人值守的常驻体）。是「厂商托管、有人在场」还是「自托管、无人值守」，决定了它把最后那道防线画在哪、也决定了它对自己能力边界诚实到什么程度——诚实到什么份上？它明明还搭了一层进程内的护栏（把 API 密钥、网关令牌在传给子进程前剥掉，堵住恶意扩展把凭据隧道出去），却仍在文档里主动写明「这只减少随手泄露，不算隔离」。既动手缓解、又亲口否认那是边界——这正是它整套「别把软门当真墙」姿态的注脚。

一个共识别漏掉：真到了「没人在家」且被正确识别的时候，两家的安全默认都是「不做」——CC 无法弹窗即默认拒，Hermes cron 默认拒 / 沉默不等于同意。分歧不在这条底线，而在谁把「无人」当常态来设计。这，就是喂给综合②「在场契约」脊椎的那一块。

06悬而未决 · 边界说明（防「默认→绝对」以偏概全）

  • [CC · 默认] 沙箱默认是关的：外部用户开箱即用并没有命令沙箱，安全全靠权限弹窗。而且开沙箱后命令反而不再逐条问——它是「换一种范式」（靠笼子物理隔离），不是「多加一层防护」。别记成「CC 在沙箱里跑命令」。
  • [HM · 默认] 后端默认 local ＝ 裸主机；容器 / 远程 / 云沙箱、整进程 Docker / OpenShell 包裹，全是 opt-in。别记成「Hermes 把 agent 关在沙箱里」——恰恰相反，默认不关。真隔离是官方推荐、但要运维自己去开的选项。
  • [两家 · 子代理不自动隔离] git worktree 隔离两家都是手动 opt-in：CC 明确「只有用户说 worktree 才用，切分支 / 改 bug 都不算」，子代理默认共享同一工作目录（可显式给某个子代理开 worktree 隔离，但不是默认）；Hermes 的 worktree 也要显式开。「并行分身各自隔离 blast radius」不是默认形态。
  • [CC · 窄例外] 「出了工作区圈一律先问、读写对称授权」有一组内部管线的窄例外：harness 自己的暂存 / 计划 / 会话记忆目录，即便在工作区之外也静默可写（不弹窗）。但这组例外只限 harness 自身的后台目录、不含任意用户文件，「CC 信任工作区、出圈就问」的大结论仍成立。
  • [HM · 诚实漏洞] 既非交互、又非网关、又非 cron 的无人会话，危险命令会被直接放行（当前主路径甚至不先做危险检测、也不拦不问）——官方标注的已知限制。所以「无人即更保守」只在被识别的 cron / 网关场景成立，不是所有无人场景都兜底。这一条不能被「无人时更安全」的干净故事压掉。
  • [HM · 别当默认支柱] 内容扫描器 Tirith 默认「挂了就放行」（fail-open）、smart 模式（小模型审查）与 off 都是 opt-in、记忆 / 技能写入审批默认关。硬禁地板是全文里唯一真正无条件的机制，其余大多是可配的默认值。
  • [CC · 分类器 ant-only] 用 AI 分类器代替人审的 auto 模式是内部 / 特性门控，外部用户默认没有；写外部产品逻辑时不当它是通用能力。
  • [不评优劣] 本篇讲的是「危险治理的形态与在场假设」，不评判两种模式孰优——取决于你是否在场、自托管还是托管、吃的是可信还是开放输入。
  • [时间切片] 两家治理都在快速演进，本篇是某一时间切片（CC main / Hermes @daf4f1a7a）。

Loop 与其他 Agent

分身之间，靠喊话还是靠看板？

CC 三种分身：subagent / fork / 团队 全新隔离子代理趋同 · CC 另有 fork（Hermes 无） 团队层：消息同事团队 vs 共享状态看板流水线 两家都能盯单个分身 托管产品的 live-team 更完整
先分清一件常被混为一谈的事：光是 Claude Code 自己，就有三种机制不同的分身——① 全新子代理（subagent）：leader 写一段自包含简报、派一个零共享上下文的隔离分身，它跑完回一条结论；② fork：不给角色时触发，分身继承 leader 的完整上下文与系统提示、共享提示缓存（所以给它的是"做什么"的指令、而非重述背景）——这是"接着我的记忆去干"，和①正相反；③ 团队（teammates）：一队具名队友围着共享看板、用邮箱互相发消息、人当队长双向协作。三者作用与机制都不同。
把两家摆开：在「全新隔离子代理」这一支上两家趋同——CC 的 subagent ≈ Hermes 的 delegation（都是 fresh 会话、隔离、只回结论；leader 可用消息续跑它，但分身之间不通信）；但 CC 还多一条 Hermes 没有的 fork（继承上下文、复用缓存），Hermes 的委派被明确设计为"只有 fresh 会话、无父历史"。真正戏剧化的分野在「多角色团队」的协调媒介：Claude Code 让代理互相发消息（邮箱 + 共享看板 + 具名队友，人以队长身份在同一张消息网里），这套 live-team 产品面做得更完整、更产品化；Hermes 则让独立实例靠共享数据库状态协调（看板/蜂群：依赖图、黑板评论、原子认领，彼此不发消息），且要显式进入才用。一句话：「消息驱动的同事团队」 vs 「共享状态驱动的看板流水线」。

01问题：拆开之后，这些分身怎么协调

用产品语言讲：多智能体不只是「怎么把活拆开并行」，更是「拆开之后，它们靠什么协调、汇聚、不互相踩脚」。

你把一个大活交给 agent——跨 12 个文件重构，或者一次调研 5 种方案。一个代理的上下文和时间扛不下，于是它拆成好几个分身。两个问题立刻冒出来：(1) 这些分身怎么不互相踩脚？(2) 它们靠什么互相协调——是像一支团队那样互相发消息，还是像流水线工人那样各自读写一块共享白板？

这第二个问题，就是两家多智能体设计真正分岔的地方——它决定了整套东西长成「一支你带的同事团队」还是「一条你摆好的看板流水线」。

Hermes 的默认做法是——模型自己把活 fan-out 成几个分身并行跑、只把结论收回来；这些委派分身之间刻意不互相对话（不是一支会喊话的团队）。真要多角色协作，就显式进入看板 / 蜂群这条重路径：一张任务图，每个角色是一个完全独立的 Hermes 实例，它们不发消息给彼此，而是各自读写一块共享数据库白板（依赖图、黑板评论、原子认领）来协调——像流水线工人从看板上领活、交活。这里也不是黑盒：你能通过 TUI 的 /agents 面板旁观、暂停、打断单个分身。差别不在「能不能监督」，而在协调的隐喻——CC 是互相喊话的同事，Hermes 是共享白板上的工人；而且 CC 把「同事团队」这层做成了更完整、更产品化的一等体验，Hermes 的看板流水线则要你主动走进去才用。

05悬而未决 · 边界说明

  • [CC 有三种分身，别混] Claude Code 的多智能体不止「团队」一种，且委派本身分两条：subagent（自包含简报、零共享上下文的全新分身）与 fork（继承 leader 全上下文、复用缓存——和 subagent 正相反）；再加上双向协作的团队。本篇把「消息驱动的同事团队」这个特征限定在团队层；「零共享上下文」只对 subagent 成立、不适用于 fork。（早期稿曾把 fork 误并进 subagent，已更正。）
  • [委派非纯单向] 即便普通 subagent，leader 也能用消息把它续跑（分身带完整上下文恢复、再回一条结果），coordinator 模式更把"续跑已完成/失败的 worker"当核心工作流。准确的说法是「分身之间不互相发消息」，而非"分身完全单向一次性"。
  • [两家都能监督] 别读成「CC 能盯、Hermes 是黑盒」——两家都能旁观、暂停、打断单个分身（CC 靠分屏面板，Hermes 靠 TUI 的 /agents 面板）。分野在协调隐喻，不在能不能监督。
  • [不是「默认关闭」] Hermes 的看板/蜂群不是被开关关掉的功能，其派发默认是开的；准确说法是「要显式走看板/蜂群这条路径才进入」，不是普通聊天的默认形态。
  • [CC 计划审批自动通过] 桥接给人的是敏感工具/权限确认与叫停；「计划模式审批」在 leader 侧是自动通过的，并非每次必经人工。别把它读成「什么都要人批」。
  • [peer 未闭环] CC 的「跨会话/跨机器 peer 通信」在当前代码树里提示词与地址已在、但若干实现文件缺失，未能闭环证明可用——本篇按存疑处理，不作为既成能力。
  • [隔离没那么强] Hermes 委派分身默认共享同一容器工作目录（除非注册隔离 override），不是每个都独立沙箱；真正的强隔离在看板蜂群（每角色独立实例）。
  • [归因是解读] 把差异归到「第 04 篇的在场重心」是一个合理解读，非代码可证的铁律——代码只能证明两套协调机制的形态差异。
  • [呼应轴一] 「委派分身结论自报、父代理须自行核验」呼应轴一——Hermes 对任何模型（含自家）的「我做完了」都不全信。
  • [计划清单不在此篇] 单个 agent 给自己列的「计划 / 待办清单」不是多智能体——它是一个 loop 内部的过程纪律，归轴一·控制侧（第 01 篇 §04 第④维）：那里对比了 CC「任务清单＋壳主动催」与 Hermes「待办工具刻意不挂钩子」。

Loop 与自己

越用越强，是真的还是宣传？

self-improving 是 README 第一句 改权重＝谁拥有模型说了算 CC 结构上没有训练代码 Hermes 有管线但不接你对话 两家都证不到权重级变强
「越用越强」这句宣传，落到代码里要先分成两半：一半是改壳（把经验写成记忆文件，越用越懂你——这半是真的，另有『它如何适应你』一章专讲）；另一半是改权重——把你的使用变成模型本身更聪明。本章只钉后一半。而后一半能不能成立，答案又一次落在轴一那根「谁拥有模型」杠杆上：CC 不拥有模型，结构上根本没有「改权重」这条路，仓库里连一行训练代码都没有，只能把它外包给母公司的下一代基座；Hermes 拥有模型，仓库里真有一条产训练数据的管线——但那条管线吃的是研究员策划好的数据集，根本不接你的真实对话。所以一处对称的诚实必须先摆上桌：两家仓库都证不到「你的使用让它权重级变强」——CC 断在「没那个能力」，Hermes 断在「有能力却没把管子接到你身上」。「self-improving / 越用越强」这个飞轮，两家都只证到了前半段，后半段都得标『意图，仓库外不可证』。

01问题：跑过一千次任务，它自己变聪明了吗？

用产品语言讲：「越用越强」有两种截然不同的含义——一种是它记住了你（改壳），一种是它这个模型本身变强了（改权重）。宣传把两者混着讲，本章只钉后一种。

它替你跑了一千次任务、被你纠正过几百回。它这个模型因此变聪明了吗？还是每一次调用的，都是同一个出厂时就定型、你怎么用都不会长进的它？

这不是抬杠。Hermes 的 README 第一句就把自己定义成「self-improving AI agent」「唯一自带学习闭环」，还专列一行讲「产轨迹、压缩，用来训练下一代工具调用模型」。这些词听起来像是「你越用，它这个脑子越聪明」。所以问题很具体：这条从『你的使用』通到『更聪明的权重』的飞轮，落到两个仓库的代码里，到底哪一段是真接通的、哪一段只是意图？

02核心机制：能不能改权重，是「谁拥有模型」说了算

「改权重」不是一个功能开关，而是一个结构性前提：你得先拥有那个模型，才谈得上把使用喂回去重训它。这一条，把两家从起点就劈成两极。

Claude Code · 不拥有模型 → 这条路结构上就不存在

CC 是托管产品，调用的是母公司的模型，不拥有权重。所以它仓库里没有、也不可能有任何训练代码。它能做的极限，是在你逐次同意下，把对话转录上传给母公司当训练素材（仅抹掉密钥、绑你的账户，并非匿名）——这是单向投喂，喂进去之后能不能变成更强的下一代基座、什么时候变，全在仓库之外、CC 自己看不到也证不到。改权重这件事，根本不在它手里。

Hermes · 拥有模型 → 握着杠杆，但没把管子接到你

Hermes 出自自训模型的 Nous，拥有权重，仓库里真备了一条产训练数据的管线：批量跑任务、生成轨迹、压缩保训练信号。这根「把使用变权重」的杠杆，它确实握在手里。但关键在于管线的进料口：那条批量管线吃的是一份研究员策划好的任务数据集，跟你平时聊天攒下的真实对话库之间，代码里没有任何自动接线。它有能力闭合飞轮，却没把入口对准你。

同一根杠杆的两极：「改权重」这条路存不存在，先看你拥不拥有模型。CC 不拥有 → 路本身不存在，只能把权重级变强外包给母公司；Hermes 拥有 → 路存在、管线也真在，但入口没接你的会话。所以两家都到不了「你的使用让它权重级变强」这个终点——一个没有能力，一个有能力没接线。

03深潜 · 把飞轮从中间切开：宣传怎么说，代码怎么接

宣传怎么说：Hermes README 把「学习闭环 + 训下一代」写进门面

README 用词很重：self-improving、closed learning loop、训练下一代模型。读者很容易把它理解成「我越用，这个模型越聪明」。

五种关系 · 一个总变量

它假设，有没有人正盯着？

五种关系看着是五件事——何时醒来、听谁改写、在哪动手、和谁协作、怎么变强。但把它们并在一起，每一次分野都由同一个问题决定：这个 agent，假设有没有人正盯着它？ Claude Code 处处假设「有人」，Hermes 处处假设「没人」。而这个假设不是品味选择——是所有权处境替它选好的。

The scene

一个任务被触发了——也许是定时器，也许是一条消息推送。此刻五个问题同时冒出来：谁让它醒着？它要改的东西，谁批准？它闯了祸，谁兜底？它派出去的分身，谁协调？它这一趟学到的，算谁的？

For PMs · 怎么用

先回答这一个问题，后面五件事就定了

学 CC
如果你的 agent 活在「有人盯着」的世界（IDE 里、有人随时能拍板）：把拍板权留给人，无人时保守到默认拒。天花板随人的在场自然抬高。
学 Hermes
如果你的 agent 要在「没人盯着」的世界跑（后台、被消息叫醒、自托管）：每一个「本该有人点头」的地方，都得提前定义「没人点头时怎么办」；并诚实到白纸黑字承认边界在哪，别假装有隔离。
一句话
先问「我假设有没有人在场」——这一个答案会决定你后面五件事怎么设计。而它往往不是你能自由选的：你的所有权处境（托管还是自托管、闭源还是开源）已经替你选了。
Loose ends · 诚实边界

这根脊，不是五等份

  • 两条线，别硬拧成一条。对时间／环境／同类，「在场」是正面主轴；对人／对自己，真正在分的是所有权链，在场只是它的侧面（治理层有没有人盯／变强要不要你签字）。硬说成「在场的五次显形」就是以偏概全——所以本章把两条线分开画（脊上的空心点即标此）。
  • 「对同类」那一行是解读，不是铁证。「人当队长 vs 数据库协调」是对机制的合理解读，非代码里可直接证的设计意图（半点标注即此）。
  • 「变强」这一行，两家都别信自己。无论 CC 的「外包给下一代基座」还是 Hermes 的「握着训练杠杆」，画的都是意图与潜力、不是已经做到——两家仓库都证不到「你的使用让它权重级变强」（CC 断在没这能力，Hermes 断在产数据管线没接你的活会话）。脊上 row5 标的是「谁有可能」，别读成「谁已经在越用越强」。
  • 五关系的边界仍有一处未闭。「对同类」是否还应含外来 agent 互通（如 Hermes 的 ACP 适配器），07 章尚未覆盖、待查证；若那格材料成立，本综合「五关系」的完备性自评要回来再调——不因为故事顺就盖章。
