轴一 · Loop 之内控制侧

控制侧：核心 Agent Loop

拆开 agent 那条「永不停歇」的主循环——它为什么转得下去，两家又在哪里分家。

TL;DR

「连续工作」不是模型的能力，是壳造出来的幻觉——模型每一轮都被重新喂进整段历史、再被唤醒一次。壳用三层防线撑住循环：机械层（唤醒重发）、纪律层（写给模型的话、拦它偷懒）、强制层（用代码兜住出错与想停）。

机械层两家几乎同构；分歧全在后两层——Claude Code 默认赌模型自觉，Hermes 用壳去补模型的不足。

本章只讲循环为什么转下去；上下文怎么拼、压缩怎么做见 02 章，输出块与错误处置见 03 章。

00 · 问题

模型说完就死，「连续工作」到底从哪儿来

这一章不谈抽象循环，只追一件具体的怪事：模型明明「说完即死」，agent 却像不知疲倦。

一个大模型每次调用只活一轮：你给它一段文字，它吐回一段文字，这次调用就结束了——它没有后台进程，也没有留在原地等下一步的能力。说完，就死。可我们看到的 agent 却能连着干几十轮：读代码、改文件、跑测试、看到报错再回去改，中途几乎不用你插手。这份不知疲倦的「连续工作」，到底是从哪儿来的？

把这个「凭什么」拆开，其实是三个层层递进的问题。头一个：模型每轮说完就死，壳凭什么能把它再叫起来一次、让循环继续往下推？就算壳能一直叫醒它，第二个问题跟着就来了——模型凭什么不偷懒？它完全可以丢下一份计划、或一句「我这就去做」就想收工。最后，模型真要停下、或者眼看就要崩了（工具报错、吐出空响应、话说到一半被截断），壳又怎么兜住、不让这一整轮的活白白报废？

头一个问题的机关其实很朴素：模型这一轮想调什么工具，壳就替它把工具真的跑一遍，把结果拼回那段对话历史，然后带着整段加长的历史再发起一次全新的调用。模型所谓的「记忆」，就是每一轮被重新喂进去的那份历史——两家做法一模一样。这三个问题的答案，正好对应壳的三层防线：机械层负责唤醒、纪律层拦住偷懒、强制层兜住出错与想停。而两家从头一个问题起就开始分道——分歧最大的，是后两个。下面就逐层拆。

两个补充。其一，物理上「agent 在工作」，是壳在一段段地把一个生成器往前摇——正因如此，中断、压缩、恢复才能干净地插在两段调用之间。其二，会话越滚越长，迟早撑爆模型的上下文窗口；让长会话不爆窗口靠的是上下文压缩——那是维持循环能转下去的地基，细节留给 02 章。

Layer 1

第一层·机械续命：继续还是收工，只看一个开关

先看最底层的机械。两家的判断同构到近乎重合：模型这一轮只要调用了任何一个工具，壳就判定「话还没说完」，再发起一轮；一个工具都没调、只吐了一段纯文字，壳才认为它真的说完了，走收尾。工具调用的有无，就是「继续还是停」的唯一开关——Claude Code 的注释直接把它称作 the sole loop-exit signal。

Hermes 在每一圈重发之前还多设了四道闸：这一回合的调用数没到上限、跨回合的迭代预算还有余额、没有收到用户中断、预算耗尽时是否还留着一次收尾调用——任何一道不满足，循环当场跳出。但主干上的那个开关，和 Claude Code 是同一个。

Layer 2 · Deep dive

第二层·提示词纪律：都看人下菜，只是看的不是同一个『人』

模型会偷懒。它会交出一份计划就想收工，会说一句「我这就去做」然后停手——这些是壳设计者反复见到的失败模式。两家的对策都是往系统提示里写「别罢工」守则，但关键在于：没有一家对所有场合用同一份守则。

Claude Code 的默认守则其实相当温和——失败别急着放弃、危险动作先确认（更硬的「如实汇报别谎报完成」只进了内部构建）。真正的高压话术全部锁在条件层：人不在场时才有「偏向行动别问」「终端在后台就大胆自主」；用户亲手开了功能才有「token 目标是硬下限」。Hermes 的常驻守则是一句朴素的「把活干完、别交桩」；更狠的「必须立刻调工具、别用一句承诺结束回合」只喂给一份弱模型名单（gpt、codex、gemini、gemma、grok、glm、qwen、deepseek 八家），而 Claude 这类强模型反倒不注入。

四格摆在一起，差别不在「加不加压」——两家都按场景和功能分层加压（CC 有自主模式，Hermes 有编码岗守则和无人值守协议）——而在 Hermes 比 CC 多出一根加压轴。Claude Code 只面对一个自家托管的模型，「模型行不行」根本不是变量，它的条件层主要跟着「人在不在看」走：终端前台还是后台、交互还是自主。Hermes 要跑一整群第三方模型，「模型行不行」就成了它独有的第一变量——纪律按模型名单发放，弱的多压、强的少压。

连催促待办的方式也照这条线分。Claude Code 的壳会在待办搁置太久时周期性塞一句软提醒（旧版待办工具还在每次回执后附一句「请继续推进」）；Hermes 刻意不让待办挂任何提示——引导只写在工具说明里，把主动催促的额度留给记忆蒸馏和技能自造（见轴二）。

诚实边界：Claude Code 生态里最锋利的那段话术——「回合结束前，检查你最后一段是不是一句没兑现的承诺」——并不在 CLI 代码库里。它是 Agent SDK／多智能体运行时另外叠加的一层，详见文末悬而未决。

Layer 3 · Deep dive

第三层·壳的强制：防死是共识，防懒是分歧

前两层管「该不该继续」，第三层管「拦不拦得住意外」。模型随时可能「死」：工具抛错、吐出空响应、话说到一半被截断、API 调用失败。两家的壳都不让它真的死——把错误包成一条工具结果回喂给模型，让它自己看到报错再改，然后重试、切备用模型、压缩后重跑。Hermes 兜得更低：连空输出、只思考不说话、工具名拼错、参数 JSON 损坏、流被网络掐断，都各有一条专门的恢复路径——弱模型的种种毛病，壳都得替它治。

每一类错误具体怎么处置，是 03 章的事。这里只需记住一件事：在两家眼里，错误都不是终点，是下一轮的输入。

分歧出现在「防懒」。模型想收工——不调工具、交出一段纯文字答案——Claude Code 默认放行，一道完成度检查都没有；Hermes 默认设一道验证门。这道门最能说明「可靠性焊进壳、而不是托付给权重」，值得逐帧拆开看：模型说「我修好了」，只是一个待核实的声明，不是结论。

The brakes

反面：每个续命机制都自带刹车

无限续命等于无限烧钱。真正的产品智慧不是「一直续」，而是分清两种失败：可恢复的失败（空输出、网络断、工具名拼错）值得猛续，确定性的失败（安全拒答、鉴权错误、压缩也救不回）就该干净地停——重试一个确定性失败，只会把它原样复现一遍。下面这张表把两家的每个刹车并在一起看。

The verdict

谁在防它罢工：一个赌模型自觉，一个用壳补模型

三层防线上的投入分布：机械层同构；纪律层都看人下菜，但看的变量不同——一个看场景（人在不在场），一个看场景加模型（还看名单）；强制层防死是共识、防懒是分歧（Claude Code 默认 0 门、Hermes 默认 1 门）。归因还是那条信任线：Claude Code 押注一个可控的托管模型——模型够强，纪律就能温和，防懒就能交给模型自觉；Hermes 面对的是一群良莠不齐的第三方模型——纪律得按名单发放，防懒只能焊进壳里。

一个反直觉的落点：做模型的那家，反而更不信任模型说的「我做完了」。越清楚权重的脾气，越不敢把「完成」的判断权交出去。

往下一章看：Claude Code 加压的那个变量——「人在不在场」——不是本章的终点。这根线在轴二「Loop 之外」会长成一整条脊椎：壳怎么随人的在场与否，动态调整它对模型的信任与放手。

Loose ends

悬而未决：九条别被主线吞掉的边界

轴一 · 输入侧Loop 之内

输入侧：每轮那个请求怎么拼出来

壳每一轮要现拼一个发给模型的请求包——这个包分三层，你以为每轮全新，其实八成逐字复用。

TL;DR

拼同一个请求包，两家的手艺八成是重合的——都为吃缓存把稳定内容冻在最前，都把每轮变化的东西塞进消息窗口、变了才补，都跑几乎一样的「先删后摘」压缩阶梯。

真正的分叉都挤在同几条缝上，而每条缝都在泄露它到底是什么：Claude Code 首先是个产品（动态层塞满 IDE、终端、任务的信号）；Hermes 首先是个面对一群自家开源模型的拥有者（连冻在最前的『静态』前缀都得按模型分叉，且只有它能把压缩训回权重）。

本章只拆「每轮那个请求包怎么拼、分几层」；循环为什么能一直转下去见 01 章；记忆怎么写、怎么蒸馏、跨会话怎么演化的完整生命周期见 08／05 章。

00 · 问题

你以为每轮是一份新输入，其实八成是逐字搬过来的

这一章不谈抽象的「上下文管理」，只追一件反直觉的事：模型每轮"看到"的那份输入，绝大部分根本不是这一轮写的。

直觉里，模型每轮"看到"的，应该是一份为这一轮现写的输入。实际恰恰相反——壳每一轮都在拼同一个请求包，而这个包的绝大部分，是上一轮逐字不变地搬过来的。于是那个"凭什么"就冒了出来：一个请求包由哪几部分组成，哪些逐字不变、哪些每轮才换，壳又凭什么这么分？更奇的是，两家壳的分法几乎一模一样——是什么共同的约束，逼得它们不约而同都这么拼？

头一个"凭什么"的答案朴素得意外：钱和速度。给模型发一次请求，最贵的成本是让它把长长的前缀重新"读"一遍；只要前缀一个字节都没变，服务端就能"认出这叠没动过"、直接命中缓存、又快又省（这就是 prompt cache）。于是两家做同一件事：把几乎不变的东西（身份守则、工具目录）死死钉在最前面当"冻结前缀"，把每轮要变的东西全赶到后面去——为的就是别惊动缓存。

所以"每轮那个请求怎么拼"，可以干净地拆成三层：最前面冻起来吃缓存的静态层，后面每轮现塞进去的动态层，以及当整个包快撑爆窗口时才启动的压缩应急阀。这一章就沿这三层各走一遍——你会看到两家在每一层都先高度趋同，再在同一处分叉。

The anatomy

一次请求长什么样：两家共享的三层装配骨架

先把一次请求摊平成一张图。最该看清的不是每一格写了什么，而是两家的装配骨架几乎是同一张——差异不在骨架，而在骨架的每一层内部（后面三节逐层拆）。自上而下就是发给模型的顺序：冻结的静态层在最前吃缓存，动态层每轮补在后面，压缩阀只在快撑爆时才动。

Layer 1 · 静态层

冻结吃缓存是共识，但 Hermes 连『静态』前缀都得看模型下菜

两家都把稳定内容冻在最前面吃缓存，逻辑一模一样。Claude Code 用一个字面分界符 __SYSTEM_PROMPT_DYNAMIC_BOUNDARY__ 把系统提示词切开，只有它之前的部分（身份→任务政策→执行守则→工具用法→语气）跨会话冻结进全局缓存；Hermes 干脆把系统提示词显式设计成三层——stable / context / volatile，拼成一整串、每会话只建一次、之后逐字重放，代码注释直说这是「唯一能让上游缓存保温的办法」。到这里，两家是同一套缓存经济学。

分叉在于——Claude Code 只面对一个自家托管的模型，这段"静态"前缀就是真·统一的。Hermes 要伺候一群异质开源模型，它的"静态"前缀根本不是一整块：命中 gemini/gemma 就注入一段 Google 模型执行守则，命中 gpt/codex/grok 换一段，遇到阿里的模型还要专门写一句「你其实是某某模型」去纠正 API 返回的错名字，弱模型名单上的还要再加一段工具强制纪律。两个不同模型，拿到的"静态层"实质是不同的。

Google 系（gemini / gemma）
注入一段专属执行守则——自主干到彻底解决、别给个计划就停
OpenAI · Grok 系（gpt / codex / grok）
换一段执行铁律——工具结果空就换姿势重试、别早停
阿里系（alibaba provider）
专门写「你其实是 X 模型」——纠正该 provider 返回的错误模型名
弱模型名单（tool-enforcement）
再加一段工具使用强制——必须同一条回复里立刻调工具

更彻底的一层：连"缓存"这件事，都得挑模型/provider。显式的 cache_control 断点主要给 Anthropic 系、外加少数几个经特定网关的兼容 provider（如 Qwen / MiniMax / 阿里）开着；更多本地/开源模型吃不到，只能退而求其次——靠把消息空白和工具调用的 JSON 规范成"逐字节一致"，去蹭底层推理引擎（llama.cpp / vLLM / Ollama）的 KV 复用。同一套省钱机制，不同模型享受的档位不同。

接回轴一：这正是 01 章那根「按模型加压」的独有轴，在装配层的第二次现身——CC 只面对一个模型，前缀能真·统一；Hermes 面对一群模型，连冻在最前面的料、连吃不吃得到缓存，都得看是哪个模型。所有权与异质性，从纪律层一路渗到了装配层。
Layer 2 · 动态层

每轮现塞什么：都塞进消息窗口、变了才补——但塞的东西暴露了产品 vs 底座

冻结前缀之外，才是每轮真正变的东西。两家的机制惊人地一致：都不把这些新内容写回那个冻结的前缀（否则缓存全废），而是塞进消息窗口——Claude Code 包成一个个 <system-reminder> 的"合成 user 回合"（模型看得见、你在对话里看不见），Hermes 直接追加到当轮 user 消息的尾部。而且两家都只在内容变了才补：CC 用一本 readFileState 台账记住每个文件上次的样子、只发变化的，还给这些信号限流（待办/任务每 10 轮、计划/自主模式每 5 轮才补一次）；Hermes 在回合开场一次性预取、整轮复用、不重复查。

分叉在于——塞进去的都是些什么。Claude Code 每轮可注入的信号有四十多种，绝大多数是产品面信号：被外部或 linter 改动过的文件、IDE 报来的 lint/类型诊断、你在编辑器里选中的行、打开的文件、待办/计划/自主模式状态、钩子输出、伴生宠物、队友信箱……Hermes 每轮的主注入只有八种左右，都是底座信号：记忆召回、插件 hook、/steer 打断、几个网关注入。

Memory

记忆每轮怎么进请求：一个当便签，一个当底稿

记忆也是每轮装配的一部分，但两家塞法不同，正好照出两种记忆观。Claude Code 每轮对你这句话做语义召回、最多挑 5 条相关记忆，包在会标注"存于几天前"的 <system-reminder> 里（而且这条每轮召回本地默认是关的，要服务端开旗才生效），并反复叮嘱"用前先读当前文件验证"——它把记忆当成会过期的便签。Hermes 则把整本 MEMORY.md 和 USER.md 原样冻进系统提示词的易变层、在压缩交接里标为 authoritative（不被摘要覆盖），外部记忆插件召回的片段也裹着一句「Treat as authoritative reference data」——它把记忆当成不该被冲淡的底稿。一个防陈旧，一个保权威。

The verdict

装配是共享手艺，分叉泄露身份

三层都先趋同、再在同一处分叉：静态层都冻结吃缓存，动态层都塞进消息窗口且变了才补，压缩层都跑"先删后摘、外包 LLM"的同一条阶梯。请求装配这件事，八成是两家共享的工程手艺——别把它想成两套哲学。

但三条分叉缝，是两个不同的根因，不是一个：静态层的分叉（前缀按模型分叉、缓存挑模型家族）和压缩层的分叉（只有它能把压缩训回权重），根子都是模型的所有权与异质性——呼应轴一脊椎；而动态层的分叉（四十多 vs 八种注入），根子是产品 vs 底座，是另一回事。我没有把三层硬收进一个漂亮的单一根因——那样会把动态层那条缝掰歪。

所以两家在输入侧的"身份"是：Claude Code 首先是个产品（它的动态层为 IDE/终端/任务而生），Hermes 首先是个面对模型动物园的拥有者（它的静态层为异质模型而分叉、它的压缩层为自家权重留了训练接口）。同一个请求包，装法八成一样；不一样的地方，恰好各自暴露了自己是什么。

Loose ends

悬而未决：八条别被主线吞掉的边界

  • [边界] "CC 约 44 种注入"是满配上限，外部默认 build 远没这么多——snip、cached-microcompact、每轮语义记忆召回、验证代理等为内部灰度或默认关；Hermes 的 8 种也含 MoA/prefill/网关注入等只在网关模式出现的项。别把"能注入的种类"当成"每轮都注入"。
  • [HM] Hermes 那层叫"volatile 易变"的其实名不副实：记忆快照/USER.md/时间戳虽叫易变，却冻在缓存的系统提示词内部，只在压缩/重建时才刷新，并非每轮变；真正每轮变的在消息窗口。所以"静态/动态"并不等于"系统提示词/volatile 层"。
  • [CC] "静态前缀"也不是整个系统提示词：分界符之后的记忆索引/环境/语言/输出风格虽在同一个 system 数组里，却按会话变、不进全局缓存。CLAUDE.md 更是根本不在系统提示词正文里，而是搭在一条合成的首条 user 消息上；git 状态又落在系统提示词尾部——三样东西分居三处。
  • [CC] prompt cache 其实有三套模式（自家全局缓存 / 含 MCP / 第三方），断点位置和块数都不同；而且"缓存断点检测器"只是事后监控 break 原因、不负责放断点。"缓存断点打在哪"没有单一答案。
  • [边界·压缩] CC 的压缩不止"整段摘要"一种：默认的 auto-compact 确实丢中段原文，但还有 partial 压缩（保留首尾原文消息）、session-memory、context-collapse 等路径（多为手动或特性门后）。"CC 只会整段摘要"只在默认路径成立。
  • [HM] 别把运行时压缩器和训练管线当成同一套：它们是两个独立模块、代码不共用，也不吐同一个字符串——运行时如今实际吐「[CONTEXT COMPACTION — REFERENCE ONLY]」长前缀，训练管线吐「[CONTEXT SUMMARY]:」，后者只是运行时至今仍认得的 legacy 格式。这道"同源"痕迹能证的是格式同源与管线存在，证不到线上权重真训过——第二根杠杆的权重级效果是意图、非本仓库可证。
  • [HM] Hermes 的请求其实有四套 api_mode（chat-completions / codex / anthropic / bedrock），序列化路径各不同，本章讲的是默认 chat-completions 那条；给 GPT-5/Codex 的系统提示词还会以 developer 角色而非 system 角色发出。"一次请求"并不是单一形状。
  • [边界] 压缩触发阈值两家都是派生值、不是固定百分比：CC≈有效窗口−13k（200k 模型约 83%），Hermes≈有效预算 50%、地板 64k、小窗退到 85%。数字差别背后是"多晚才肯动手"的不同保守度——Hermes 明显更早出手。

输出行动侧：模型吐生料，壳每轮加工两样成品

一样是能用的行动，一样是能回传给模型的历史。

TL;DR

模型一次吐出的不是一句话，是一叠异构块——有它自己想的（推理）、有说给你听的（正文）、有要动手的（工具调用），混在一坨里。壳每一轮干的是两件加工活：把这叠生料变成 ① 能用的行动、② 能合法回传给模型的历史。

两条加工线结构一模一样——底层一层协议地板两家共享（CC 这层一点不薄，回传前重塑是它最厚的工程）；上面 Hermes 各多铺一层：行动线多铺弱模型语义纠偏、历史线多铺异质 provider 适配。这是两股不同的根因，不是一股。CC 之所以能不铺，只因它只伺候一个受托管模型 + 一个 API。

本章只讲「模型吐的块本轮怎么消费、回传前怎么规整」；循环为什么不致死、每类错误怎么续命见 01 章；每轮新塞什么进请求、历史怎么压缩见 02 章。

00 · 问题

你以为模型回你一句话，其实它吐回来一叠块

这一章不追"模型说了什么"，只追一件事：模型一次吐回来的那叠东西，壳每一轮到底拿它怎么办。

你让 agent 改一行配置。它「回答」你的时候，底层其实一次性吐回了几段性质完全不同的东西：一段它自己的盘算（该改哪、会不会连累别处）、一句说给你听的话（「我来改」）、还有一条真要落到磁盘上的动作（改 config 第 12 行）。这三样挤在同一次响应里，壳收到的不是一句干净的指令，是一叠混排的块。问题就在这儿：壳每一轮拿这叠混排的生料，到底要做成什么？而两家做的，又在哪儿一样、哪儿不一样？

于是壳每一轮都要把这叠生料拆开、分类、分别加工。哪些是「要动手」的，得挑出来真去执行——这条是行动线；而这一整轮（它想的、说的、干的，加上工具跑出来的结果）又必须原样塞回下一次调用，好让模型「记得」自己刚才干了啥——这条是历史线。

本章就沿这两条线各走一遍。你会看到：两条线的底层地板两家几乎一样，真正的分野都在上面那薄薄一层——而且行动线和历史线上那一层，是被两件完全不同的事逼出来的。

Line 1 · Action

行动线：把「要动手」的块，变成真落地的一次操作

行动线只认那叠块里要动手的那种。壳把它挑出来、按名字找到对应工具、过一道权限门、真去改文件或跑命令，再把结果翻译成模型看得懂的块送回去——骨架刚才那张图已经摆过：地板两家共享，上层只有 Hermes 铺一层。这一节把上层那层补成真实制品。它多铺这层，是因为不敢假设模型总能把工具名喊对。

具体是三种条件触发的纠偏（不是每轮都做，别读成「Hermes 总在猜」）：工具名拼错了就模糊匹配改对；工具名空着——壳判定这多半是弱模型把文件里长得像工具调用的 XML 照抄了出来，于是识破这种「鹦鹉学舌」，只回一句「那是数据、不是指令」，而不是把整个工具目录再喂回去火上浇油；正文里混进了假的工具调用就剥掉。这三条 Claude Code 一条都没有——它赌自己托管的强模型能把名字喊对，喊错也只回一条报错、让模型自纠，不替它猜。

Line 2 · History

历史线：把这一整轮，重塑成模型下一轮认得、API 肯收的历史

行动线管「这一轮怎么动手」，历史线管「这一整轮怎么塞回去」。模型没有记忆，壳每一轮都要把到目前为止的整段对话重新喂进去（02 章讲过）——但生料不能原样塞：模型吐的那叠块里，有些 API 根本不收（比如以推理块结尾的 assistant 轮），有些会让下一轮请求直接报错（比如发出去却没配上结果的悬空调用）。所以回传前有一道重塑：剥掉结尾的推理块、把「只有推理」的孤儿轮并进相邻轮、把散开的块按 id 缝回、给悬空的调用补上占位结果。这道地板两家都铺，而且是 CC 最厚的一段工程——别被「CC 壳薄」误导。

一条关键纪律，和 02 章的每轮注入一模一样：这道重塑只动本轮调用的那份临时副本，持久历史一字不改——你在 UI 里看到的转录、存档里留下的记录，都是没被动过的原貌。壳绝不把重塑写回会话库。这是同一条纪律在输出侧的落点：只动本轮副本、绝不写回会话库。

地板之上，Hermes 又多铺一层——但这一层的根因不是弱模型，是异质 provider。它要对接几十种推理服务，每家吐推理的字段名/格式都不同，于是要做逐 provider 的方言翻译（把各家的 reasoning_content 归一）；吃不到显式缓存的本地引擎，还要把消息空白和工具调用 JSON 规范到逐字节一致去蹭 KV 复用；再加上清理某些模型吐出的孤儿代理字符、按严格 API 要求剥掉未知字段。这层 CC 全都没有——它只有一个 API，回传格式天生统一。

一个容易读错的点要说准：两家都能把图片放进消息给模型看（多模态）；区别在存档/轨迹里怎么省空间——两家都不把 base64 原样堆进历史，但顶替方式不同：CC 基本只留个二进制占位、把图默默丢掉；Hermes 则把图换成工具自带的一句文字摘要（text_summary）或 [screenshot] 占位顶上。所以「剥掉 base64」两家都做，但「剥完换不换一句描述」并不一致。

前面两条线讲的是「块怎么变行动、怎么变历史」。收尾看两个更具体的切面：工具跑完了，结果怎么翻回模型眼前（结果回捕)；以及出错了，错误变成什么样的块回到模型眼前（错误处置）。两者都属输出行动侧，都是纯改壳。

Claude Code · 结果回捕

每个工具自带「结果翻译器」，转成标准结果块、用 id 配对送回。文本带行号、图片转图像块、PDF 转文档块（多模态）、出错包成 <tool_use_error>。结果太大多数不硬截而落盘、只回预览+路径（Read 类豁免防死循环；Bash 超 64MB 仍硬截）。防撑爆是双层：单结果落盘 + 每轮聚合预算（约 20 万字符）。

Hermes Agent · 结果回捕

运行时结果是 OpenAI 风格的工具结果消息 + id 配对（<tool_response> 是训练/存档表示）。同样双层：单结果超阈值落盘指路 + 每轮聚合预算（约 20 万字符）。真正独有的一层是按模型上下文窗口缩放阈值：小上下文模型缩到约 16k 地板、大模型 clamp 回默认。图片进消息给模型看；存档/轨迹里换成工具自带的 text_summary 或 [screenshot] 占位顶替（不堆 base64）。

为何不同（维度修正 · 改壳）：两家都用「落盘+指路」代替硬截断，也都有「单结果落盘 + 每轮聚合预算」双层——别把「整轮总预算」当成 Hermes 独有（那是早先的误读，已证伪）。Hermes 真正独有的只有一处：阈值按模型上下文窗口缩放——小上下文的开源模型缩得更狠。根因还是那条：CC 只伺候一个大窗口的托管模型、可以用固定阈值；Hermes 面对一群窗口大小不一的开源模型，只能让阈值随模型走。

Tool exposure

工具太多怎么办：两家几乎同一手，都是纯改壳

牌太多会撑爆上下文。两家的解法几乎一样，而且都是纯改壳：工具目录常驻，详情按需拉取——初始只露名字，模型要用哪个，再用一把「搜索 / 取说明书」的钥匙把完整参数表拉回来。Claude Code——超阈值（auto 模式 10%）就把 MCP 及可延迟工具撤成按需；Hermes——可延迟工具超默认 10% 阈值就撤柜，只留 tool_search / tool_describe / tool_call 三把钥匙，且桥工具的 tool_call 原路穿过同一分发入口，权限门一视同仁。

差异只在工程细节、不改结论：CC 把按需拉来的工具伪装成和常驻工具同一种编码，降低模型认知负担；Hermes 让桥工具复用统一分发管线，避免「绕一层桥就漏掉权限门」。壳的智慧，不是模型的。

综合①：一个 Loop 是怎么设计的

三侧拆完拼回来：一个回合的一生，与所有分叉的最终去向。

TL;DR

读完三侧，最该带走的不是「两家有多不同」，而是两家的 Loop 八成是同一门手艺——机械循环同构、装配骨架同一张、协议地板互相通用。真正的分叉只有 8 处，而且几乎全部通向两个变量：敢多信这个模型（信任，决定壳的厚度）；拥不拥有这个模型（所有权，决定有没有第二根杠杆）。

Claude Code 押注一个自家托管的强模型——通向两杠杆的 7 处分叉上，它的答案全是「不铺、不分、不训」，单杠杆且轻用；Hermes 面对一群参差的开源模型、又自己训模型——改壳、改模型两根杠杆一起拉。外加一处诚实的例外：有一处分叉谁的杠杆都不归，它只是「产品和底座长得不一样」。

00 · 问题

三张解剖图都对，可你还是没见过这台机器跑起来

前三章像三张解剖图：01 拆「循环为什么转得下去」，02 拆「每轮那个请求包怎么拼」，03 拆「吐出来的生料怎么加工」。每张图都对——可解剖图有个天生的缺陷：看完三张，你还是没见过这台机器跑起来的样子。更要紧的是，三章各自拆出的那些分叉散落在各处，它们到底通向哪里、是不是同几个变量在反复作祟，光靠解剖图也拼不出来。

所以这一章不新增任何机制，只干两件事：先把一个回合从头到尾走一遍（8 站，每站标清「这里两家一样」还是「这里分了叉」），让机器真的转起来；再把全部分叉收拢到一张图上，看它们最终汇向哪里。每一站、每处分叉，都回指三章里已过门的结论。

The lock

两套系统各自咬合，拆件混装谁都装不上

把分叉归了因，还剩最后一层要说透：这些选择不是八个独立的取舍，是两套互相咬合的系统——每家的每个选择，都以它其余的选择为前提。

Claude Code —— 「一个可控托管模型」咬合出的一套

  • 前缀真·统一——不必分叉，因为只有一个模型；
  • 敢用固定阈值——窗口大小是已知数；
  • 壳对格式不设防——模型受训可靠，喊错名只回报错让它自纠；
  • 防懒交给模型自觉——相信「我做完了」的自述；
  • 回报是：判断权交给模型，天花板随模型升级自动抬高。

Hermes —— 「一群异质模型＋自有权重」咬合出的一套

  • 前缀按模型分叉——Google 系、GPT 系各要各的话术；
  • 阈值随窗口走——群里有小窗口也有大窗口；
  • 壳加纠偏层——不能假设第三方模型把名字喊对；
  • 收工设验证门——编码入口不放行任何模型（含自家）的自述；
  • 能训进权重的训进去——第二根杠杆只有它拿得起。

用三个反事实试一下就知道「咬合」有多紧。把 CC 的统一前缀搬进 Hermes？第一天就碎——Google 系要「别停」、GPT 系要「执行铁律」、阿里系连自己叫什么都要壳来纠正。前缀分叉不是设计品味，是模型群的物理现实。把 Hermes 的方言归一层搬进 CC？无物可翻——它只有一个 API，那一层一行都用不上。壳厚不是勤奋，壳薄也不是懒惰，都是处境的倒影。

最有意思的是第三个反事实：给 CC 装 Hermes 的验证门？能装——而且正在装。停止钩子的口子一直留着（结构上与验证门同族），内部灰度里已经有「非平凡改动须独立验证代理裁决」的契约。这说明「锁定」分两种：处境锁死的不会收敛（异质、所有权——除非处境本身变了），纯信任偏好的正在收敛（防懒这道门，两家方向一致、只差默认与否）。信任线是活的，处境线是死的。

判词：别问「谁做得对」，问「你的处境像谁」。这份对比的单位从来不是一张功能对照表，而是两套各自咬合的系统——单拆一件装进对方，多半既装不上、也没必要。
The payoff

01 章留下的反直觉，现在可以讲完整了

01 章结尾留了一个反直觉：做模型的那家，反而更不信任模型说的「我做完了」。只看控制侧，这像一个悖论；把三章摊平到汇流图上，它拆成两层来源不同的不信任——

一层是对「任何模型的自述」的不信，连自家也不例外：在编码入口，验证门对自家模型照拦不误（这道门按产品场景发放——聊天入口默认关，见 01 章；它不信的是「模型自述」这件事本身，不是无差别猜疑）。这一层来自专业——越清楚权重是怎么训出来的，越知道「我做完了」只是一个待核实的声明，不是结论。另一层是对「别人家模型」的经验性不信：纠偏层、名单加压，兜的是没被这样训练过的第三方模型，自家模型反倒不是主要服务对象。这一层来自生态——开源世界里没法假设可靠。

所以那句反直觉的完整版不是「做模型的人更悲观」，而是：做模型的人，处境里的变量更多。Anthropic 的处境里这两层都不存在——一个模型、自家调教、持续变强，押注它就好；Nous 的处境里两层都在，于是能训进权重的训进去，训不进去的用壳挡。

越清楚权重的脾气，越不敢把「完成」的判断权交出去。
For PM

你的处境像谁，就抄谁的作业

若你基于一个强托管模型做 agent：抄 Claude Code——壳做薄、判断权交给模型，天花板随模型涨；但要接受模型犯浑时薄壳不兜底，并且记住它的「薄」只薄在上层，协议地板（回传重塑、配对补全）一点不能省。

若你要跑多种／开源／自托管模型：抄 Hermes——那六处改壳分叉就是你的施工清单：三处加厚（验证门、按模型下菜、纠偏层）、三处适配（缓存分档、方言归一、窗口缩放）。至于第七处（训回权重），只在你也训模型时才对你开放。

一句话带走：壳的厚度不是能力问题，是信任问题；第二根杠杆不是技术问题，是所有权问题。

Loop 与时间：你不在的时候，它把「你不在」当成什么

交代完就走了——从这一刻起，它干活的时间线和你的注意力时间线彻底分了家。

TL;DR

你把要做的事交代完，人就走了。从这一刻起，agent 干活的时间线和你的注意力时间线解了耦——它继续走，你已经不在看。

同样面对「你不在」，两家的读法相反：CC 把它当成一个被持续测量、要设法弥合的信号——设法把你的注意力找回来；Hermes 把它当成一个由语境注定、要照章办的事实——超时=没同意、错过=不补、无事=沉默。

00问题：你不在，才是这台机器的常态

凌晨三点，你睡了。一个新 PR 刚合进来，或者你上周设的定时冒烟任务正好到点——agent 该动的时候，你不在。这段时间里它得能存在、被叫醒、把活干完；万一半路卡住需要你拍板，它还得找到你；干完了，把结果送到你早上会看见的地方。你不在，是常态，不是异常。于是真正的问题不是「它会不会干活」，而是：当你不在，它到底把「你不在」这件事读成了什么——一个要设法弥合的缺口，还是一个照章办事的既定事实？

01你交出去的，是「在场监督权」

交代完就走，你让渡的其实是一样东西：在场监督权——不再守在旁边随时喊停、随时确认。这一让渡，内生出四件它必须替你兜住的事。这四件事，就是本章的问题骨架，也是下面两家各自的答卷。

干得完 —— 你不在，它的时间线不能断在半路（谁让它醒着、崩了怎么办、错过了补不补）。

找得到我 —— 真需要你拍板时，它得够得到你（等多久、超时算什么、无人时敢不敢自己定）。

送得到我 —— 干完了，结果要送到你回来会看见的地方（送哪、要不要主动把你叫回来）。

别乱来 —— 没人盯着的时候，谁替你踩刹车（危险动作在无人时走向哪）。

在应答四诉求之前，先看最底层的一道分野：它到底怎么知道你不在。CC 靠传感器测出来（信号长在自家终端/桌面上）；Hermes 靠语境注定——你从哪扇门进来，就注定了你在不在。

Loop 与用户：它该多懂用户，这份「懂」谁说了算

和 agent 打交道，是一场双重让渡——它记得住你交出去的东西吗？

TL;DR

你跟一个 agent 打交道，本质是一场双重让渡——你交出意图（想要什么，说给它听），也交出行动权（让它替你改文件、跑命令）。

让渡内生两个期望：它得懂你的意图、它的行动得顺你的心意。所以「懂用户」不是锦上添花的功能，是委托关系里长出来的内生需求。这一章拆两家怎么定义「懂用户」、用什么手段做到、以及懂了之后凭什么不无条件照办。

00问题：第二次打交道，它还得从头认识你吗

你昨天花十分钟跟它交代过你是谁：后端出身、第一次碰前端；讨厌它擅自 mock 数据库；这个项目的规矩是所有改动合成一个 PR、别拆；还嫌它答得太啰嗦。今天你新开一个对话——它还记得吗？还是又得从头认识你一遍？这四句话，正好对应它该「懂」你的四个方面：你是谁、你的边界、你的规矩、你的口味。可一旦它真开始记你，问题就不止「记不记得」，而是：该由谁来决定它记什么、记多深、记了之后归谁、又凭什么信？

2.2哪句话值得记一辈子？最反直觉的一条是——连你成功的选择也要记

不是你说的每句都往长期记忆里塞。两家都有一套「该记 / 不该记」的判据——真正拉开认知的，是那条大多数人想不到的：只记你的纠正是不够的，你点头认可的做法也得记，否则它会越用越畏手畏脚。

✓ 该记（会让它更懂你）
  • 你是谁、你的规矩、你验证过的偏好
  • 你纠正它的走法（「别这么做」）
  • 连你成功接受的选择也要记——只记纠正，它会慢慢躲开你其实认可过的做法、变得过度谨慎
✕ 不该记（记了迟早变噪音）
  • 能从代码 / 项目现状直接读出来的（架构、文件结构、谁改了什么）
  • 一周后就过时的：PR 号、任务进度、「Phase N 做完了」、「修好了 X」
  • 连你明说「把这个存下来」，若属上面两类它也会反问：这里面真正意外、非显然的点是什么？
CC · 为什么「成功也要记」（原文）Record from failure AND success: if you only save corrections, you will avoid past mistakes but drift away from approaches the user has already validated, and may grow overly cautious.

另一头，Hermes 给「不该记」下了一条很硬的时间线判据：一周后就会过时的事实，根本不该进记忆——那些该用历史检索去找，不该沉淀成长期记忆。

Hermes · 一周会过时的别进记忆（原文）Do NOT save task progress, session outcomes, completed-work logs, or temporary TODO state... If a fact will be stale in a week, it does not belong in memory.

2.3记进去这一步本身，会不会被人塞毒、或者卡住你回复

自动写记忆有两个你想不到的坑：记忆文件可能被人动手脚（供应链、别的进程），成了往它脑子里注入指令的通道；反复整理记忆又可能陷进死循环，把该给你的回复卡住。Hermes 在写入这一步筑了三道墙，其中两道直接带着 issue 号、是被真实线上事故逼出来的。

2.4用户建模：单开一个子系统，还是揉进记忆？

上面这条学习线，落到「你是谁」这一维时，两家分野最大——它决定了下一节「你能改它多深」的起点，别抹平。

Hermes 把「你是谁」做成一等公民：独立的用户档案（USER.md，跟它记自己教训的那份记忆文件分开、两套预算、系统提示里分两块渲染）＋开机就主动提议建档（你有史以来第一条消息就触发，还带一道「先问你同不同意」的门）＋每轮请求前由外接组件现取现注一份画像。（但要诚实标注：USER.md 内部，你手写的偏好与它自学的混在同一本、无来源标记。）

03不止让它懂你——你还能直接改它，改多深？

记忆让它懂你（被动理解你是谁）；但你也能反过来主动规定它——从写一句指令，到教它一个可复用技能、设权限、挂钩子，一路到改它的源码。这条线不为「懂」，为的是让它的行为直接顺你，正好接上开篇那个「顺」的期望。关键看两家让你能走多深——而这条光谱的尽头，就是本章反复要问的那句：谁拥有代码。

04懂了，不等于就该无条件照办

这是最容易被忽略、也最值得单拎的一点：一个会自己攒记忆的 agent，不只越用越懂你，也会越用越固执——把一次临时故障硬记成永久规则。所以两家都在「懂」和「顺」之间设了一道闸，正好呼应开篇那两个期望。姿态却相反。

Claude Code · 预防式：把「不信任记忆」焊进出厂设计

系统提示里专门有一段召回纪律：记忆提到某文件就先确认它还在、要据此行动就必须先核实；「记忆说 X 存在」不等于「X 现在存在」。夜间做梦（非默认）也内建「删掉被现场推翻的旧事实」。它把「记忆天然会过期」当默认前提来设计。

用前必核实 · 原文A memory that names a specific function, file, or flag is a claim that it existed *when the memory was written* ... "The memory says X exists" is not the same as "X exists now."

Hermes · 事后式：被真实投诉后才补的护栏

后台复盘出过真实事故：把「这工具没装好」误记成「这工具坏了」，几个月后还拿这条错误结论拒绝干活——用户抱怨了。整改两手：复盘指令加一段「不要捕获」黑名单，再加一道记忆写入的审批闸（默认关、可开成暂存待审）。

「不要捕获」黑名单（直击事故）· 原文Do NOT capture ... Negative claims about tools or features ('X tool is broken'). These harden into refusals the agent cites against itself for months after the actual problem was fixed.

Hermes 还有一条更细的纪律：把偏好写成声明、不写成命令，防止「懂」硬化成机械的「顺」——这条是它独有的，CC 靠的是上面那套用前核实。

Hermes 声明非命令的写法规范 · 原文'User prefers concise responses' ✓ — 'Always respond concisely' ✗ Imperative phrasing gets re-read as a directive in later sessions and can cause repeated work or override the user's current request.

Loop 与环境：护栏趋同，分野在闯了祸算谁的

兜底不是把模型变聪明，是让「模型不聪明」也不致命。

TL;DR
环境这一侧，两家的安全机制惊人地趋同：默认都不把 agent 关进沙箱、动作直落你的真机器；都把「隔离」和「审批」当成互相替代的两条路；都留了一道连全放行都压不过的绝对地板；被拒后都塞给模型一段几乎同款的「别耍花招绕过」训诫。趋同不奇怪——兜底要防的是「模型会犯错」这个普适前提，下限设计的答案空间本就窄。

真正的分野不在机制清单，在两处。一是那道绝对地板，两家用了两套不同的判据：Hermes 按「物理可不可挽回」护住机器，CC 按「动没动管着 agent 的那层」护住规则本身——同一个 rm -rf /，一家无条件拦、一家全放行可放行。二是闯了祸算谁的，以及由此逼出的对能力边界多诚实：越是没有厂商接盘、越是你自担，它越被迫在文档里亲口写明「这不是墙」。

00问题：动作落在真机器上，兜底该怎么分类

agent 干着干着，要删一整个目录、要强推覆盖远端分支、要读你家目录里的密钥、要连一个陌生外网地址。这些动作不在沙盒里排练，是真的落在你的电脑上——删了就是真删了，推了就是真推了。

于是壳（harness）要替它兜底，而且兜的不是「让模型想得更清楚」（那是 Loop 之内的事）——是万一模型没想清楚、做错了，能不能保住下限、让代价可控。这就带出这一章的问题：如果损失真的发生，会是哪几类？针对每一类，壳把防线摆在了哪一步——动手前拦、动手时隔、还是闯了祸之后兜？

把两家的答案摆进一张「损失类型 × 防线阶段」的网里，会撞见一件反直觉的事：两家填格的方式几乎一样，真正的差别只藏在少数几个格子里。

02分野一：同一道绝对地板，两家用了两套判据

03分野二：机密外泄这一行，一家做了减法却亲口否认那是墙

在覆盖网里，「机密外泄」这一行两家的填法正好相反——而这处相反，把「闯了祸算谁的」如何反过来决定「它对自己诚实到什么份上」摆到了明面。

Claude Code · 基本敞开：赌你在可信环境

子进程继承全部密钥（不做 env 清洗）、读密钥文件无专项拦（~/.ssh／.env 只受「出没出工作区」的通用约束）、网络外发的闸只在沙箱里成立——而沙箱默认关。唯一一处密钥专拦（写团队记忆时扫密钥）还是特性门控。默认形态下，外泄这一行几乎没有防线。

04共同盲区：两家的护栏都只盯着「本机」

覆盖网里有两处，是两家一起留白的——不是谁的取舍，是这类产品共同的射程边界，对读者判断「我能把什么托付给它」尤其要紧。

盲区一：祸落在别人头上的那一类，两家事前几乎都不设防。强推覆盖共享分支、往外发消息／邮件、发布 npm／pip 包——这些「外部世界后果」在覆盖网第五行几乎全空。CC 这边，force push 只有一句信息提示（不影响放行）、npm publish 要靠你自己写规则、系统提示里「不主动 push」是写给模型的纪律、不是代码闸门；Hermes 这边，agent 主动发消息／发邮件零审批，publish 无拦，只有 force push 被某条 shell 正则偶然覆盖到。两家的防线都画在「别毁了这台机器、这个仓库」，对「别把祸甩给别人」基本沉默。

盲区二：护栏几乎只长在「shell 命令」这一个面上。那套危险删除拦截、配置保护、沙箱，两家都只覆盖终端命令和文件编辑。CC 的 MCP／自定义工具直接放行（passthrough，完全不过环境侧的路径闸和沙箱）；Hermes 的 MCP 调用、发消息、发布也都绕开命令闸，只靠一个默认关（opt-in）的通用钩子兜——插件默认不加载，非 shell 面开箱即无闸。含义很实：一个改用 MCP 工具去删数据、去联网外传的 agent，能绕开两家几乎全部的环境护栏。

为什么一起留白：两家的环境治理，本质是「本机资产治理」——把 shell 能对这台机器、这个仓库干的坏事拦住。一旦动作换个出口（对外发送、发布、经由 MCP 工具落地），或者后果换个落点（砸在协作者而非你自己头上），就掉出了射程。这不是 bug，是这一代 harness 危险治理的形状：它守的是「你的机器」，不是「你的机器能波及的世界」。

05verdict · 机制趋同，账本分家

把这一章收成一句：环境侧的安全机制之所以惊人趋同，是因为它们都在给「模型会犯错」这个普适前提做下限设计——答案空间窄，就收敛到同一批动作上（拦、隔、兜）。真正把两家分开的，是账本。

第一本账：那道绝对地板护什么。Hermes 按「物理可不可挽回」护机器，CC 按「动没动治理层」护规则本身。一个防的是物理上收不回的动作，一个防的是能改写游戏规则本身的动作——各自把对方留在软层的那一样，单独升格成了硬地板。

第二本账：闯了祸算谁的。CC 厂商托管、多半跑在你自己可信的仓库里、身后有 git 兜文件、有企业总闸能远程锁死全放行档；Hermes 开源自托管、你的机器你的账号、没有厂商接盘、责任白纸黑字归运维。风险归属的不同，又反过来逼出了「对能力边界多诚实」的不同：越是你自担，它越被迫把「真边界只有操作系统」说死。

一个共识别漏掉：两家其实都在对你说「别全信我」，只是指向了不同的出口——CC 说「只对你信任的代码用我」，Hermes 说「进程内没有任何东西算隔离，唯一边界是操作系统」。是「厂商托管、身后有人兜」还是「自托管、你自担」，决定了它把最后那道防线画在哪，也决定了它对自己诚实到什么份上。

06边界说明（防「默认→绝对」以偏概全）

  • [两家 · 默认不隔离] CC 命令沙箱默认关、开箱即用全靠权限弹窗；Hermes 后端默认 local ＝ 裸主机，容器／远程／云沙箱全是 opt-in。别记成「它们在沙箱里跑命令」——恰恰相反，默认不关。且沙箱一开是换范式（用隔离换掉逐条问）不是加一层。
  • [两家 · 事后只兜半格] 「事后兜底全外包给 git」不成立：两家都有原生撤销，但都只覆盖自己经结构化编辑工具改的文件——CC 的 rewind 交互默认开、Hermes 的 rollback 默认关（v2 从开翻成关，注释「多数人从不用」）；bash 层的 rm／dd／重定向、已推 commit、已发外部动作，两家事后都兜不住。

Loop 与其他 Agent：为几乎相同的理由拆开，却在拆开之后分道扬镳

单个 agent 的上下文与时间总有极限——拆成多个分身是必然。真正分野的，是拆开「之后」。

TL;DR

多 agent 不是为了「更聪明」，是被上下文极限逼出来的手段——两家都提供三种模式：自己干、派一个隔离分身替你干一段、组一支共用看板的团队。模式清单相似，不是差异所在。

真正的产品判断，不在「有哪些模式」，而在什么时候该拆、什么时候升级、谁来决定——这一层两家分道。本章像一份产品报告，按一个任务被拆开、干完、合回来的顺序走：为什么拆 → 有哪些模式 → 何时拆（脊）→ 派出去的代价 → 分身怎么协调 → 怎么合回来。不碰「谁假设有人在场盯着」那根线——那是综合②的活。

00问题：一个 agent 干不完的活，拆开之后谁跟谁怎么协调

你交给它一个大活：把一个中型仓库从一套框架整体迁到另一套，顺带补全测试、更新文档。一个 agent 干不完——光是把相关文件逐个读进来、逐个尝试、逐个回滚，几十上百轮工具调用的原始输出就把它的上下文塞爆了，它开始「健忘」，也拖到你等不起。于是它把活拆开，派出几个分身各干一段。拆开这一步，两家几乎一样；可真正的问题从拆开之后才开始：什么时候才真该拆、拆出来的分身带着什么代价、它们之间靠什么协调、干完又怎么合回一个连贯结果？下面按这个顺序，一件件摊开。

01为什么非拆不可：两家给出的理由几乎逐字重合

要理解分野，先得承认起点是趋同的。两家在提示词里自认的「为什么要多 agent」几乎一模一样：核心不是分工，而是防中间原始输出淹没父的上下文——把一段会产生大量一次性输出的子任务派出去，只让摘要回来。

Claude Code 的 subagent 机制上也是隔离的——子在自己的子进程里跑、只把一条结果回你，护上下文是这个机制的自然结果。把这层判据讲得最直白的，是它 fork 实验里的一句质性问题（不是「任务大不大」，而是「这些中间输出我以后还要不要」）：

CC · 何时该拆（fork 自己 · 灰度实验）· 原文Fork yourself ... when the intermediate tool output isn't worth keeping in your context. The criterion is qualitative — "will I need this output again" — not task size.

Hermes 把同一件事讲成一句机制承诺——中间结果根本不进你的窗口，只回摘要：

HM · 委派的意义 · 原文Only the final summary is returned -- intermediate tool results never enter your context window.

连「何时不该拆」也趋同：目标已知就直接读、单个工具调用直接调、机械活别专门开一支队伍——两家都在提示词里劝模型别为拆而拆。这一层是全章最没有分歧的地方，本章不在这里制造对立。既然「为什么拆」趋同，先把「有哪些协作模式」这张地图铺开——你会发现两家的模式清单也高度相似。

02有哪些协作模式：三种，而不是一条阶梯

把多 agent 能力摊开，真实结构是三种模式：自己干、派一个隔离分身替你干一段、一群分身共用一块看板协作。注意：并行、嵌套这些不是独立模式，是「委派」这一种模式的用法参数——能同时发生（比如一次并行派 3 个后台分身）。下面逐个讲清：干嘛用、什么时候触发、CC / HM 触发后的特性差异。

模式一不拆 · 单 agent 自己干

干嘛用：目标明确、机械活，自己直接做最省——多拆一个分身都是白白的 token 与协调开销。什么时候触发：目标已知、单个工具能解决、不会产生海量中间输出。特性：CC → 直接 Read / Glob；Hermes → 直接 execute_code。

模式二 · 主力委派 subagent · 派个隔离分身替你干一段两家人人可用

干嘛用：把会产生大量中间输出的子任务派出去，只让摘要回来，别让原始输出淹没父的上下文。什么时候触发：中间产物会塞爆上下文、或有真正独立可并行的活。这是两家真正的主力能力（CC 的 AgentTool、Hermes 的 delegate_task，都无门控常驻）。

03何时拆：拆不拆 · 拆多重 · 谁来判断——这一层才是产品判断的心脏

机制清单只是坐标系，真正的产品判断在「何时动用」。每拆一个分身，都多一份 token、多一份协调、多一个跑偏风险，还丢了主 agent 的上下文连续性。所以「何时拆」不是一个点，是一条有三个刻度的线——两家在第一个刻度趋同，在后两个分道。

刻度 ①拆不拆两家趋同

什么时候真该拆、什么时候单 agent 就够——两家几乎逐字重合地划同一条「别滥用」的红线。

该拆的信号
  • 中间产物会淹没父的上下文
  • 有真正独立、可并行的活
  • 推理密集、需要独立思考的子任务
不该拆的信号
  • 机械活——别专门派一支队伍
  • 单个工具调用能解决——直接调
  • 目标已知——直接读，别绕分身

04派出去之后：分身带着什么代价干活

分身派出去了，它带着几笔代价在外面干活：父信不信它自报的「我做完了」、它烧多少 token、它够不够得到你。第一笔最能照出两家的态度分野，也直接呼应轴一里两家对「自报」的整体态度——姿态正好相反。

Claude Code · 默认信任，只有编排模式才另加核验层

子分身的输出默认就该信——这是常规提示词里的原话、不受任何实验开关门控。只有显式开启的 coordinator 编排模式（灰度实验），才会额外插一层独立验证角色，并且强调验证要证明代码真能跑、不是盖章走过场。

CC · 子输出默认可信（常规·非门控）· 原文The agent's outputs should generally be trusted
CC · 唯编排模式才加的核验纪律 · 原文Verification means proving the code works, not confirming it exists. A verifier that rubber-stamps weak work undermines everything.

Hermes · 默认不信，把「亲自复核」焊进委派工具本身

子分身的摘要被当成自报、不是已核实的事实——这是委派工具的常态纪律，不是某个模式才开：要求子分身回一个可核验的凭据（URL / ID / 绝对路径 / HTTP 状态），父自己去验。

HM · 摘要是自报、须亲验 · 原文Subagent summaries are SELF-REPORTS, not verified facts ... require the subagent to return a verifiable handle (URL, ID, absolute path, HTTP status) and verify it yourself
为何相反：CC 把默认信任当基线、按需在编排模式加核验；Hermes 把默认怀疑写进委派常态。这正是轴一「对任何自报不全信」的态度在多 agent 场景里的延续——不是新立场，是同一立场的贯彻。

其余两笔代价，两家趋同。　成本：都对烧钱敏感、只是抓手不同——CC 优化缓存命中（fork 保持字节级相同的前缀、复用 prompt 缓存，还叮嘱别乱设模型），Hermes 优化上下文预算（按父剩余额度的一半再除以批量裁剪每条摘要，并发默认 3、超 10 个警告成本线性上涨）。　够不到用户：两家的分身都联系不上你——CC 异步 agent 的工具白名单里没有「问用户」，Hermes 直接把 clarify 硬 block 掉，结论都必须由父转达。　隔离（各自沙箱、防互踩）是每一级都能拧的旋钮，已在上一节阶梯的横切条里交代。

06干完怎么合回来：成果的回流、汇聚与落地

Loop 的 self-improvement：自改的手段都在，评估的环节缺席

两家都能改写自己、都会把经验写回下一轮——却都没有一段代码，回头确认这次自改到底改对了没有。

TL;DR

自我进化看起来像训练问题，其实是一个控制回路问题。两家都具备自改的手段（写记忆、造技能）和固化的机制（写盘、下次会话载入），积累这半个回路是闭合的——写入的经验，下一轮一定会被重新读到。

缺的是评估这一环：没有任何代码在一次自改之后，去检验它是否真的改善了后续结果。评估缺席，回路就无法真正闭合。差别只在这一环由谁来补：CC 把评估外包给厂商，Hermes 拥有模型、却尚未把评估构建出来。

00问题：自我进化，进化的是「变好」还是「变大」

「自我进化」是这类 agent 最诱人的一个承诺：用得越久，它越懂你、越能干，仿佛会自己成长。这个画面背后藏着一个更基本、也更少被追问的问题——一个 agent 说自己在自我进化，它究竟在进化什么：是真的变得更好，还是只是攒下越来越多的东西？

两家确实都在不停地往磁盘写记忆、造技能，下次开机再读回来，积累这件事实打实地转着。但攒得多，不等于变得好。于是真正的问题落在一个很具体的地方：有没有哪一段代码，在一次自改之后，回头确认过这次改动真的让后来的结果变好了？这一章要挖的，正是被「自我进化」四个字盖住的这道缺口——它到底在闭合哪半个回路。

01先厘清一个误读：你的使用没有在重训它的权重

Hermes 把自己写成「self-improving」，README 第一句就自称自带学习闭环。最容易产生的误读是「你每天跟它聊天，就在实时重训它的权重」——这个读法不成立，两家代码都不支持。

Hermes 的训练管线吃的是研究员策划好的数据集；跑批量时把持久记忆整个关掉、也不自存轨迹，轨迹压缩是离线后处理——跟你的活会话之间，没有一根自动接线。

02自我进化是一个控制回路，两家只闭合了积累半环

把「自我进化」摊开，它是一个有若干环节的控制回路：先定义什么算「变好」（目标 / fitness），再发现问题（误差信号），再动手自改，再评估这次自改是否奏效（信用分配），最后把有效的经验固化上线、进入下一轮。本章只盯评估这一环。

两家的自改与固化都真接通了——自动抽取记忆、后台造/改技能，直接写进磁盘，下次会话开场就载入进系统提示。积累这半个回路是实打实转起来的。（记忆怎么攒、怎么懂你，是第 5 篇的事；这里只看回路闭没闭。）

在别处、而且是真的：发版前的实验室 eval ＋ 厂商拿线上做 A/B ＋ 在场的人逐次当裁判。
哪都没接：拥有模型、造了最自主的自改机器，改进回路却最不闭合。

CC 把评估外包得很彻底，而且这套评估是真的。记忆相关的 prompt 里到处标着「Eval-validated」，某条排除规则旁边就注着一个 eval 案例从 0/2 跑到 3/3——说明这些措辞是在实验室里用离线跑分调出来的。但那些 eval 文件不随产品发货：它约束的是发版前的 prompt，不是你会话里的运行时行为。CC 的改进回路确实闭合——闭在厂商的发版节奏上，不在你的会话里。

Hermes 恰恰相反：它拥有模型、造出了最自主的自改机制，改进回路却最不闭合——训练管线是真的，但吃的是策划集、不接活会话；运行时的评估，一处都没接上。

为什么是这两种缺法？往回接两根全书的主脊——

① 谁拥有模型（轴一）：CC 不拥有模型，改进注定是厂商的活，于是它自觉把评估整个外包（实验室 eval ＋ 厂商 A/B ＋ 在场的人）；Hermes 拥有模型、本可本地闭环，却构建了自改、没构建评估。

② 谁假设有人盯（综合②）：CC 假设人在场、厂商在后，就不需要自主评估——在场的人就是那个逐次校准的裁判；Hermes 常态无人值守，最需要自主评估，偏偏最缺——这是最刺眼的落差。

回到「对它而言，『变好』由谁定义」这个实然问题——两条答案都立在机制上，不评谁更好：

对 CC 而言，变好是一件托管产品的事：由厂商按版本迭代、由在场的人当场校准；你只管用，改进由外部完成。

对 Hermes 而言，它更像一个追求自立、但尚未具备自我评估的系统：能自己修改自己，却还不能判断这些修改是否让自己变好。

05悬而未决 · 可证边界与防夸大

  • [训练那半 · 对称标] 两家都只证到「攒了数据 / 攒了轨迹」；「训进权重、下一代更强」都在仓库外、都是意图。必须对称标「意图 · 非本仓可证」，不许只给一家戴帽。
  • [fitness 残余 · 仓外 / 未审] 「仓内运行时无」是高置信，但边界要老实划：CC 的 eval / A/B 在厂商侧、不在仓库，另有一处 confidenceRating 服务只在注释里被提到、本快照无源文件；Hermes 接的外部记忆 provider（Honcho）内部打不打分未审，随附的 datagen 示例配置里还有个 held-out eval 旋钮（eval_every / eval_size）、但本仓未见实现、且属训练侧、不接活会话。这些都标「仓外 / 未审」——不写「绝对没有」，也别把未实现的示例抬成「它有 fitness」。
  • [状态纪律 · 别把门后当默认] 记忆 / 技能自造多是门控或示例配置默认开：Hermes 代码里默认关、随附示例配置默认开；CC 的 dream / 记忆调查都在门控后面。写审批闸默认关、curator 合并默认关。别把门后的实验讲成「默认就这么强」。
  • [回合内质检 ≠ 跨改进信用分配] 会话内的对错核验，和「我这次自改跨任务变好了没」是两件事，别混。写入侧那些护栏都是负向 / 事故驱动的，细节见第 5 篇。

综合② · 轴二Loop 之外是怎么设计的

五种关系 · 一个总变量

它假设，有没有人正盯着？

五种关系看着是五件事——何时醒来、听谁改写、在哪动手、和谁协作、怎么变强。但把它们并在一起，每一次分野都由同一个问题决定：这个 agent，假设有没有人正盯着它？ Claude Code 处处假设「有人」，Hermes 处处假设「没人」。而这个假设不是品味选择——是所有权处境替它选好的。

The scene

半夜三点，没有人在屏幕前

一个任务被触发了——也许是定时器，也许是一条消息推送。此刻五个问题同时冒出来：谁让它醒着？它要改的东西，谁批准？它闯了祸，谁兜底？它派出去的分身，谁协调？它这一趟学到的，算谁的？

每一个答案，都取决于设计者当初赌没赌「会有人在」。这就是把轴二五章串成一个故事的那根脊。

For PMs · 怎么用

先回答这一个问题，后面五件事就定了

学 CC
如果你的 agent 活在「有人盯着」的世界（IDE 里、有人随时能拍板）：把拍板权留给人，无人时保守到默认拒。天花板随人的在场自然抬高。
学 Hermes
如果你的 agent 要在「没人盯着」的世界跑（后台、被消息叫醒、自托管）：每一个「本该有人点头」的地方，都得提前定义「没人点头时怎么办」；并诚实到白纸黑字承认边界在哪，别假装有隔离。
一句话
先问「我假设有没有人在场」——这一个答案会决定你后面五件事怎么设计。而它往往不是你能自由选的：你的所有权处境（托管还是自托管、闭源还是开源）已经替你选了。
Loose ends · 诚实边界

这根脊，不是五等份

  • 两条线，别硬拧成一条。对时间／环境／同类，「在场」是正面主轴；对人／对自己，真正在分的是所有权链，在场只是它的侧面（治理层有没有人盯／变强要不要你签字）。硬说成「在场的五次显形」就是以偏概全——所以本章把两条线分开画（脊上的空心点即标此）。
  • 「对同类」那一行是解读，不是铁证。「人当队长 vs 数据库协调」是对机制的合理解读，非代码里可直接证的设计意图（半点标注即此）。
  • 「变强」这一行，两家都别信自己。无论 CC 的「外包给下一代基座」还是 Hermes 的「握着训练杠杆」，画的都是意图与潜力、不是已经做到——两家仓库都证不到「你的使用让它权重级变强」（CC 断在没这能力，Hermes 断在产数据管线没接你的活会话）。脊上 row5 标的是「谁有可能」，别读成「谁已经在越用越强」。
  • 五关系的边界仍有一处未闭。「对同类」是否还应含外来 agent 互通（如 Hermes 的 ACP 适配器），07 章尚未覆盖、待查证；若那格材料成立，本综合「五关系」的完备性自评要回来再调——不因为故事顺就盖章。
